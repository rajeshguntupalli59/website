How to find missing indexes in PostgreSQL in 30 seconds

Rajesh Guntupalli · May 2026 · 5 min read

Every PostgreSQL DBA has been there — a query that was fast last month is suddenly taking 10 seconds. You run EXPLAIN ANALYZE, stare at the output, and wonder where to start.

The raw output looks like this:

Seq Scan on orders  (cost=0.00..450.00 rows=2000 width=48)
  Filter: (customer_id = 42)
Planning Time: 0.5 ms
Execution Time: 892.3 ms

You can see "Seq Scan" which sounds bad, and 892ms which definitely is bad. But what exactly do you do about it?

The problem with manual EXPLAIN reading

Reading EXPLAIN output well takes years of practice. You need to know that Seq Scan on a large table means no index is being used, that the cost numbers are estimates not milliseconds, and that the fix in this case is an index on customer_id.

Most developers don't have that experience. And even experienced DBAs spend more time than necessary on this for every single slow query.

A faster way — let the catalog tell you

PostgreSQL tracks which indexes exist and which columns your queries filter on. You can query pg_stat_user_indexes and information_schema directly to find columns that are heavily filtered but have no index.

The query looks something like this:

SELECT
    t.relname AS table_name,
    a.attname AS column_name,
    n.nspname AS schema_name
FROM pg_class t
JOIN pg_attribute a ON a.attrelid = t.oid
JOIN pg_namespace n ON n.oid = t.relnamespace
WHERE t.relkind = 'r'
  AND a.attnum > 0
  AND NOT EXISTS (
    SELECT 1 FROM pg_index i
    JOIN pg_attribute ia ON ia.attrelid = i.indrelid
      AND ia.attnum = ANY(i.indkey)
    WHERE i.indrelid = t.oid
      AND ia.attname = a.attname
  );

This returns every column with no index — a starting point. But you still need to cross-reference with your actual queries to know which ones matter.
