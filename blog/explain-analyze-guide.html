# Reading PostgreSQL EXPLAIN ANALYZE like a pro

Every time I sit down to debug a slow query, the first thing I run is EXPLAIN ANALYZE. After years of doing this professionally, I've come to think of the output as a conversation — PostgreSQL is telling you exactly what it did, how long it took, and where it struggled. Most people skim it. I want to help you actually read it.

Let's dig in with a real example.

The basic command

Always run this version when you're debugging:

EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT o.id, o.created_at, c.name
FROM orders o
JOIN customers c ON c.id = o.customer_id
WHERE o.status = 'pending'
  AND o.created_at > NOW() - INTERVAL '30 days';

The BUFFERS flag is one I see people skip constantly. Don't skip it. It tells you whether PostgreSQL pulled pages from shared memory or had to hit disk — which is often the difference between a 5ms query and a 500ms one.

Anatomy of the output

Here's what you'll typically get back (I've trimmed it for readability):

Hash Join  (cost=1284.00..4821.33 rows=3200 width=48)
           (actual time=18.432..67.841 rows=3187 loops=1)
  Buffers: shared hit=1423 read=287
  ->  Seq Scan on orders o  (cost=0.00..3200.00 rows=3200 width=32)
                             (actual time=0.021..41.223 rows=3187 loops=1)
        Filter: ((status = 'pending') AND (created_at > (now() - '30 days'::interval)))
        Rows Removed by Filter: 96813
        Buffers: shared hit=1100 read=287
  ->  Hash  (cost=820.00..820.00 rows=20000 width=24)
             (actual time=17.901..17.901 rows=20000 loops=1)
        Buckets: 32768  Batches: 1  Memory Usage: 1234kB
        Buffers: shared hit=323
        ->  Seq Scan on customers c  (cost=0.00..820.00 rows=20000 width=24)
                                      (actual time=0.012..9.442 rows=20000 loops=1)
              Buffers: shared hit=323

Planning Time: 1.234 ms
Execution Time: 68.102 ms

Let me walk through each piece.

Cost estimates: `cost=1284.00..4821.33`

The two numbers are startup cost and total cost. These are not milliseconds — they're abstract units PostgreSQL uses internally, based on things like seq_page_cost and cpu_tuple_cost. What matters is relative comparison between nodes. If a child node has a high cost relative to its parent, that's where you look first.

The estimate here is rows=3200. The actual was rows=3187. That's a great estimate — PostgreSQL knew what it was dealing with. When estimated and actual rows diverge wildly (say, estimated 10, actual 90000), you've got a statistics problem. Run ANALYZE orders; and check pg_stats for that column.

Actual time: `actual time=18.432..67.841`

Two numbers again — time to return the first row, and time to return the last row. For most queries you care about total time, but for streaming or cursor-based queries, startup time matters a lot.

Loops

The loops=1 value tells you how many times this node executed. In a nested loop join, an inner node might show loops=5000. The actual time shown is per loop, so you need to multiply: if it says actual time=0.5..1.2 rows=10 loops=5000, the real cost is 6,000ms just on that node. This trips up even experienced engineers.

Buffers: `shared hit=1423 read=287`

This is gold. shared hit means data came from PostgreSQL's shared buffer cache (fast). read means it went to disk (slow). In this example, about 17% of pages came from disk. If you're seeing high read counts on a frequently-queried table, your shared_buffers setting might be too low, or you need better indexing to reduce the number of pages touched altogether.

You might also see written (dirty pages flushed) or dirtied — those usually signal something more nuanced, like a lot of HOT updates happening in the background.

What to fix in this specific query

Looking at our output: Rows Removed by Filter: 96813 on the sequential scan is a red flag. We scanned 100,000 rows to get 3,187. An index on (status, created_at) would turn this into an index scan and likely cut the execution time by 80%.

CREATE INDEX CONCURRENTLY idx_orders_status_created
ON orders (status, created_at DESC)
WHERE status = 'pending';

The WHERE status = 'pending' makes it a partial index — smaller, faster, and exactly targeted at this query pattern.

After creating that index, re-run your EXPLAIN ANALYZE. The seq scan should become an Index Scan, read buffers should drop significantly, and you'll see the actual row count match much earlier in the plan.
