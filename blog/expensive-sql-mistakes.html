# The 7 Most Expensive SQL Mistakes DBAs See Every Day

*By Rajesh Guntupalli*

---

After years of tuning queries across PostgreSQL and SQL Server environments, I keep seeing the same mistakes. Not obscure edge cases — the same anti-patterns, over and over, written by developers who genuinely don't know they're torching performance. Some of these can take a query from 200ms to 45 seconds. I've seen it happen in production.

Here are the seven that cost teams the most.

---

1. SELECT * in Production Queries

Everyone knows this is bad. Almost nobody fixes it.

SELECT * forces the engine to read every column, blows up your network transfer, prevents index-only scans, and breaks silently when schema changes. On a table with 40 columns and 10 million rows, you might be pulling 8GB of data when you only needed 3 columns.

Fix it. Always name your columns.

---

2. Functions on Indexed Columns in WHERE Clauses

This one kills indexes silently. If you wrap an indexed column in a function, the index is useless.

-- BAD: Full table scan, index on created_at is ignored
SELECT order_id, total
FROM orders
WHERE YEAR(created_at) = 2024;

-- GOOD: Index on created_at is used
SELECT order_id, total
FROM orders
WHERE created_at >= '2024-01-01'
  AND created_at < '2025-01-01';

In PostgreSQL, the same problem shows up with LOWER(), DATE(), TO_CHAR(). The optimizer can't use a B-tree index on email if you're querying WHERE LOWER(email) = 'test@example.com'. Use a functional index or fix the data at write time.

---

3. Implicit Cross Joins from Missing JOIN Conditions

This one is a production killer. One missing ON clause and you have a Cartesian product.

-- BAD: Returns customers * orders rows (accidental cross join)
SELECT c.name, o.total
FROM customers c, orders o
WHERE c.region = 'US';

-- GOOD: Explicit JOIN with proper condition
SELECT c.name, o.total
FROM customers c
INNER JOIN orders o ON o.customer_id = c.customer_id
WHERE c.region = 'US';

I once investigated a SQL Server query running for 20 minutes. A developer had added a second table to the FROM clause and forgot to add the join condition. The result set was 4.2 billion rows. The old-style comma-separated FROM syntax makes this far too easy to miss.

---

4. N+1 Queries from Application Code

This isn't just an ORM problem. I see raw SQL loops in stored procedures and application code constantly.

If your application runs one query to fetch 500 orders, then runs 500 individual queries to fetch the customer for each one — that's N+1. Replace it with a single JOIN or a batch IN () clause. On a remote database with 5ms round-trip latency, that's 2.5 seconds of pure network waste per request.

---

5. Missing Indexes on Foreign Keys

SQL Server doesn't automatically index foreign key columns. PostgreSQL doesn't either. If you're joining on orders.customer_id and there's no index on that column, every join is a sequential scan.

Run this in PostgreSQL to find foreign keys without indexes:

SELECT
    tc.table_name,
    kcu.column_name,
    ccu.table_name AS foreign_table
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
    ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.referential_constraints rc
    ON tc.constraint_name = rc.constraint_name
JOIN information_schema.key_column_usage ccu
    ON rc.unique_constraint_name = ccu.constraint_name
WHERE tc.constraint_type = 'FOREIGN KEY'
  AND NOT EXISTS (
    SELECT 1 FROM pg_indexes
    WHERE tablename = tc.table_name
      AND indexdef LIKE '%' || kcu.column_name || '%'
  );

Run this on any legacy schema and you'll usually find at least a few unindexed foreign keys.

---

6. Using OR Instead of UNION for Index Utilization

In many cases, OR across different columns prevents the optimizer from using indexes effectively. Splitting into UNION ALL lets each branch use its own index.

If you have indexes on both email and phone, a WHERE email = ? OR phone = ? query may degrade to a full scan. A UNION ALL of two targeted queries often runs 3–10x faster on large tables.

---

7. Ignoring Query Plan Warnings

Both PostgreSQL and SQL Server surface warnings directly in execution plans — implicit conversions, missing statistics, spills to disk. Most developers never look at the plan. They see that the query "works" and move on.

In SQL Server, look for yellow warning triangles in the graphical plan. In PostgreSQL, run EXPLAIN (ANALYZE, BUFFERS) and look for high Buffers: shared hit numbers, sequential scans on large tables, or nested loop joins on large result sets. These are the plan telling you exactly where the problem is.
