Chapter 28: Containerization

Running databases in containers was once considered a risky experiment — something you did for development environments and testing pipelines, not for anything you cared about. That perception has changed substantially. Today, containerized PostgreSQL and SQL Server instances run in production at companies of every size, managed by orchestration platforms like Kubernetes, backed by persistent storage, monitored with the same rigor as bare-metal systems, and capable of surviving node failures gracefully. This chapter covers the full arc: what containers are, why databases present unique challenges inside them, how to run PostgreSQL and SQL Server correctly in Docker and Kubernetes, how to handle storage and networking, and how to operate containerized databases in production without losing sleep.

---

Why Containers and Why Now

Containers are lightweight, isolated runtime environments that package an application together with everything it needs to run — libraries, configuration files, binaries — into a single portable image. Unlike virtual machines, containers share the host operating system kernel, which makes them fast to start and cheap on resources. Docker popularized the container format, and the Open Container Initiative (OCI) standardized it so that images built with Docker run equally well under containerd, Podman, or any OCI-compliant runtime.

For databases, the appeal is real. A containerized database image is reproducible. The same image you tested last Tuesday is the exact image running in production on Friday — same PostgreSQL version, same shared libraries, same locale settings. Deployment becomes a configuration problem rather than an installation problem. When a new team member joins, they run docker compose up and have a fully functional database environment within seconds, not hours.

The deeper value emerges in orchestrated environments. Kubernetes can restart a failed database container automatically, schedule it onto healthy nodes, apply resource limits so a runaway query doesn't starve the host, and roll out configuration changes without manual intervention. For organizations running dozens or hundreds of database instances, the operational leverage is significant.

None of this comes free. Databases are stateful. Containers are designed to be ephemeral. Bridging that mismatch — making containers behave reliably for workloads that absolutely cannot lose data — is where most of the complexity lives, and where this chapter spends most of its time.

Running PostgreSQL and SQL Server in Docker

Docker is the natural starting point. Even if your production environment is Kubernetes, you will work with Docker images constantly — building them, inspecting them, testing configurations locally.

The official PostgreSQL image is maintained by the Docker community and available on Docker Hub as postgres. Microsoft maintains the SQL Server image under mcr.microsoft.com/mssql/server. Both follow the same basic pattern: pull an image, provide environment variables for initialization, mount a volume for data persistence.

A minimal PostgreSQL container:

docker run -d \
  --name pg-prod \
  -e POSTGRES_USER=appuser \
  -e POSTGRES_PASSWORD=strongpassword \
  -e POSTGRES_DB=appdb \
  -v pgdata:/var/lib/postgresql/data \
  -p 5432:5432 \
  postgres:16

A minimal SQL Server container:

docker run -d \
  --name mssql-prod \
  -e ACCEPT_EULA=Y \
  -e MSSQL_SA_PASSWORD=StrongP@ssword1 \
  -e MSSQL_PID=Developer \
  -v mssqldata:/var/opt/mssql \
  -p 1433:1433 \
  mcr.microsoft.com/mssql/server:2022-latest

The -v flag mounts a named Docker volume. Without it, the database files live inside the container's writable layer, which disappears the moment the container is removed. Named volumes persist independently of container lifecycle — a distinction that cannot be overstated.

Once a container is running, you connect and manage it exactly as you would any other instance. The database doesn't know it's in a container.

PostgreSQL:

-- Check PostgreSQL version and data directory inside container
SELECT version();
SHOW data_directory;

SQL Server:

-- Check SQL Server version and data directory inside container
SELECT @@VERSION;
SELECT physical_name
FROM sys.master_files
WHERE database_id = DB_ID('master');

Docker Compose becomes essential as soon as you add more than one service. A typical development stack includes a database, an application server, and perhaps a caching layer. Compose lets you define all of them in a single YAML file and manage them together.

# docker-compose.yml for PostgreSQL
version: "3.9"
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_USER: appuser
      POSTGRES_PASSWORD: strongpassword
      POSTGRES_DB: appdb
    volumes:
      - pgdata:/var/lib/postgresql/data
    ports:
      - "5432:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U appuser -d appdb"]
      interval: 10s
      timeout: 5s
      retries: 5

volumes:
  pgdata:

The healthcheck entry is not cosmetic. Compose and Kubernetes both use health checks to know when a container is genuinely ready to accept connections, not just running as a process. For a database, there is a meaningful gap between "process started" and "accepting queries."

Persistent Storage: The Non-Negotiable Requirement

Storage is where containerized database deployments succeed or fail. Understanding the options and their trade-offs is not optional knowledge — it's the foundation every other decision rests on.

Docker volumes are the default recommendation for most scenarios. They are managed by Docker, stored outside the container filesystem, survive container removal, and can be backed up with standard tooling. When you declare pgdata:/var/lib/postgresql/data, Docker creates a volume at a path like /var/lib/docker/volumes/pgdata/_data on the host and mounts it into the container.

Bind mounts map a specific host directory into the container. They give you direct visibility into the files from the host:

docker run -d \
  --name pg-dev \
  -e POSTGRES_USER=devuser \
  -e POSTGRES_PASSWORD=devpass \
  -e POSTGRES_DB=devdb \
  -v /data/postgres:/var/lib/postgresql/data \
  postgres:16

Bind mounts are convenient for development and for situations where your storage team provisions specific mount points. In production Kubernetes, bind mounts give way to Persistent Volumes.

Kubernetes introduces PersistentVolume (PV) and PersistentVolumeClaim (PVC) as abstractions over physical storage. A PVC requests storage of a specific size and access mode, and Kubernetes satisfies it by binding a matching PV — which could be backed by AWS EBS, Azure Disk, GCP Persistent Disk, NFS, or a local SSD. The StatefulSet controller, which is the right workload type for databases, automatically creates and manages PVCs for each database pod.

The access mode matters enormously for databases. ReadWriteOnce (RWO) means only one node can mount the volume at a time in read-write mode — correct for a single database instance. ReadWriteMany (RWX) allows multiple nodes to mount simultaneously, which sounds useful but is almost never appropriate for a database data directory. Two PostgreSQL instances writing to the same data directory simultaneously is not high availability — it is corruption.

I/O performance from container storage varies widely. A database on local NVMe storage gets very different throughput than one writing to a network-attached volume over iSCSI. When evaluating storage options, measure directly. Run fio benchmarks against the actual storage class you intend to use before committing to it for a production workload.

PostgreSQL:

-- Check I/O statistics inside a running PostgreSQL container
SELECT
    datname,
    blks_read,
    blks_hit,
    ROUND(blks_hit::numeric / NULLIF(blks_read + blks_hit, 0) * 100, 2) AS cache_hit_ratio,
    tup_returned,
    tup_fetched
FROM pg_stat_database
WHERE datname NOT IN ('template0', 'template1');

SQL Server:

-- Check I/O statistics inside a running SQL Server container
SELECT
    DB_NAME(vfs.database_id) AS database_name,
    vfs.io_stall_read_ms,
    vfs.io_stall_write_ms,
    vfs.num_of_reads,
    vfs.num_of_writes,
    mf.physical_name
FROM sys.dm_io_virtual_file_stats(NULL, NULL) vfs
JOIN sys.master_files mf
    ON vfs.database_id = mf.database_id
    AND vfs.file_id = mf.file_id
ORDER BY vfs.io_stall_read_ms + vfs.io_stall_write_ms DESC;

One configuration detail that catches teams off guard: PostgreSQL's data_checksums and fsync settings. When running inside a container, the database is relying on the storage driver and volume backend to honor fsync calls faithfully. Most production storage backends do. Some cloud-managed NFS implementations have quirks. Always verify that your storage backend flushes durably on fsync before disabling it for performance reasons — the result of getting that wrong is a data directory that looks intact but has corrupted pages after an unexpected shutdown.

Configuration Management for Containerized Databases

Running a database in a container without customizing its configuration means accepting defaults, which are rarely appropriate for production workloads. The challenge is doing configuration correctly within the container model, where immutable images and dynamic environments are the norm.

PostgreSQL reads postgresql.conf and pg_hba.conf at startup. There are three approaches to customizing them in containers.

The first approach uses environment variables that the official Docker entrypoint translates into configuration. The PostgreSQL image supports passing any configuration parameter using the POSTGRES_ prefix convention, or by appending parameters directly to the postgres command:

docker run -d \
  --name pg-tuned \
  -e POSTGRES_USER=appuser \
  -e POSTGRES_PASSWORD=strongpassword \
  -v pgdata:/var/lib/postgresql/data \
  postgres:16 \
  -c shared_buffers=2GB \
  -c work_mem=64MB \
  -c max_connections=200 \
  -c wal_level=replica \
  -c max_wal_senders=5

The second approach mounts a custom configuration file as a volume:

docker run -d \
  --name pg-custom \
  -v /etc/postgres/postgresql.conf:/etc/postgresql/postgresql.conf \
  -v pgdata:/var/lib/postgresql/data \
  postgres:16 \
  -c config_file=/etc/postgresql/postgresql.conf

The third approach builds a custom image with the configuration baked in — appropriate when you want the configuration version-controlled alongside the image:

FROM postgres:16
COPY postgresql.conf /etc/postgresql/postgresql.conf
COPY pg_hba.conf /etc/postgresql/pg_hba.conf

In Kubernetes, ConfigMaps and Secrets serve this role. ConfigMaps hold non-sensitive configuration like postgresql.conf. Secrets hold credentials. Both are mounted into pods as files or environment variables, keeping sensitive data out of images and out of source control.

For SQL Server, configuration works through environment variables and mssql.conf, the SQL Server configuration file that lives at /var/opt/mssql/mssql.conf inside the container. You can mount a custom mssql.conf the same way you would mount a PostgreSQL config file.

PostgreSQL:

-- Verify configuration settings applied at startup
SELECT
    name,
    setting,
    unit,
    source,
    sourcefile
FROM pg_settings
WHERE name IN (
    'shared_buffers',
    'work_mem',
    'max_connections',
    'wal_level',
    'checkpoint_completion_target'
)
ORDER BY name;

SQL Server:

-- Verify SQL Server configuration settings
SELECT
    name,
    value,
    value_in_use,
    description
FROM sys.configurations
WHERE name IN (
    'max degree of parallelism',
    'cost threshold for parallelism',
    'max server memory (MB)',
    'min server memory (MB)'
)
ORDER BY name;

Memory limits require special attention. When you set --memory=8g on a Docker container or a resources.limits.memory in a Kubernetes pod spec, the container's memory is capped at that value. PostgreSQL's shared_buffers should typically be set to 25% of the memory limit, not 25% of the host's total RAM. If you let PostgreSQL inherit defaults from a host with 256 GB of RAM while your container is limited to 8 GB, the configuration won't match reality, and you will encounter OOM kills. The same logic applies to SQL Server's max server memory setting — it must be configured to stay within the container's memory limit.

Kubernetes Orchestration for Database Workloads

Kubernetes is where containerized databases get serious. It provides automated scheduling, self-healing, resource management, rolling upgrades, and service discovery — capabilities that matter enormously when you're running databases at scale. It also introduces enough complexity that understanding the database-specific concerns is essential before going to production.

StatefulSets are the right Kubernetes workload type for databases. Unlike Deployments, StatefulSets give each pod a stable network identity and a persistent volume that follows it. When a pod named postgres-0 restarts, Kubernetes ensures it gets the same PVC it had before. The pod name is stable and predictable, which matters for primary-replica configurations.

A minimal StatefulSet for PostgreSQL:

apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: "postgres"
  replicas: 1
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:16
          env:
            - name: POSTGRES_USER
              valueFrom:
                secretKeyRef:
                  name: postgres-secret
                  key: username
            - name: POSTGRES_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: postgres-secret
                  key: password
            - name: POSTGRES_DB
              value: appdb
          resources:
            requests:
              memory: "4Gi"
              cpu: "2"
            limits:
              memory: "8Gi"
              cpu: "4"
          volumeMounts:
            - name: pgdata
              mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:
    - metadata:
        name: pgdata
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: "fast-ssd"
        resources:
          requests:
            storage: 100Gi

The volumeClaimTemplates section is what separates StatefulSets from Deployments for database use. Each replica gets its own dedicated PVC. For a primary-only deployment, there is one PVC. For a three-node replica set, there are three PVCs, each attached exclusively to its pod.

Services in Kubernetes expose database pods to other workloads. A headless service (with clusterIP: None) allows direct DNS-based resolution to individual pod addresses, which replication configurations use to identify specific peers. A standard ClusterIP service load-balances across all matching pods — appropriate for read replicas but not for writes if you have multiple instances.

Liveness and readiness probes replace the healthcheck you define in Docker Compose. A readiness probe controls when a pod receives traffic. A liveness probe controls when Kubernetes restarts an unhealthy pod:

livenessProbe:
  exec:
    command:
      - pg_isready
      - -U
      - appuser
      - -d
      - appdb
  initialDelaySeconds: 30
  periodSeconds: 10
  failureThreshold: 3
readinessProbe:
  exec:
    command:
      - pg_isready
      - -U
      - appuser
      - -d
      - appdb
  initialDelaySeconds: 5
  periodSeconds: 5

The initialDelaySeconds on the liveness probe deserves thought. PostgreSQL running crash recovery after an unexpected shutdown may take minutes to complete before accepting connections. If the liveness probe fires before recovery finishes and restarts the pod, it will interrupt recovery, causing the pod to loop indefinitely. Set initialDelaySeconds conservatively — longer than the worst-case recovery time for your workload.

Operators extend Kubernetes with database-specific intelligence. The CloudNativePG operator for PostgreSQL and the SQL Server Operator for Kubernetes from Microsoft both automate tasks that would otherwise require manual intervention: failover, replica provisioning, backup scheduling, configuration updates. Operators encode operational runbooks as code, which is exactly what large-scale database management needs.

PostgreSQL:

-- Inside a Kubernetes-hosted PostgreSQL, check replication status
SELECT
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    (sent_lsn - replay_lsn) AS replication_lag_bytes
FROM pg_stat_replication;

SQL Server:

-- Inside a Kubernetes-hosted SQL Server, check AG replica status
SELECT
    ar.replica_server_name,
    rs.role_desc,
    rs.operational_state_desc,
    rs.connected_state_desc,
    rs.synchronization_health_desc,
    ars.log_send_queue_size,
    ars.redo_queue_size
FROM sys.availability_replicas ar
JOIN sys.dm_hadr_availability_replica_states rs
    ON ar.replica_id = rs.replica_id
JOIN sys.dm_hadr_database_replica_states ars
    ON ar.replica_id = ars.replica_id;

Production Operations: Backups, Upgrades, and Observability

Getting a containerized database running is one challenge. Keeping it running reliably over months and years — while upgrading it, backing it up, and troubleshooting it when problems arise — is the sustained work of production operations.

Backups in containerized databases require connecting the backup tooling to both the database process and the underlying persistent volume. The container abstraction doesn't change what constitutes a valid backup, but it does change how you execute one.

For PostgreSQL, pg_dump and pg_basebackup run inside the container via docker exec or Kubernetes exec. Alternatively, a sidecar container — a second container in the same pod sharing the same volume mount — can run backup tools continuously without interfering with the primary container.

# Run pg_dump from inside a running container
docker exec -t pg-prod pg_dump -U appuser -d appdb -Fc > /backups/appdb_$(date +%Y%m%d).dump

# Run pg_basebackup for a physical backup
docker exec -t pg-prod pg_basebackup -U replicator -D /backups/base -Ft -z -P

For Kubernetes, backup CronJobs are a common pattern — a scheduled pod that runs, performs the backup, ships the result to object storage (S3, Azure Blob, GCS), and exits. CloudNativePG manages this automatically if you configure it with a backup policy.

Major version upgrades of containerized databases are operationally cleaner than bare-metal upgrades in one important way: you can run the old and new versions side by side with separate containers, migrate data between them, and cut over with minimal downtime. For PostgreSQL major upgrades (e.g., 15 to 16), pg_upgrade is still the right tool, but running it in a container requires mounting both the old and new data directories:

docker run --rm \
  -v pgdata_old:/var/lib/postgresql/15/data \
  -v pgdata_new:/var/lib/postgresql/16/data \
  tianon/postgres-upgrade:15-to-16 \
  --link

The --link flag runs pg_upgrade in link mode, which rewrites the data in place rather than copying it — dramatically faster for large databases.

Observability inside containers uses the same tools as observability outside them, with the addition of container-level metrics. Prometheus with the postgres_exporter sidecar collects database metrics and exposes them for scraping. Grafana dashboards built on top of those metrics give you the same visibility you'd have on bare metal. For SQL Server, the mssql_exporter serves the same function.

Resource metrics at the container level — CPU usage, memory consumption, network I/O — come from cAdvisor, which Kubernetes ships as part of the node agent. Correlating database-level metrics (query latency, lock waits, buffer cache hit ratios) with container-level metrics (memory near the limit, CPU throttling) gives you the full picture when diagnosing performance problems.

CPU throttling is a particularly common issue for database containers. When a container's CPU usage hits the cpu.limits value, the kernel throttles it — slices of CPU time are withheld. A database that looks fine from a query perspective but shows elevated latency inconsistently may be experiencing CPU throttling. Check the container_cpu_cfs_throttled_seconds_total metric in Prometheus before assuming the problem is a slow query.

PostgreSQL:

-- Check table and index bloat inside a production container instance
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname || '.' || tablename)) AS total_size,
    pg_size_pretty(pg_relation_size(schemaname || '.' || tablename)) AS table_size,
    pg_size_pretty(
        pg_total_relation_size(schemaname || '.' || tablename) -
        pg_relation_size(schemaname || '.' || tablename)
    ) AS index_size
FROM pg_tables
WHERE schemaname NOT IN ('pg_catalog', 'information_schema')
ORDER BY pg_total_relation_size(schemaname || '.' || tablename) DESC
LIMIT 20;

SQL Server:

-- Check index fragmentation inside a production SQL Server container
SELECT
    DB_NAME() AS database_name,
    OBJECT_NAME(ips.object_id) AS table_name,
    i.name AS index_name,
    ips.index_type_desc,
    ROUND(ips.avg_fragmentation_in_percent, 2) AS fragmentation_pct,
    ips.page_count
FROM sys.dm_db_index_physical_stats(
    DB_ID(), NULL, NULL, NULL, 'LIMITED'
) ips
JOIN sys.indexes i
    ON ips.object_id = i.object_id
    AND ips.index_id = i.index_id
WHERE ips.avg_fragmentation_in_percent > 10
    AND ips.page_count > 1000
ORDER BY ips.avg_fragmentation_in_percent DESC;

Log management in containers follows the principle that logs should go to stdout and stderr, where the container runtime captures them. For PostgreSQL, setting log_destination = 'stderr' and logging_collector = off directs logs to stdout, where Docker or Kubernetes picks them up. From there, a log aggregator like Fluentd, Loki, or CloudWatch Logs collects and indexes them. For SQL Server, the error log writes to /var/opt/mssql/log/errorlog by default — a volume-backed path that persists across container restarts.
