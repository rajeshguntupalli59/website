Chapter 8: Security Foundations

Security is the layer of your database infrastructure that, when done right, nobody notices — and when done wrong, becomes the headline. Whether you're managing a healthcare system bound by HIPAA requirements, a financial platform subject to PCI-DSS audits, or a startup that just doesn't want its customer data on Pastebin, the principles are the same: control who gets in, control what they can do once they're inside, and know what happened after the fact. This chapter walks through the foundational security model of both PostgreSQL and SQL Server — from authentication and authorization through privilege management, row-level security, auditing, and encryption at rest and in transit. The goal is to give you the mental model and the practical tools to build a defensible database environment, not just one that passes a checkbox audit.

---

How Authentication Works — and Where It Can Go Wrong

Authentication answers a single question: are you who you claim to be? Both PostgreSQL and SQL Server have layered authentication systems, but they approach that question differently.

PostgreSQL's authentication is controlled by a file called pg_hba.conf — Host-Based Authentication configuration. This file is read top-to-bottom, and the first rule that matches a connection request is applied. Each line specifies a connection type (local socket, host, hostssl, hostnossl), a database, a user, an address range, and an authentication method. If you've ever been confused about why a password you're certain is correct keeps getting rejected, the answer is almost always in pg_hba.conf.

The authentication methods available in PostgreSQL include trust (no password required — frightening in production), md5 and scram-sha-256 (password-based), peer (matches OS username to database username for local connections), ldap, gss, radius, and cert (SSL client certificate). In modern PostgreSQL deployments, scram-sha-256 is preferred over md5 because MD5 is cryptographically broken and transmits a hash that can be replayed. You should audit your pg_hba.conf for any remaining md5 entries and update them.

SQL Server takes a different approach with two authentication modes: Windows Authentication (also called Integrated Security) and SQL Server Authentication (username and password stored in SQL Server itself). Mixed Mode enables both. In domain-joined environments, Windows Authentication is the stronger choice — credentials are managed by Active Directory, password policies are enforced by Group Policy, and Kerberos or NTLM handles the handshake without passwords appearing in connection strings. SQL Server Authentication has its place — cloud environments, cross-domain connections, application service accounts that can't use Kerberos — but it puts the burden of credential management back on the DBA.

A common production mistake in both systems is leaving the highest-privileged accounts reachable from the network. In PostgreSQL, the postgres superuser should either have its pg_hba.conf entry restricted to local connections only or be protected by certificate authentication. In SQL Server, the sa login should be disabled outright in most production environments — AD service accounts and application logins should carry only the privileges they need.

You can inspect active authentication configuration in both systems:

PostgreSQL:

-- View pg_hba.conf rules as seen by PostgreSQL
SELECT type, database, user_name, address, auth_method
FROM pg_hba_file_rules;

-- Check which authentication method a current session used
SELECT usename, application_name, client_addr, auth_method
FROM pg_stat_ssl
JOIN pg_stat_activity USING (pid)
WHERE ssl = true;

SQL Server:

-- Check the server authentication mode (1 = Windows only, 2 = Mixed)
SELECT SERVERPROPERTY('IsIntegratedSecurityOnly') AS windows_auth_only;

-- Review all logins, their type, and whether they are enabled
SELECT name,
       type_desc,
       is_disabled,
       create_date,
       modify_date
FROM sys.server_principals
WHERE type IN ('S', 'U', 'G')  -- SQL login, Windows user, Windows group
ORDER BY type_desc, name;

One trap worth calling out explicitly: in PostgreSQL, setting auth_method to trust for a local socket connection might seem harmless because you assume only OS processes can reach the socket. But if your application runs on the same server as the database and an attacker achieves code execution on that host, trust means they walk straight in. The principle is defense in depth — even local connections should require authentication in sensitive environments.

Users, Roles, and the Principle of Least Privilege

Once a connection is authenticated, the database needs to know what that identity is allowed to do. Both PostgreSQL and SQL Server use a role-based model, but the terminology and mechanics differ enough to cause genuine confusion if you're switching between the two.

In PostgreSQL, everything is a role. There is no separate concept of a "user" versus a "group" — a role can log in (making it effectively a user), own objects, hold other roles as members, and carry attributes like SUPERUSER, CREATEDB, CREATEROLE, and REPLICATION. When you run CREATE USER, PostgreSQL simply creates a role with LOGIN attribute enabled. When you run CREATE GROUP, it creates a role with NOLOGIN. The inheritance model means that when role A is granted to role B, B automatically inherits A's privileges — unless the role is created with NOINHERIT, which forces B to explicitly SET ROLE A before exercising those privileges.

SQL Server separates server-level principals (logins) from database-level principals (users). A login authenticates at the server. A database user maps to a login (or is a contained user, or maps to a certificate or asymmetric key) and holds permissions within a specific database. This two-tier model gives you granular control — a login can exist at the server level without having access to any individual database, or can be mapped to a guest account, or can be given different database-level identities in different databases.

The practical implication is that SQL Server requires two steps to give someone database access: create the login, then create the user in the target database and map it to that login. PostgreSQL has one step — create the role with appropriate attributes and grant it permissions.

PostgreSQL:

-- Create an application role with login (no superuser privileges)
CREATE ROLE app_user LOGIN PASSWORD 'use_a_strong_password_here';

-- Create a functional role that will hold object permissions
CREATE ROLE app_readonly NOLOGIN;

-- Grant the functional role to the application role
GRANT app_readonly TO app_user;

-- Grant specific permissions on a schema and its tables
GRANT USAGE ON SCHEMA orders TO app_readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA orders TO app_readonly;

-- Ensure future tables in the schema are also covered
ALTER DEFAULT PRIVILEGES IN SCHEMA orders
  GRANT SELECT ON TABLES TO app_readonly;

SQL Server:

-- Step 1: Create a server-level login
CREATE LOGIN app_user WITH PASSWORD = 'use_a_strong_password_here';

-- Step 2: In the target database, create a user mapped to that login
USE OrdersDB;
CREATE USER app_user FOR LOGIN app_user;

-- Create a database role to hold permissions
CREATE ROLE app_readonly;

-- Assign the user to the role
ALTER ROLE app_readonly ADD MEMBER app_user;

-- Grant the role read access to a schema
GRANT SELECT ON SCHEMA::orders TO app_readonly;

The principle of least privilege is not just a compliance checkbox — it's practical defense. An application that only reads data from a reporting schema should not hold INSERT, UPDATE, or DELETE on that schema, and it certainly should not hold schema ownership or DDL privileges. In practice, DBAs often find that developers were given db_owner during development and that role was never narrowed before production deployment. Auditing existing privileges is the first step to correcting this.

In PostgreSQL, privilege visibility comes from the system catalog and information schema. In SQL Server, the sys.database_permissions and fn_my_permissions views are your tools.

PostgreSQL:

-- See all privileges on tables in a given schema
SELECT grantee, table_schema, table_name, privilege_type
FROM information_schema.role_table_grants
WHERE table_schema = 'orders'
ORDER BY table_name, grantee, privilege_type;

-- Check effective role memberships
SELECT r.rolname AS role,
       m.rolname AS member
FROM pg_auth_members am
JOIN pg_roles r ON r.oid = am.roleid
JOIN pg_roles m ON m.oid = am.member
ORDER BY role, member;

SQL Server:

-- List all explicit permissions in the current database
SELECT dp.name         AS principal,
       dp.type_desc    AS principal_type,
       perm.class_desc AS object_class,
       perm.permission_name,
       perm.state_desc AS grant_state,
       OBJECT_NAME(perm.major_id) AS object_name
FROM sys.database_permissions perm
JOIN sys.database_principals dp
  ON dp.principal_id = perm.grantee_principal_id
WHERE dp.name NOT IN ('public', 'sys', 'INFORMATION_SCHEMA')
ORDER BY dp.name, perm.class_desc, object_name;

One pattern worth institutionalizing: never grant permissions directly to individual users. Always grant to roles, and assign users to roles. This makes permission management auditable and maintainable — when an employee changes teams or leaves, you change role membership, not dozens of individual permission grants scattered across dozens of objects.

Row-Level Security — Controlling What Data Each User Sees

Privilege grants at the table level are coarse instruments. They answer the question "can this role access this table?" but they can't answer "can this role access *these rows* in this table?" Row-Level Security (RLS) answers that second question, and it does so in a way that is enforced by the database engine itself — not by application code, which can be bypassed.

Both PostgreSQL and SQL Server support RLS, and both work on the same conceptual model: you define a policy that adds a filter expression to queries against a table. When a user queries the table, the database engine automatically appends that filter condition, as though the user had written it in their WHERE clause — except they can't remove it.

In PostgreSQL, RLS must be explicitly enabled on a table. Table owners bypass RLS by default (superusers do too, unless you use SET row_security = force). The policy definition specifies whether it applies to SELECT, INSERT, UPDATE, DELETE, or ALL commands, and defines a USING expression for read policies and a WITH CHECK expression for write policies.

A canonical example: a multi-tenant SaaS platform where one table stores orders for all tenants. Each tenant's application authenticates as a dedicated database role, and you want to guarantee at the database level that Tenant A can never read Tenant B's rows.

PostgreSQL:

-- Enable RLS on the orders table
ALTER TABLE orders.line_items ENABLE ROW LEVEL SECURITY;

-- Create a policy that filters rows by tenant identifier
-- current_user matches the PostgreSQL role name
CREATE POLICY tenant_isolation ON orders.line_items
  USING (tenant_id = current_user::text);

-- For write operations, also enforce the check on new rows
CREATE POLICY tenant_isolation_insert ON orders.line_items
  FOR INSERT
  WITH CHECK (tenant_id = current_user::text);

-- Confirm RLS is enabled on the table
SELECT tablename, rowsecurity, forcerowsecurity
FROM pg_tables
WHERE schemaname = 'orders' AND tablename = 'line_items';

SQL Server:

-- Create a schema to hold the security predicate function
CREATE SCHEMA Security;
GO

-- The predicate function returns 1 (allow) or 0 (deny)
CREATE FUNCTION Security.fn_tenant_predicate (@tenant_id NVARCHAR(100))
RETURNS TABLE
WITH SCHEMABINDING
AS
RETURN SELECT 1 AS fn_result
       WHERE @tenant_id = USER_NAME();
GO

-- Create a security policy binding the predicate to the table
CREATE SECURITY POLICY TenantIsolationPolicy
  ADD FILTER PREDICATE Security.fn_tenant_predicate(tenant_id)
    ON orders.line_items,
  ADD BLOCK PREDICATE Security.fn_tenant_predicate(tenant_id)
    ON orders.line_items AFTER INSERT
WITH (STATE = ON);
GO

SQL Server's approach separates the predicate logic (an inline table-valued function) from the policy binding, which gives you more flexibility — you can update the function without recreating the policy — but it also means you have an additional object to manage and secure. The function itself must be protected from modification by anyone other than the security administrator.

A nuance in PostgreSQL's RLS: if you have a role that needs to bypass RLS for administrative purposes (running ETL, batch updates, producing audit reports), you can use SET row_security = off — but this requires the BYPASSRLS role attribute, which should be granted sparingly. For application roles, the policy is always enforced.

Performance is a legitimate concern with RLS. Every query against a protected table gets the predicate joined in, and if the predicate column isn't indexed, you're doing full table scans on every single row access. Index the columns referenced in your RLS predicates. In a multi-tenant scenario, a composite index on (tenant_id, created_at) or similar is not optional — it's required for acceptable performance at scale.

Auditing — Building a Trail You Can Actually Use

Knowing what's happening in your database right now is important. Knowing what happened three weeks ago when a data anomaly is discovered is irreplaceable. Auditing is the discipline of recording who did what, to which objects, at what time — and storing that record in a way that can't be quietly altered.

PostgreSQL doesn't ship with a built-in audit framework. The native logging controlled by postgresql.conf can capture statements (log_statement = 'all'), connection events (log_connections, log_disconnections), and DDL. This is a starting point, but it's not a proper audit trail — logs are sequential text files, they're not queryable, they don't have per-table granularity, and they can be rotated away. For production audit requirements, the standard approach is the pgaudit extension, which integrates with PostgreSQL's logging infrastructure but provides object-level and statement-level audit categories with proper session context.

After installing pgaudit, you configure it via postgresql.conf (for session-level auditing) or via the role attribute pgaudit.role (for object-level auditing of specific tables or columns):

PostgreSQL:

-- In postgresql.conf:
-- shared_preload_libraries = 'pgaudit'
-- pgaudit.log = 'write, ddl'  -- logs INSERT/UPDATE/DELETE/TRUNCATE and DDL

-- Enable object-level auditing via a dedicated audit role
CREATE ROLE auditor NOLOGIN;

-- Grant SELECT on tables you want to audit reads on
GRANT SELECT ON orders.line_items TO auditor;

-- Tell pgaudit to use this role as the audit marker role
-- In postgresql.conf: pgaudit.role = 'auditor'

-- Confirm extension is installed
SELECT * FROM pg_extension WHERE extname = 'pgaudit';

SQL Server:

-- SQL Server has a built-in Server Audit mechanism

-- Step 1: Create a server audit (writes to Windows Event Log here,
--         but file destination is common for compliance)
CREATE SERVER AUDIT OrdersDBAudit
  TO APPLICATION_LOG
  WITH (QUEUE_DELAY = 1000, ON_FAILURE = CONTINUE);
GO

-- Step 2: Create a database audit specification
USE OrdersDB;
CREATE DATABASE AUDIT SPECIFICATION AuditLineItemAccess
  FOR SERVER AUDIT OrdersDBAudit
  ADD (SELECT, INSERT, UPDATE, DELETE
       ON orders.line_items BY public)
WITH (STATE = ON);
GO

-- Step 3: Enable the server audit
ALTER SERVER AUDIT OrdersDBAudit WITH (STATE = ON);
GO

-- Query the audit log
SELECT event_time, action_id, succeeded,
       server_principal_name, database_name,
       object_name, statement
FROM sys.fn_get_audit_file
  ('C:\AuditLogs\OrdersDBAudit_*.sqlaudit', DEFAULT, DEFAULT)
ORDER BY event_time DESC;

SQL Server's audit system is more fully integrated — it ships in the box, writes structured records to queryable files, supports both server-level and database-level specifications, and can write to Windows Event Log, Security Log, or binary files. The binary file format is tamper-evident by design: SQL Server uses a hash chain across audit records so any deletion or modification of records is detectable.

Several things are worth being deliberate about when designing an audit strategy. First, audit data should not live on the same server as the database it's auditing — an attacker who compromises the database server should not be able to silently wipe the evidence. Ship audit logs to a centralized SIEM or append-only log store. Second, be thoughtful about what you audit — logging every SELECT in a high-throughput OLTP system can generate enough volume to degrade performance and to make the logs themselves unmanageable. Audit writes and DDL universally; audit reads selectively based on data sensitivity classification. Third, establish a retention policy that meets your regulatory requirements before you need it — discovering during an incident response that logs only go back 14 days when the breach started 30 days ago is a particularly unpleasant experience.

Encryption — Protecting Data in Transit and at Rest

Authentication tells you who is connecting. Auditing tells you what they did. Encryption protects the data itself — from eavesdropping on the wire, from being read if storage media is stolen, and from unauthorized access to backup files. These are three distinct problems that require three distinct mechanisms.

Encryption in Transit

Both databases use TLS for encrypting connections. In PostgreSQL, TLS support is compiled in and enabled by setting ssl = on in postgresql.conf, providing a certificate and key, and configuring pg_hba.conf with hostssl entries. Client connections can be required to use SSL by setting sslmode=require (verifies the connection is encrypted) or sslmode=verify-full (verifies the connection is encrypted and that the server certificate is valid and matches the hostname). In production, verify-full is the right choice — require without verification is susceptible to man-in-the-middle attacks.

In SQL Server, TLS is enabled by obtaining a certificate, binding it to the SQL Server service via SQL Server Configuration Manager, and setting "Force Encryption" to Yes. Applications connect with Encrypt=True;TrustServerCertificate=False in the connection string — the TrustServerCertificate=False portion forces certificate validation, which is the difference between actual protection and theater.

Encryption at Rest

SQL Server's Transparent Data Encryption (TDE) encrypts the data files, log files, and backups at the storage level using a database encryption key (DEK) protected by a certificate or asymmetric key stored in the master database (or, in enterprise deployments, in Azure Key Vault or an HSM via Extensible Key Management). The application sees nothing different — all encryption and decryption happens in the buffer pool, transparent to queries.

SQL Server:

-- Check TDE status across all databases
SELECT db.name,
       dek.encryption_state,
       dek.encryption_state_desc,
       dek.key_algorithm,
       dek.key_length,
       dek.percent_complete
FROM sys.dm_database_encryption_keys dek
JOIN sys.databases db ON db.database_id = dek.database_id
ORDER BY db.name;

-- Enable TDE on a database (certificate must already exist in master)
USE master;
CREATE MASTER KEY ENCRYPTION BY PASSWORD = 'StrongPassphraseHere!';
CREATE CERTIFICATE TDECert WITH SUBJECT = 'TDE Certificate for OrdersDB';

USE OrdersDB;
CREATE DATABASE ENCRYPTION KEY
  WITH ALGORITHM = AES_256
  ENCRYPTION BY SERVER CERTIFICATE TDECert;
ALTER DATABASE OrdersDB SET ENCRYPTION ON;

PostgreSQL does not have an equivalent TDE implementation at the storage engine level (as of PostgreSQL 16, though development work is ongoing). The conventional approach is to use filesystem-level encryption — dm-crypt/LUKS on Linux, or BitLocker on Windows — which encrypts the entire volume containing the PostgreSQL data directory. This provides the same protection against physical media theft but is managed at the OS level rather than the database level. Some cloud-managed PostgreSQL offerings implement their own storage encryption transparently.

Column-Level Encryption

For highly sensitive fields — Social Security numbers, credit card numbers, medical identifiers — you may want encryption that is visible at the column level, so that even users with table access can't read the plaintext values unless they hold the key. Both systems support this, but the operational complexity is significantly higher.

PostgreSQL's pgcrypto extension provides pgp_sym_encrypt and pgp_sym_decrypt functions for symmetric encryption, and asymmetric variants for key management scenarios. SQL Server offers Always Encrypted, which keeps the column-level encryption keys on the client side — the database server itself never sees the plaintext, which protects against a compromised DBA or backup theft.

The trade-off is functionality: encrypted columns cannot be indexed in the normal sense, can't be used in WHERE clause comparisons without decrypting first (unless using randomized versus deterministic encryption carefully), and require application-level key management. These are real constraints that affect schema design, query patterns, and operational procedures. Column-level encryption is the right choice for specific high-value fields, not a blanket approach for entire tables.

Hardening a Database in a Real Production Environment

Understanding individual security mechanisms is necessary but not sufficient. A hardened production database is a system where all these mechanisms work together and where the configuration has been deliberately reviewed rather than left at defaults.

The defaults in both PostgreSQL and SQL Server are designed for broad compatibility and ease of getting started, not for security. The public schema in PostgreSQL historically granted CREATE to all users — a change made in PostgreSQL 15 restricts this, but environments on older versions may still carry it. In SQL Server, the public role has EXECUTE permission on a set of system stored procedures by default, which is a larger attack surface than most organizations realize.

A useful starting exercise is enumerating what's listening and who can reach it. Database ports (5432 for PostgreSQL, 1433 for SQL Server) should be reachable only from application servers, not from developer laptops, not from the general office network, and certainly not from the internet. Network-level controls and database-level authentication are separate layers — having both means an attacker must defeat two independent controls rather than one.

Superuser and sa-level accounts deserve special treatment. Their use should be exceptional, logged, and ideally require a break-glass procedure. In SQL Server, sa should be disabled. In PostgreSQL, postgres should have password authentication required even locally, and all routine administrative tasks should be performed as application-specific roles with only the necessary privileges.

Object ownership is another overlooked surface. In PostgreSQL, the role that owns a schema or table can drop it, alter it, or change its permissions — regardless of what other roles are configured. Application roles should not own production objects. A dedicated schema_owner role (which holds no login attribute) should own production schemas, and that role should be used only when DDL changes are made, not for routine application connections.

PostgreSQL:

-- Revoke the CREATE privilege from public on the public schema
-- (for PostgreSQL < 15, where this was granted by default)
REVOKE CREATE ON SCHEMA public FROM PUBLIC;

-- Review which roles have superuser privileges
SELECT rolname, rolsuper, rolinherit, rolcreaterole,
       rolcreatedb, rolcanlogin, rolbypassrls
FROM pg_roles
WHERE rolsuper = true OR rolcreaterole = true OR rolbypassrls = true
ORDER BY rolname;

SQL Server:

-- Identify server-level roles and their members
SELECT r.name AS server_role,
       m.name AS member_name,
       m.type_desc
FROM sys.server_role_members rm
JOIN sys.server_principals r ON r.principal_id = rm.role_principal_id
JOIN sys.server_principals m ON m.principal_id = rm.member_principal_id
WHERE r.name IN ('sysadmin', 'securityadmin', 'serveradmin')
ORDER BY r.name, m.name;

-- Check whether sa is disabled (it should be)
SELECT name, is_disabled
FROM sys.server_principals
WHERE name = 'sa';

Periodic privilege reviews should be formalized, not left to "we'll do it eventually." A quarterly review that produces a report of all roles, members, and permission grants — compared against the previous quarter — catches privilege creep before it becomes a liability
