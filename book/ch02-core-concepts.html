Chapter 2: Understanding the Core Concepts

Before you can write efficient queries, design reliable schemas, or troubleshoot production issues, you need a firm grip on how relational databases actually think. Both PostgreSQL and SQL Server are built on decades of relational theory, and while their syntax and internal mechanics differ in meaningful ways, they share the same foundational DNA. This chapter walks through the core concepts that every DBA and developer must internalize — from how relational data is organized, to how transactions protect your data, to how the two databases differ in architecture and behavior. Understanding these building blocks will make everything else in this book click into place.

---

How Relational Databases Organize Data

At its heart, a relational database stores data in tables — structured grids of rows and columns. Each table represents a single entity type: customers, orders, products. Each row is one instance of that entity, and each column holds a specific attribute of that entity. This sounds simple, and it is. The elegance of the relational model lies in what you can do with that simplicity.

The concept was formalized by Edgar F. Codd in 1970, and the rules he established — known as Codd's relational rules — are still the theoretical backbone of every major relational database today. The most important practical idea from that work is that data should be stored without redundancy and that relationships between entities should be expressed through keys, not through embedding one table's data inside another.

A primary key is a column or set of columns that uniquely identifies every row in a table. No two rows can share the same primary key value. In practice, most tables use either a surrogate key — an auto-generated integer or UUID with no business meaning — or a natural key, which is a value that already exists in the real world and is inherently unique, like a tax identification number.

A foreign key enforces a relationship between two tables. It says: the value in this column must exist as a primary key in that other table. When you have an orders table with a customer_id column, the foreign key constraint ensures you can never create an order that points to a nonexistent customer. This referential integrity is one of the most valuable guarantees a relational database provides — and one of the most commonly disabled by developers who don't understand the cost of losing it.

Normalization is the process of structuring tables to eliminate data redundancy and dependency anomalies. First Normal Form (1NF) requires that every column holds atomic values — no lists, no repeating groups. Second Normal Form (2NF) requires that every non-key column depends on the entire primary key, not just part of it. Third Normal Form (3NF) requires that non-key columns depend only on the primary key, not on each other. In practice, most production schemas target 3NF as a baseline, with strategic denormalization applied in specific places for performance reasons.

Data Types: Choosing the Right Tool for the Column

Every column in a relational table has a data type, and that choice matters more than most developers initially appreciate. The right data type enforces correctness at the storage level, reduces storage overhead, and enables the query engine to process data more efficiently.

PostgreSQL and SQL Server share many of the same conceptual data types but differ significantly in naming, behavior, and available options.

Integers are the most common data type for identifiers and counts. Both databases offer small, regular, and large integer variants.

PostgreSQL:

-- Integer variants in PostgreSQL
CREATE TABLE product_catalog (
    product_id    SERIAL PRIMARY KEY,       -- auto-incrementing 4-byte integer
    sku_code      SMALLINT,                 -- 2-byte integer, range -32768 to 32767
    warehouse_id  INTEGER,                  -- 4-byte integer
    total_units   BIGINT                    -- 8-byte integer for large counts
);

SQL Server:

-- Integer variants in SQL Server
CREATE TABLE product_catalog (
    product_id    INT IDENTITY(1,1) PRIMARY KEY,  -- auto-incrementing 4-byte integer
    sku_code      SMALLINT,                        -- 2-byte integer
    warehouse_id  INT,                             -- 4-byte integer
    total_units   BIGINT                           -- 8-byte integer for large counts
);

One of the most consequential data type choices you will make is how to store text. PostgreSQL's TEXT type is variable-length with no practical size limit and carries no performance penalty compared to VARCHAR(n). In SQL Server, VARCHAR(n) and NVARCHAR(n) are the standard choices. NVARCHAR stores Unicode data using two bytes per character and should be used whenever you store multilingual content. A common production mistake is using NVARCHAR(MAX) for every text column — this disables certain indexing optimizations and inflates storage unnecessarily.

Dates and timestamps are another area where the two databases diverge. PostgreSQL's TIMESTAMPTZ stores timestamps with full time zone awareness — it always converts to UTC internally and presents the value converted to the session's time zone. SQL Server's DATETIMEOFFSET carries the UTC offset with the value itself. For any application that runs across multiple time zones, or that stores audit records, using a timezone-aware type is not optional.

JSON storage has become a first-class concern in both databases. PostgreSQL offers JSON and JSONB — JSONB stores the data in a decomposed binary format that supports indexing and is almost always the right choice. SQL Server has no native JSON column type, but it provides JSON functions that operate on NVARCHAR columns storing JSON text. This is a meaningful architectural difference when your workload mixes structured and semi-structured data.

Choosing NUMERIC(precision, scale) or DECIMAL for financial figures is non-negotiable. Floating-point types (FLOAT, REAL, DOUBLE PRECISION) are approximations — they cannot exactly represent many decimal fractions, and you will eventually see rounding errors in financial calculations if you use them.

NULL: The Three-Valued Logic Problem

NULL is the most misunderstood concept in SQL, and it causes real production bugs. NULL does not mean zero, empty string, or unknown-but-similar. NULL means the absence of a value. And because of this, SQL operates under three-valued logic: a comparison can be TRUE, FALSE, or UNKNOWN.

When you compare any value to NULL using an equality operator, the result is UNKNOWN — not FALSE. This is why WHERE column = NULL never returns rows. The correct test is WHERE column IS NULL. This behavior catches developers off guard most often in WHERE clause filters and JOIN conditions.

Consider what happens with an outer join:

PostgreSQL:

-- Finding customers with no orders using LEFT JOIN and NULL check
SELECT
    c.customer_id,
    c.customer_name,
    o.order_id
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE o.order_id IS NULL;

SQL Server:

-- Finding customers with no orders using LEFT JOIN and NULL check
SELECT
    c.customer_id,
    c.customer_name,
    o.order_id
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE o.order_id IS NULL;

The query is identical here, but the underlying behavior differs when NULL propagation enters aggregate functions. Both databases correctly ignore NULLs in SUM(), AVG(), MIN(), and MAX(). But COUNT(column) counts only non-NULL values, while COUNT(*) counts all rows. This distinction has tripped up many a reporting query.

NULL also interacts with NOT IN in a particularly dangerous way. If a subquery used with NOT IN returns even one NULL value, the entire NOT IN expression evaluates to UNKNOWN for every row, and the outer query returns zero results. In production, this manifests as a query that silently returns no data and no error. The safest pattern when using subquery exclusions is NOT EXISTS.

Both PostgreSQL and SQL Server offer a COALESCE function that returns the first non-NULL value in a list. This is the standard, portable way to substitute a default for a NULL. SQL Server also has ISNULL(value, replacement), which is a two-argument shortcut, but COALESCE is preferred because it is part of the SQL standard and works identically in both databases.

Transactions, ACID, and What They Actually Mean in Practice

A transaction is a unit of work that either completes entirely or does not happen at all. The ACID properties — Atomicity, Consistency, Isolation, and Durability — define what guarantees a transaction must provide. These are not aspirational goals; they are contractual obligations that the database engine upholds at the cost of significant engineering complexity.

Atomicity means all operations in a transaction succeed, or all are rolled back. If you are transferring funds between two bank accounts — debiting one and crediting the other — atomicity ensures you can never debit without also crediting. A crash between the two operations leaves neither change in the database.

Consistency means the database moves from one valid state to another. Constraint checking — primary keys, foreign keys, check constraints — happens within the transaction boundary and ensures that a committed transaction never violates the schema's rules.

Isolation determines how concurrent transactions interact with each other. SQL defines four isolation levels: Read Uncommitted, Read Committed, Repeatable Read, and Serializable. Most production systems run at Read Committed, which prevents dirty reads but allows non-repeatable reads and phantom reads. PostgreSQL defaults to Read Committed and implements it using Multi-Version Concurrency Control (MVCC), meaning readers never block writers. SQL Server defaults to Read Committed as well, but implements it using locking by default — readers can block writers unless you enable Read Committed Snapshot Isolation (RCSI).

Durability means once a transaction is committed, it survives even a power failure. Both databases accomplish this through a write-ahead log (WAL in PostgreSQL, the transaction log in SQL Server). Every change is written to the log before it is applied to the data files, so the log can be replayed after a crash.

Explicit transaction control is straightforward in both systems:

PostgreSQL:

-- Explicit transaction with error handling in PostgreSQL
BEGIN;

UPDATE accounts SET balance = balance - 500.00 WHERE account_id = 101;
UPDATE accounts SET balance = balance + 500.00 WHERE account_id = 202;

-- If both updates succeeded, commit
COMMIT;

-- If something went wrong, this rolls back both updates
-- ROLLBACK;

SQL Server:

-- Explicit transaction with error handling in SQL Server (T-SQL)
BEGIN TRANSACTION;

BEGIN TRY
    UPDATE accounts SET balance = balance - 500.00 WHERE account_id = 101;
    UPDATE accounts SET balance = balance + 500.00 WHERE account_id = 202;
    COMMIT TRANSACTION;
END TRY
BEGIN CATCH
    ROLLBACK TRANSACTION;
    THROW;  -- re-raise the original error to the caller
END CATCH;

A pattern that trips up PostgreSQL developers coming from other databases: in PostgreSQL, if any statement inside a transaction raises an error, the entire transaction enters an aborted state and every subsequent statement — including COMMIT — will fail until you issue a ROLLBACK. SQL Server does not behave this way by default; a statement error does not automatically abort the entire transaction. This is why SET XACT_ABORT ON is a recommended practice in SQL Server stored procedures — it makes error behavior closer to what most developers expect.

Savepoints allow a transaction to establish intermediate rollback points without abandoning the entire transaction. This is useful in long-running batch operations where a single record failure should not roll back hours of work.

Schemas, Databases, and the Object Hierarchy

Both PostgreSQL and SQL Server organize objects into a hierarchy, but the hierarchy means different things in each system, and conflating them causes genuine confusion.

In SQL Server, the hierarchy is: Server → Database → Schema → Object. A single SQL Server instance can host many databases, and each database is fully isolated — cross-database queries require three-part naming (database.schema.table) and cross-database transactions have significant limitations. Within a database, schemas are namespaces for grouping objects. The default schema is dbo. Schemas in SQL Server are primarily an organizational and permission-management tool.

In PostgreSQL, the hierarchy is: Cluster → Database → Schema → Object. A PostgreSQL cluster is the running instance, and it hosts multiple databases. Like SQL Server, databases within a cluster are isolated — you cannot join tables across databases within a single query without extensions like dblink or postgres_fdw. Within a database, schemas serve the same organizational purpose. The default schema is public. PostgreSQL also has a search_path variable that determines which schemas are searched when an unqualified object name is used — misconfigured search paths are a source of subtle bugs in multi-schema applications.

Querying the system catalog tells you a great deal about how your database is structured:

PostgreSQL:

-- List all tables in the current database with their schemas and row estimates
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname || '.' || tablename)) AS total_size,
    n_live_tup AS estimated_rows
FROM pg_stat_user_tables
ORDER BY pg_total_relation_size(schemaname || '.' || tablename) DESC;

SQL Server:

-- List all tables in the current database with their schemas and row counts
SELECT
    s.name              AS schema_name,
    t.name              AS table_name,
    p.rows              AS estimated_rows,
    SUM(a.total_pages) * 8 / 1024.0 AS total_size_mb
FROM sys.tables t
JOIN sys.schemas s ON t.schema_id = s.schema_id
JOIN sys.indexes i ON t.object_id = i.object_id AND i.index_id IN (0, 1)
JOIN sys.partitions p ON i.object_id = p.object_id AND i.index_id = p.index_id
JOIN sys.allocation_units a ON p.partition_id = a.container_id
GROUP BY s.name, t.name, p.rows
ORDER BY total_size_mb DESC;

Understanding the object hierarchy matters particularly for permission management. In SQL Server, granting SELECT on a schema grants access to all current and future objects within it — a powerful pattern for role-based access control. In PostgreSQL, default privileges need to be set explicitly with ALTER DEFAULT PRIVILEGES to extend access to future objects. Many PostgreSQL environments have been burned by creating a new table only to discover that application users cannot access it because the default privilege grant was not set up correctly.

Users and roles are another area of divergence. PostgreSQL has a unified concept: everything is a role. A role with the LOGIN attribute is what most people call a user. A role without LOGIN is a group role. SQL Server separates logins (server-level identities) from users (database-level identities) — a login maps to a user within each database, and a user who exists in one database has no automatic presence in another.

---

Key Takeaways

  • → Relational databases express entity relationships through primary and foreign keys, and referential integrity enforced by foreign key constraints is one of the most valuable correctness guarantees the database provides — disabling it trades short-term convenience for long-term data quality problems.
  • → Data type selection is a design decision with lasting consequences: use `NUMERIC`/`DECIMAL` for financial data, timezone-aware types for any timestamp that crosses time zones, and understand the difference between PostgreSQL's native `JSONB` and SQL Server's JSON-over-`NVARCHAR` approach before choosing an architecture.
  • → NULL represents the absence of a value and participates in three-valued logic — comparisons involving NULL produce UNKNOWN, not FALSE, which causes silent failures in `WHERE` filters and especially in `NOT IN` subqueries.
  • → ACID transactions are enforced differently in the two databases: PostgreSQL uses MVCC so readers never block writers, while SQL Server uses locking by default and requires Read Committed Snapshot Isolation to achieve non-blocking reads; additionally, PostgreSQL's error-within-transaction behavior aborts the entire transaction, unlike SQL Server's default.
  • → The object hierarchy — server, database, schema, table — functions similarly in both databases conceptually, but the details of cross-database access, search paths, default privileges, and the login-versus-user distinction are meaningful differences that every DBA working across both platforms must internalize.
