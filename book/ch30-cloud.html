Chapter 30: Cloud Deployments

Running PostgreSQL or SQL Server in the cloud is no longer a specialized skill — it is the default expectation for most modern DBA roles. Whether your organization is migrating an on-premises warehouse to AWS, spinning up a new application on Azure, or evaluating managed database services across GCP, the decisions you make during the initial architecture phase will shape performance, cost, and operational overhead for years. This chapter walks through how cloud database deployments actually work: the managed service landscape, how cloud-native features differ from on-premises expectations, networking and security considerations, high availability in cloud environments, and the monitoring and cost discipline that separates a well-run cloud database from an expensive surprise at the end of the month.

---

The Managed Service Landscape: RDS, Azure SQL, Cloud SQL, and What You Give Up

The first decision in any cloud deployment is whether to run your database on a managed service or on a self-managed virtual machine. Both options are legitimate in production — the right choice depends on your team's operational capacity and your application's requirements.

Managed services (AWS RDS, Amazon Aurora, Azure SQL Database, Azure Database for PostgreSQL, Google Cloud SQL) handle the operating system, storage provisioning, patch management, and basic backup scheduling on your behalf. You interact with the database engine itself, but the infrastructure beneath it is abstracted away. For most teams, this is the right starting point. The tradeoff is real, though: you lose the ability to tune certain OS-level parameters, access server logs directly from the filesystem, or install extensions that the provider has not whitelisted.

Self-managed VMs (EC2, Azure VMs, GCE instances) give you full control. You install PostgreSQL or SQL Server yourself, configure storage, manage patches, and handle everything a traditional on-premises DBA would handle — except now the hardware is rented. This path makes sense when you need specific extensions (PostGIS, pg_partman at a version the provider hasn't adopted, or advanced SQL Server features like PolyBase), when you need OS-level tuning for extreme workloads, or when licensing economics favor Bring Your Own License (BYOL) arrangements.

AWS RDS for PostgreSQL and Amazon Aurora PostgreSQL are distinct offerings. RDS is a managed community PostgreSQL with a subset of extensions available. Aurora PostgreSQL is a cloud-native reimplementation of the PostgreSQL wire protocol on top of a distributed storage layer. Aurora's shared storage model means replicas share the same underlying storage rather than streaming WAL — which makes failover faster (typically under 30 seconds) but changes how you think about replication lag and storage costs.

Azure Database for PostgreSQL comes in two modes: Flexible Server (the current, recommended offering) and the legacy Single Server (which Microsoft is retiring). Flexible Server gives you a real choice of availability zones, stop/start capability for dev environments, and more parameter customization than earlier Azure Postgres offerings.

Azure SQL Database is Microsoft's managed SQL Server offering. It is not simply "SQL Server in the cloud" — it is a continuously updated, cloud-first version of the SQL Server engine that lacks certain on-premises features (SQL Server Agent in the traditional sense, cross-database queries without elastic query, linked servers) while adding cloud-native features like automatic index management, intelligent query processing tier adjustments, and built-in geo-redundancy options.

Google Cloud SQL supports both PostgreSQL and SQL Server. It is a solid managed offering but lags behind RDS and Azure in extension availability and advanced configuration surface area.

The practical guidance: start with a managed service unless you have a specific, documented reason not to. The operational savings are real. Revisit the self-managed path only when you hit a concrete wall with the managed service.

Sizing, Storage, and Instance Selection in Cloud Environments

Cloud database sizing is iterative in a way that on-premises sizing never was — you can resize an instance in minutes, and you pay for what you use. This flexibility is powerful, but it also leads to a common failure mode: teams either under-provision and wonder why their database is slow, or over-provision and never review the bill.

Instance sizing in cloud environments means choosing a compute tier (vCPUs and memory) and a storage type. These are separate decisions.

For PostgreSQL on RDS or Aurora, instance classes map to EC2 instance families: db.t4g instances are burstable (appropriate for dev/test, dangerous in production under sustained load), db.m7g and db.r7g are general-purpose and memory-optimized respectively. For SQL Server-heavy workloads on RDS, the db.r family (memory-optimized) is almost always the right choice because SQL Server's buffer pool benefits disproportionately from RAM.

Storage matters as much as compute. Most cloud providers decouple IOPS from storage capacity. On AWS RDS, you choose between:

  • → **gp3** (general-purpose SSD): You provision storage size and IOPS separately. IOPS up to 16,000 are included in the base price; beyond that, you pay per IOPS. This is the right default for most workloads.
  • → **io1/io2** (provisioned IOPS SSD): For workloads that need consistent, high IOPS above what gp3 provides. More expensive, but the performance floor is guaranteed.
  • → **Aurora storage**: Automatically scales up to 128 TiB. You pay per GB-month actually used, not provisioned capacity, which is a meaningful advantage for variable workloads.

    • A common mistake is provisioning an RDS instance with 100 GB of gp3 storage and 3,000 IOPS (the minimum), then being surprised when a batch workload causes storage latency. Size your IOPS based on your peak write throughput requirements, not your average.

    Memory and shared_buffers in the cloud. PostgreSQL's shared_buffers is typically set to 25% of instance RAM on managed services by default. On a db.r7g.2xlarge with 64 GB RAM, that's 16 GB. For read-heavy OLAP workloads, you can push this to 40% — but on managed RDS you do this through parameter groups, not postgresql.conf directly.

    SQL Server memory configuration on RDS is similarly managed through parameter groups. The max server memory equivalent maps to a percentage of instance memory. AWS handles this automatically for most RDS SQL Server instances, but you can override it for fine-tuned workloads.

    The following queries help you verify the current effective configuration on each platform:

    PostgreSQL:

    -- Check current memory-related configuration on RDS/managed PostgreSQL
SELECT name,
       setting,
       unit,
       source,
       context
FROM   pg_settings
WHERE  name IN ('shared_buffers',
                'effective_cache_size',
                'work_mem',
                'max_connections',
                'wal_buffers')
ORDER BY name;

    SQL Server:

    -- Check current server memory configuration on RDS/Azure SQL
SELECT name,
       value_in_use,
       description
FROM   sys.configurations
WHERE  name IN ('max server memory (MB)',
                'min server memory (MB)',
                'max degree of parallelism',
                'cost threshold for parallelism')
ORDER BY name;

    Right-sizing after deployment. Cloud databases generate enough observability data that you can make informed resize decisions after a week of production traffic. On RDS, CloudWatch metrics CPUUtilization, FreeableMemory, ReadIOPS, and WriteIOPS tell most of the story. A database consistently sitting at 20% CPU and 60% freeable memory is a candidate for a smaller instance class. A database at 90% CPU during business hours with queue depth spikes is under-provisioned on compute.

    Networking, Security, and Access Control

    Cloud database networking is where many organizations introduce security gaps without realizing it. The defaults on most managed services are not production-safe out of the box — they require deliberate hardening.

    VPC isolation is the foundation. Every production database should live inside a private subnet with no direct internet route. On AWS, this means your RDS instance is in a VPC subnet with no internet gateway route, and access is mediated through your application subnet (or a bastion host/VPN for DBA access). On Azure, Azure Database for PostgreSQL Flexible Server supports VNet integration, which puts the managed service inside your own VNet rather than on a shared network with a service endpoint. This is the preferred model.

    Security groups and firewall rules should follow the principle of least privilege. The database security group should allow inbound traffic on the database port (5432 for PostgreSQL, 1433 for SQL Server) only from the application tier's security group ID — not from a CIDR block, and certainly not from 0.0.0.0/0. Audit this periodically. Cloud environments tend to accumulate overly permissive rules during emergency access windows that never get cleaned up.

    SSL/TLS enforcement is non-negotiable in cloud deployments. On RDS PostgreSQL, enforce this by setting rds.force_ssl = 1 in the parameter group. On Azure Database for PostgreSQL Flexible Server, SSL is enforced by default and can be configured for TLS version minimums.

    PostgreSQL:

    -- Verify SSL is being used for current connections
SELECT pid,
       usename,
       application_name,
       client_addr,
       ssl,
       cipher,
       bits
FROM   pg_stat_ssl
JOIN   pg_stat_activity USING (pid)
WHERE  ssl = true
ORDER BY pid;

    SQL Server:

    -- Check encrypted connections on SQL Server (RDS or Azure)
SELECT session_id,
       login_name,
       host_name,
       program_name,
       encrypt_option,
       auth_scheme
FROM   sys.dm_exec_connections
WHERE  session_id > 50   -- exclude system sessions
ORDER BY session_id;

    IAM authentication is a cloud-native security enhancement that most teams underuse. AWS RDS supports IAM database authentication for PostgreSQL and MySQL — instead of storing a password in your application's configuration, the application uses its IAM role to generate a short-lived authentication token. This eliminates the problem of long-lived credentials stored in environment variables or secrets that are rarely rotated. Azure SQL Database supports Azure Active Directory authentication in the same vein.

    Secrets management for database credentials should use AWS Secrets Manager, Azure Key Vault, or GCP Secret Manager — not environment variables in a Lambda function or application server configuration file. Most managed database services integrate natively with these secret stores and support automatic credential rotation. Set up rotation from day one; retrofitting it later is painful.

    Network latency between your application and database deserves attention in multi-region or multi-AZ architectures. Cross-AZ traffic within the same region is typically sub-millisecond but is not free on AWS (data transfer costs apply). Cross-region replication adds meaningful latency — a replica in us-west-2 serving reads for an application in us-east-1 will see 60–80ms round-trip times, which can be acceptable for read replicas but is catastrophic if your application mistakenly routes writes there.

    High Availability, Failover, and Read Replicas in the Cloud

    High availability in the cloud looks superficially similar to on-premises HA but works differently in important ways. Understanding the mechanics of how your cloud provider handles failover prevents unpleasant surprises during an actual incident.

    Multi-AZ on RDS maintains a synchronous standby replica in a different availability zone. The standby is not addressable for reads — it exists purely for failover. When a failover occurs, RDS updates the DNS endpoint to point to the standby, which is promoted to primary. The DNS TTL is 5 seconds, so applications with short connection timeouts and proper reconnect logic recover within 30–60 seconds of a failover event. Applications that cache DNS or that open a single long-lived connection without reconnect handling will stay down until someone restarts them. This is the most common cause of extended outages during planned RDS failovers.

    Aurora's Multi-AZ is architecturally different. The shared storage layer is inherently replicated across three AZs — there's no traditional "standby with WAL replay." Failover promotes one of the read replicas to become the new writer, which is faster (typically 20–30 seconds) and does not require storage synchronization. Aurora also supports up to 15 read replicas sharing the same storage cluster.

    Azure Database for PostgreSQL Flexible Server with zone-redundant HA maintains a hot standby in a different availability zone with synchronous replication. Failover is automatic and typically completes in under 60 seconds. The standby is not readable — a design decision consistent with RDS Multi-AZ.

    Read replicas exist for scaling read workloads, not for HA. This distinction matters operationally. A read replica on RDS is asynchronous — replication lag is a function of write throughput and network conditions. During heavy write periods, replicas can fall seconds or even minutes behind. If your application reads from replicas and writes to the primary, you must handle eventual consistency: a user who just created a record may not see it on a subsequent read if that read hits a lagged replica.

    PostgreSQL:

    -- Check replication lag on a PostgreSQL primary (RDS or self-managed)
SELECT client_addr,
       state,
       sent_lsn,
       write_lsn,
       flush_lsn,
       replay_lsn,
       (sent_lsn - replay_lsn) AS replication_lag_bytes,
       write_lag,
       flush_lag,
       replay_lag
FROM   pg_stat_replication
ORDER BY replay_lag DESC NULLS LAST;

    SQL Server:

    -- Check replication / availability group lag on SQL Server
SELECT ag.name                     AS ag_name,
       ar.replica_server_name,
       drs.database_name,
       drs.synchronization_state_desc,
       drs.synchronization_health_desc,
       drs.redo_queue_size,
       drs.log_send_queue_size,
       drs.last_commit_time
FROM   sys.dm_hadr_database_replica_states drs
JOIN   sys.availability_replicas ar
    ON drs.replica_id = ar.replica_id
JOIN   sys.availability_groups ag
    ON ar.group_id = ag.group_id
ORDER BY drs.log_send_queue_size DESC;

    Connection pooling in cloud environments deserves special attention. Cloud databases frequently have lower max_connections limits than on-premises installations because the managed service shares infrastructure resources. A db.r7g.large RDS PostgreSQL instance has a default max_connections around 200. A microservices architecture with 20 services each opening 20 connection pool connections will exhaust this immediately. PgBouncer (for PostgreSQL) or a connection proxy layer is essential. AWS offers RDS Proxy, which provides connection pooling as a managed service and maintains connections to the database regardless of application-side connection churn — particularly valuable in Lambda-heavy architectures where function instances open and close connections on every invocation.

    Planned failover testing should be part of your regular operations. Both RDS and Azure provide a "Reboot with failover" option that simulates a primary failure and promotes the standby. Run this quarterly on non-production environments, and at least annually on production during a maintenance window. Know your actual recovery time before you need it.

    Backup, Point-in-Time Recovery, and Cross-Region Strategies

    Cloud databases provide automated backups by default, but "automated" does not mean "tested" or "sufficient." Understanding what the backup system actually captures — and what it does not — is the difference between a backup strategy and a false sense of security.

    Automated backups on RDS consist of daily snapshots plus continuous WAL archiving. Together, these enable point-in-time recovery (PITR) to any second within your retention window (1–35 days). The snapshot alone is insufficient for PITR — you need both the snapshot and the transaction logs. RDS manages this transparently; when you initiate a point-in-time restore, AWS selects the appropriate base snapshot and replays WAL to your target time. The restore creates a new RDS instance — it does not restore in place. Factor the time to launch a new instance (5–15 minutes), restore from snapshot, and replay WAL when calculating your actual RTO.

    Azure Database for PostgreSQL Flexible Server automated backups follow a similar model: full backups weekly, differential backups daily, WAL backups every 5 minutes. PITR is available to the nearest 5-minute boundary within your retention period (7–35 days).

    Manual snapshots are your insurance against the automated backup retention window. If you need to recover data from 45 days ago and your retention is 35 days, you cannot do it without a manual snapshot. Create manual snapshots before major deployments, schema migrations, and bulk data operations. They persist until you explicitly delete them and are billed at standard storage rates.

    Cross-region backup copies protect against regional outages. Both RDS and Azure support copying automated backups or manual snapshots to another region. This is a separate configuration from read replicas — backup copies are for disaster recovery from a cold start, while read replicas are for live traffic distribution.

    For SQL Server on RDS, native backup files (.bak) can be generated and exported to S3, which is valuable when you need to move data between RDS and self-managed SQL Server instances or between AWS regions with custom retention policies.

    The following queries help validate that your backup and PITR mechanisms are functioning as expected:

    PostgreSQL:

    -- Check WAL archiving status on a self-managed PostgreSQL
-- (On RDS, use CloudWatch metrics for archive_lag instead)
SELECT archived_count,
       last_archived_wal,
       last_archived_time,
       failed_count,
       last_failed_wal,
       last_failed_time,
       stats_reset
FROM   pg_stat_archiver;

    SQL Server:

    -- Check recent backup history for all databases
SELECT bs.database_name,
       bs.type                                  AS backup_type,
       bs.backup_start_date,
       bs.backup_finish_date,
       DATEDIFF(MINUTE,
                bs.backup_start_date,
                bs.backup_finish_date)          AS duration_minutes,
       CAST(bs.backup_size / 1048576.0 AS DECIMAL(18,2)) AS backup_size_mb,
       bmf.physical_device_name
FROM   msdb.dbo.backupset bs
JOIN   msdb.dbo.backupmediafamily bmf
    ON bs.media_set_id = bmf.media_set_id
WHERE  bs.backup_start_date >= DATEADD(DAY, -7, GETDATE())
ORDER BY bs.backup_start_date DESC;

    Recovery testing is the most neglected part of backup strategy in cloud environments. The fact that RDS shows "backup enabled" in the console tells you nothing about whether you can actually restore a usable database within your RTO. Establish a quarterly practice of restoring a production snapshot to a test instance, validating data integrity with application-level queries, and timing the process end-to-end. Document the results. This exercise regularly surfaces problems — missing permissions for the restore role, outdated application connection strings pointing to the old instance, missing secrets in the restored environment — that would otherwise only be discovered during an actual disaster.

    Cost Management, Performance Monitoring, and Cloud-Specific Tuning

    Cloud database costs have a way of growing quietly. A deployment that costs $800 per month in year one can reach $4,000 per month by year three through a combination of storage growth, unreviewed read replicas, snapshot accumulation, and data transfer charges — without the database actually doing more useful work. Cost discipline is a legitimate DBA responsibility in cloud environments.

    Understand your cost components. For RDS, the main cost drivers are: instance compute (hourly, varies by instance class and Multi-AZ status), storage (per GB-month plus IOPS for io1/io2), data transfer (cross-AZ and egress to the internet), automated backup storage beyond the free tier (equal to your provisioned database storage), and manual snapshot storage. For Azure SQL Database, the model differs — you pay for a DTU or vCore tier, and storage is billed separately above a certain threshold depending on the service tier.

    Storage growth is often the most insidious cost driver. PostgreSQL tables that are frequently updated accumulate dead tuples. On managed services, autovacuum handles reclamation, but it does not return freed space to the OS — it reclaims it for reuse within the same table. If a table bloats significantly and then has data deleted, the file on disk does not shrink. To reclaim actual storage on RDS PostgreSQL, you need VACUUM FULL (which takes an access exclusive lock) or pg_repack (which many managed services now support via extensions). Monitor storage growth trends and investigate tables with high bloat ratios before they grow into a storage tier increase.

    Right-sizing is an ongoing process, not a one-time decision. After 90 days of production traffic, review your CloudWatch or Azure Monitor metrics and ask: Is CPUUtilization consistently above 80%? Is FreeableMemory consistently below 20%? Is the storage throughput hitting provisioned IOPS limits regularly? Each of these points to a different resource bottleneck requiring a different response. CPU saturation calls for a larger instance class or query optimization. Memory pressure calls for a memory-optimized instance or reducing max_connections. IOPS saturation calls for gp3 IOPS increase or storage type migration.

    Query performance in cloud environments has the same root causes as on-premises — missing indexes, inefficient query plans, lock contention — but the observability tools are different. AWS Performance Insights provides a visual breakdown of wait events and active sessions over time, mapped directly to the queries causing them. This is the first tool to reach for when investigating a cloud PostgreSQL or SQL Server slowdown. Azure SQL Database has Query Performance Insight built into the portal, backed by the Query Store.

    PostgreSQL:

    -- Identify top queries by total execution time using pg_stat_statements
-- (Available on RDS/Aurora/Azure when the extension is enabled)
SELECT LEFT(query, 100)                         AS query_snippet,
       calls,
       ROUND((total_exec_time / 1000)::NUMERIC, 2)  AS total_seconds,
       ROUND((mean_exec_time)::NUMERIC, 2)       AS avg_ms,
       ROUND((stddev_exec_time)::NUMERIC, 2)     AS stddev_ms,
       rows
FROM   pg_stat_statements
WHERE  query NOT ILIKE '%pg_stat_statements%'
ORDER BY total_exec_time DESC
LIMIT  20;

    SQL Server:

    -- Identify top queries by CPU consumption on Azure SQL / RDS SQL Server
SELECT TOP 20
    SUBSTRING(qt.text,
              (qs.statement_start_offset / 2) + 1,
              ((CASE qs.statement_end_offset
                    WHEN -1 THEN DATALENGTH(qt.text)
                    ELSE qs.statement_end_offset
               END - qs.statement_start_offset) / 2) + 1) AS query_text,
    qs.execution_count,
    qs.total_worker_time / 1000                AS total_cpu_ms,
    qs.total_worker_time / qs.execution_count / 1000 AS avg_cpu_ms,
    qs.total_elapsed_time / 1000               AS total_elapsed_ms,
    qs.total_logical_reads,
    qs.creation_time                           AS plan_cached_time
FROM   sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) qt
ORDER BY qs.total_worker_time DESC;

    Reserved instances and savings plans can cut compute costs by 30–50% compared to on-demand pricing. If you have a production database that will run continuously for the next 12–36 months — and most do — the economics of reserved capacity are compelling. Review your stable, long-running instances annually and convert them to reserved or committed-use pricing. This is a financial decision, not a technical one, but it belongs in the DBA's operational review.

    Multi-AZ cost awareness. Multi-AZ on RDS doubles the compute cost because you are running two instances (primary and standby). This is the right investment for production databases. However, development, test, and staging environments almost never need Multi-AZ. Audit your non-production RDS instances and verify that Multi-AZ is disabled unless there is a specific reason for it. The savings across a large estate of dev and test databases can be substantial.

    Parameter tuning for cloud instances follows the same principles as on-premises tuning but has some cloud-specific considerations. On RDS PostgreSQL, checkpoint_completion_target should be set to 0.9 to spread checkpoint I/O across a longer window, reducing write spikes. random_page_cost should be reduced to 1.1 for databases backed by SSD storage (which all cloud databases are) — the default of `4.
