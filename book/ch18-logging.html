Chapter 18: Logging and Auditing

Every production database tells a story through its logs. When a query suddenly degrades at 2 AM, when a compliance auditor asks who changed a table's data last Tuesday, or when a security team needs to know which accounts accessed sensitive records — the answer lives in your logging and auditing infrastructure, assuming you built it correctly. This chapter covers how PostgreSQL and SQL Server each approach logging and auditing, from the raw mechanics of writing log entries to building robust audit trails that satisfy regulatory requirements. By the end, you will know how to configure both systems to capture exactly what you need, interpret what you find, and avoid the common pitfalls that leave DBAs without the evidence they need when it matters most.

---

What Gets Logged and Why It Matters

Logging and auditing are related but distinct disciplines. Logging is the continuous recording of database activity — errors, connections, slow queries, lock waits, startup events, and more. It is primarily an operational tool. Auditing is the deliberate, structured capture of who did what to which data, and when. It is primarily a compliance and security tool. Most production environments need both, and the two often share infrastructure.

Both PostgreSQL and SQL Server generate logs at multiple layers. There is the database engine's internal log, which captures operational events. There is the transaction log (WAL in PostgreSQL, the LDF file in SQL Server), which is a durability mechanism and not a readable audit trail. And there are purpose-built audit mechanisms that can be configured to write structured records of specific activities. Understanding which layer handles which concern keeps you from building redundant systems or, worse, assuming something is being captured when it is not.

The practical reasons DBAs care about logging are consistent across environments. You need logs to diagnose performance problems — slow query logs show which statements are causing pain. You need logs to investigate security incidents — who connected, from where, and when. You need logs for compliance — HIPAA, PCI-DSS, SOX, and GDPR all have specific requirements around data access records. And you need logs for capacity planning — connection counts, error rates, and query volume over time reveal trends before they become crises.

A common mistake is treating logging as a set-and-forget configuration. Log files fill up, rotate poorly, or capture so much noise that the signal is buried. Getting logging right means understanding not just what to turn on, but what the tradeoffs are between verbosity, storage cost, and query overhead.

---

Configuring the PostgreSQL Logging System

PostgreSQL's logging is controlled through postgresql.conf, and the depth of what it can capture is impressive once you understand the options. The two most operationally important areas are where logs go and what gets written.

Destination and format are set via log_destination. PostgreSQL can write logs to stderr, csvlog, jsonlog (added in PostgreSQL 15), or syslog. For most environments, csvlog or jsonlog paired with the logging_collector process is the right choice because it gives you structured, parseable output. The logging_collector is a background process that intercepts stderr output and writes it to files, handling rotation cleanly.

# postgresql.conf
logging_collector = on
log_destination = 'csvlog'
log_directory = 'log'
log_filename = 'postgresql-%Y-%m-%d_%H%M%S.log'
log_rotation_age = 1d
log_rotation_size = 100MB

What gets logged is where most of the decisions live. The most important settings for a production DBA are:

  • → `log_min_duration_statement`: Logs any statement taking longer than the specified number of milliseconds. Set this to something like `1000` (one second) to start, and adjust based on your workload. Setting it to `0` logs everything, which is useful for debugging but ruinous in production due to I/O overhead.
  • → `log_connections` and `log_disconnections`: Log when sessions start and end. Modest overhead, high value for security audits.
  • → `log_lock_waits`: Log when a session waits longer than `deadlock_timeout` for a lock. Essential for diagnosing contention.
  • → `log_error_verbosity`: Controls how much detail appears in error messages — `terse`, `default`, or `verbose`.
  • → `log_line_prefix`: A format string that prefixes each log line with contextual data. A production-quality prefix captures timestamp, PID, user, database, and client address.
    • # A production-grade log_line_prefix
log_line_prefix = '%t [%p]: [%l-1] user=%u,db=%d,app=%a,client=%h '

    This prefix means every log line includes the timestamp, process ID, line number within that session, username, database name, application name, and client host. When you are searching logs for a specific user's activity during an incident window, this context is what makes the search tractable.

    Statement-level logging can be tuned further. log_statement accepts values of none, ddl, mod, or all. Setting it to ddl logs all DDL statements (CREATE, ALTER, DROP) without logging every SELECT, which is a reasonable middle ground for many environments. Setting it to mod adds DML (INSERT, UPDATE, DELETE). all should only be used temporarily during troubleshooting.

    To query what PostgreSQL has logged recently when using csvlog, you can load the CSV log into a temporary table or use the pg_read_file() function for quick inspection. PostgreSQL 15 and later expose recent logs through the pg_log_backend_memory_contexts() function and, for structured logs, through external tooling. For most teams, shipping logs to a centralized system (ELK stack, Splunk, CloudWatch) is the right long-term approach.

    Configuring SQL Server Logging and the Error Log

    SQL Server's equivalent of PostgreSQL's operational log is the SQL Server Error Log. It is written to a flat text file by default and captures server startup events, backup completions, login failures, database corruption alerts, and various informational messages. Unlike PostgreSQL's configurable log verbosity, the Error Log is less flexible — you cannot easily tell it to log slow queries there. That responsibility falls to other mechanisms.

    The Error Log location is set during installation but can be changed. SQL Server keeps a configurable number of archived logs (default is 6 archives plus the current one), which can be dangerously small on busy systems. Increase this in production:

    -- View current Error Log configuration
EXEC sp_configure 'show advanced options', 1;
RECONFIGURE;
EXEC sp_configure 'number of error log files', 30;
RECONFIGURE;

    Reading the current Error Log programmatically:

    PostgreSQL:

    -- Read the current PostgreSQL log file (last 100 lines via pg_read_file)
SELECT *
FROM pg_read_file(
    pg_current_logfile(),
    0,
    pg_stat_file(pg_current_logfile()).size
);

    SQL Server:

    -- Read the current SQL Server Error Log
EXEC sp_readerrorlog 0, 1;

-- Read with filtering for a specific keyword
EXEC sp_readerrorlog 0, 1, 'Login failed';

    For slow query capture in SQL Server, the primary tool is the Query Store (covered in depth in Chapter 12), combined with Extended Events. Extended Events is SQL Server's lightweight, high-performance event tracing framework, and it replaced the older SQL Trace and SQL Server Profiler mechanisms. Never use SQL Server Profiler on a production server — it imposes server-side tracing overhead that can cripple a busy system. Extended Events runs with far less impact.

    A minimal Extended Events session to capture slow queries looks like this:

    SQL Server:

    -- Create an Extended Events session to capture queries over 2 seconds
CREATE EVENT SESSION [SlowQueryCapture] ON SERVER
ADD EVENT sqlserver.sql_statement_completed (
    WHERE duration > 2000000  -- duration is in microseconds
    ACTION (
        sqlserver.sql_text,
        sqlserver.database_name,
        sqlserver.username,
        sqlserver.client_hostname,
        sqlserver.plan_handle
    )
)
ADD TARGET package0.ring_buffer (SET max_memory = 51200)
WITH (
    MAX_DISPATCH_LATENCY = 5 SECONDS
);

ALTER EVENT SESSION [SlowQueryCapture] ON SERVER STATE = START;

    The ring buffer target holds events in memory. For persistent capture, switch to package0.event_file with a file path. Be mindful of disk space — a high-traffic server can write event files quickly if the filter threshold is too low.

    Building Audit Trails: pgAudit and SQL Server Audit

    Operational logs capture what the database is doing. Audit trails capture what users are doing to data. These are different goals, and both databases have dedicated mechanisms for it.

    PostgreSQL: pgAudit

    PostgreSQL does not include a built-in SQL-level audit facility in its core — the log_statement setting gets you some of the way there, but it lacks the structure that compliance teams need. The pgaudit extension fills this gap. It hooks into PostgreSQL's standard logging infrastructure and writes detailed audit records for the statement types you configure.

    pgAudit has two levels of configuration: session audit (captures all activity from specific sessions) and object audit (captures activity on specific tables, views, or sequences). In practice, object-level auditing is what most compliance requirements need — you want to know who touched the payments table, not every query in the database.

    -- After installing the extension and adding it to shared_preload_libraries:
CREATE EXTENSION IF NOT EXISTS pgaudit;

-- Set session-level audit logging (in postgresql.conf or via ALTER SYSTEM)
-- pgaudit.log = 'write, ddl'
-- This captures INSERT, UPDATE, DELETE, TRUNCATE and all DDL

-- For object-level auditing, create an audit role and grant it SELECT on sensitive tables
CREATE ROLE audit_role;
GRANT SELECT ON TABLE customer_payments TO audit_role;

-- Then set the object audit log to use that role
-- pgaudit.role = 'audit_role'

    With pgAudit enabled and pgaudit.log = 'write, ddl', each captured statement generates a log entry structured like:

    AUDIT: SESSION,1,1,WRITE,INSERT,TABLE,public.customer_payments,
INSERT INTO customer_payments VALUES (...)

    The fields — audit type, statement ID, substatement ID, class, command, object type, object name, and statement text — give compliance tools something consistent to parse.

    SQL Server: SQL Server Audit

    SQL Server has a mature, built-in audit framework that was introduced in SQL Server 2008. It works through a three-layer model: Server Audit (defines where audit records go), Server Audit Specification (captures server-level events like logins and server role membership changes), and Database Audit Specification (captures database-level events like SELECT, INSERT, UPDATE, DELETE on specific objects).

    SQL Server:

    -- Step 1: Create the Server Audit (writes to file)
CREATE SERVER AUDIT [ComplianceAudit]
TO FILE (
    FILEPATH = 'C:\SQLAudit\',
    MAXSIZE = 100 MB,
    MAX_ROLLOVER_FILES = 10,
    RESERVE_DISK_SPACE = OFF
)
WITH (
    QUEUE_DELAY = 1000,
    ON_FAILURE = CONTINUE
);

ALTER SERVER AUDIT [ComplianceAudit] WITH (STATE = ON);

-- Step 2: Create a Database Audit Specification
USE SensitiveDB;
GO

CREATE DATABASE AUDIT SPECIFICATION [CustomerDataAudit]
FOR SERVER AUDIT [ComplianceAudit]
ADD (SELECT, INSERT, UPDATE, DELETE ON dbo.customer_payments BY public),
ADD (EXECUTE ON dbo.process_payment BY public)
WITH (STATE = ON);

    The ON_FAILURE setting controls what happens if the audit target is unavailable. CONTINUE lets the server keep running; SHUTDOWN stops the server to prevent unaudited operations from occurring. Regulated environments often require SHUTDOWN to guarantee audit completeness, but this is a serious decision — a full disk or unavailable network share will bring down your SQL Server instance.

    Querying SQL Server audit files after the fact:

    SQL Server:

    -- Read audit records from the audit files
SELECT
    event_time,
    action_id,
    succeeded,
    server_principal_name,
    database_name,
    object_name,
    statement,
    client_ip
FROM sys.fn_get_audit_file(
    'C:\SQLAudit\ComplianceAudit*.sqlaudit',
    DEFAULT,
    DEFAULT
)
WHERE object_name = 'customer_payments'
ORDER BY event_time DESC;

    PostgreSQL:

    -- Query pgAudit entries from the log (loaded into a staging table or parsed externally)
-- If using csvlog, load it into a temporary table first:
COPY tmp_pg_log FROM '/var/log/postgresql/postgresql-2024-01-15_000000.csv'
WITH (FORMAT csv);

SELECT
    log_time,
    user_name,
    database_name,
    message
FROM tmp_pg_log
WHERE message LIKE 'AUDIT%'
  AND message LIKE '%customer_payments%'
ORDER BY log_time DESC;

    Capturing Login Activity and Privilege Escalations

    One of the most important audit concerns in any regulated environment is tracking authentication events and privilege changes. A compromised credential or an unauthorized privilege grant can be the beginning of a serious breach, and the audit trail is often the only way to establish timeline and scope after the fact.

    PostgreSQL Login Auditing

    PostgreSQL's log_connections and log_disconnections settings are the first line here. When enabled, every connection attempt — successful or failed — appears in the server log with the username, database, client address, and authentication method. Failed authentication attempts log at the LOG severity level, which means they appear in the log regardless of log_min_messages settings.

    For privilege changes, log_statement = 'ddl' captures GRANT and REVOKE statements because they are DDL-class operations in PostgreSQL's categorization. pgAudit with pgaudit.log = 'role' specifically captures role membership changes, which is the PostgreSQL equivalent of group membership auditing.

    The pg_stat_activity view gives you a real-time picture of current sessions, which complements the historical log:

    PostgreSQL:

    -- Find all active connections, who they are, and what they're running
SELECT
    pid,
    usename,
    application_name,
    client_addr,
    backend_start,
    state,
    wait_event_type,
    wait_event,
    LEFT(query, 100) AS query_snippet
FROM pg_stat_activity
WHERE state != 'idle'
  AND pid != pg_backend_pid()
ORDER BY backend_start;

    SQL Server Login Auditing

    SQL Server's Server Audit Specification can capture login events explicitly:

    SQL Server:

    -- Add login success and failure tracking to the server audit specification
CREATE SERVER AUDIT SPECIFICATION [LoginAudit]
FOR SERVER AUDIT [ComplianceAudit]
ADD (SUCCESSFUL_LOGIN_GROUP),
ADD (FAILED_LOGIN_GROUP),
ADD (LOGOUT_GROUP),
ADD (SERVER_ROLE_MEMBER_CHANGE_GROUP),
ADD (DATABASE_ROLE_MEMBER_CHANGE_GROUP)
WITH (STATE = ON);

    The *_GROUP notation refers to SQL Server's audit action groups — predefined collections of related events. SERVER_ROLE_MEMBER_CHANGE_GROUP fires whenever someone is added to or removed from a fixed server role (like sysadmin). DATABASE_ROLE_MEMBER_CHANGE_GROUP does the same for database roles. These two groups are non-negotiable for any security-focused audit configuration.

    For real-time session visibility, SQL Server's equivalent of pg_stat_activity is sys.dm_exec_sessions joined to sys.dm_exec_requests:

    SQL Server:

    -- Current active sessions with query text and wait information
SELECT
    s.session_id,
    s.login_name,
    s.host_name,
    s.program_name,
    s.login_time,
    r.status,
    r.wait_type,
    r.wait_time,
    DB_NAME(r.database_id) AS database_name,
    SUBSTRING(t.text, 1, 100) AS query_snippet
FROM sys.dm_exec_sessions s
LEFT JOIN sys.dm_exec_requests r
    ON s.session_id = r.session_id
OUTER APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE s.is_user_process = 1
ORDER BY s.login_time;

    Log Management, Retention, and Performance Tradeoffs

    Logging infrastructure that is not maintained eventually becomes a liability. Logs that fill up disk cause PostgreSQL to crash or SQL Server to stop auditing (depending on your ON_FAILURE setting). Logs retained longer than needed create compliance exposure of a different kind — old records containing sensitive data you no longer need. And verbose logging on a high-throughput system can measurably degrade query performance.

    Rotation and retention policies should be written down and automated. For PostgreSQL, log_rotation_age and log_rotation_size handle file rotation. Deletion of old log files needs an external process — a cron job or a log management tool. A common pattern is retaining 90 days of operational logs locally and shipping to cold storage (S3, Azure Blob) for long-term compliance retention.

    For SQL Server audit files, the MAX_ROLLOVER_FILES parameter controls how many archived files are kept before the oldest is overwritten. This alone is not a retention policy — it is a circular buffer. You need an external process to archive and purge files on a schedule that matches your compliance requirements.

    Performance impact of logging deserves direct attention. The costs come from three places: the I/O of writing log records, any server-side processing to filter or format events, and the overhead of the audit hook intercepting query execution.

    For PostgreSQL, log_min_duration_statement has essentially no overhead when queries complete faster than the threshold, because the decision is made post-execution. Logging itself is asynchronous through the logging collector process. The real performance risk is log_statement = 'all' or very aggressive pgAudit settings on a write-heavy OLTP system — benchmark before enabling in production.

    For SQL Server, the Extended Events ring buffer target is lightweight, but the event_file target writing to a slow disk can introduce latency. The SQL Server Audit framework was designed for this use case and is generally efficient, but auditing high-frequency tables (millions of rows per hour) with row-level SELECT auditing will produce enormous audit files and measurable overhead. Profile your write rates before committing to fine-grained object-level auditing.

    A practical approach is to tier your logging: capture everything at the connection and DDL level always, capture slow queries with a reasonable threshold always, and enable fine-grained data-access auditing only for explicitly identified sensitive objects. This gives you comprehensive coverage of security-relevant events without logging every SELECT on a lookup table that changes once a year.

    Log centralization is the other operational imperative. Logs sitting on the database server are only as available as the server itself. If you are investigating a security incident on a compromised host, you cannot trust logs stored there. Shipping logs in real-time to an independent, append-only destination (a SIEM, a managed log service, or even a write-once S3 bucket) gives you a trustworthy audit record. For regulated environments, this is often not optional.

    ---

    Key Takeaways

  • → **Logging and auditing serve different masters.** Operational logging supports incident diagnosis and performance monitoring; audit logging supports compliance, security, and forensic investigations. Configure them separately with their respective goals in mind rather than trying to satisfy both with a single verbose log stream.
  • → **PostgreSQL's logging is highly configurable through `postgresql.conf`.** `log_min_duration_statement`, `log_connections`, `log_lock_waits`, and `log_statement = 'ddl'` form a solid production baseline. The `pgaudit` extension adds structured, compliance-grade audit output when statement-level logging is insufficient.
  • → **SQL Server provides first-class audit infrastructure through Server Audit.** The three-layer model (Server Audit → Server Audit Specification → Database Audit Specification) gives you precise control over what is captured and where it is written. Always configure the `ON_FAILURE` behavior deliberately based on whether your environment prioritizes availability or audit completeness.
  • → **Authentication and privilege events are the highest-value audit targets.** Failed logins, privilege escalations, and role membership changes are the signals that matter most for security investigations. Both databases can capture these with modest configuration overhead, and most compliance frameworks require it explicitly.
  • → **Log management is not optional infrastructure.** Rotation, retention schedules, centralized shipping, and performance impact analysis are all production responsibilities. Verbose logging without a management strategy degrades database performance, fills disks, and leaves you with an untrustworthy record when you need it most. Design your logging architecture with the same rigor you apply to your backup strategy.
