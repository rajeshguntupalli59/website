Chapter 15: Automation

Automation is what separates a reactive DBA from a proactive one. When you are managing one database, doing everything by hand is manageable. When you are managing dozens — or when those databases serve millions of users across time zones — manual processes become a liability. This chapter covers how to automate the everyday tasks that consume DBA time: maintenance jobs, health checks, alerting, deployments, and routine administrative work. Both PostgreSQL and SQL Server offer native scheduling and scripting capabilities, and this chapter walks through both ecosystems in depth, from writing your first scheduled job to building fully automated operational pipelines that require minimal human intervention.

---

Why Automation Belongs at the Core of DBA Work

The instinct to do things manually is understandable. When something breaks, you want to touch it directly, see what is happening, and fix it with your own hands. That instinct is correct in the moment of an incident. But that same instinct, applied to routine work, creates fragility. Maintenance that depends on someone remembering to run it will eventually not get run. A VACUUM that should have happened at 2 AM did not happen because the DBA was on vacation and no one else knew about the script sitting in a home directory.

Automation is not about removing the DBA from the picture. It is about making the DBA's decisions persistent. When you decide that index maintenance should happen every Sunday at midnight, automation ensures that decision executes reliably every week, whether you are in the office or not. When you decide that any table bloat exceeding 20% should trigger an alert, automation ensures that condition is monitored continuously without anyone having to run a query.

There is also a compounding benefit. Every task you automate is a task you are no longer paying attention to during on-call hours. That freed attention can go toward architecture decisions, performance tuning, and the kinds of problems that actually require human judgment. Automation raises the floor of your operational quality and frees you to work on the ceiling.

Both PostgreSQL and SQL Server have matured significantly in this area. PostgreSQL relies on cron-based scheduling at the OS level, the pg_cron extension for database-native scheduling, and scripting in Bash or Python. SQL Server has SQL Server Agent, which provides a full job scheduling framework inside the engine. Each approach has trade-offs, and experienced DBAs often combine native tools with external orchestration depending on the scale of the environment.

Scheduling Maintenance Jobs

Every production database accumulates dead rows, stale statistics, and fragmented indexes. Left unmanaged, these problems degrade performance. Automation ensures that maintenance runs on schedule without manual intervention.

PostgreSQL: pg_cron and OS-Level Cron

PostgreSQL does not ship with a built-in job scheduler in the way SQL Server does. The two most common approaches are OS-level cron and the pg_cron extension.

OS cron is the simpler path. A typical maintenance script runs psql to connect to the database and execute maintenance commands. This works well for environments where the database and the cron host are the same machine or where a centralized cron server has network access to the database.

The pg_cron extension runs inside the PostgreSQL process itself, which means it uses a database connection to schedule and execute jobs. This is useful in cloud environments where you may not have shell access but do have access to SQL — for example, RDS or Aurora PostgreSQL. After installing pg_cron, jobs are scheduled using standard cron syntax directly from SQL.

PostgreSQL:

-- Install pg_cron (add to postgresql.conf: shared_preload_libraries = 'pg_cron')
-- Then in psql:

-- Schedule nightly VACUUM ANALYZE on a high-churn table
SELECT cron.schedule(
    'nightly-vacuum-orders',
    '0 2 * * *',
    $$VACUUM ANALYZE orders;$$
);

-- Schedule weekly REINDEX on a bloated index
SELECT cron.schedule(
    'weekly-reindex-orders-idx',
    '0 1 * * 0',
    $$REINDEX INDEX CONCURRENTLY orders_customer_id_idx;$$
);

-- View all scheduled jobs
SELECT jobid, schedule, command, nodename, active
FROM cron.job;

-- Remove a job
SELECT cron.unschedule('nightly-vacuum-orders');

SQL Server:

-- Create a SQL Server Agent job for nightly index maintenance
USE msdb;
GO

EXEC sp_add_job
    @job_name = N'Nightly Index Maintenance';

EXEC sp_add_jobstep
    @job_name = N'Nightly Index Maintenance',
    @step_name = N'Rebuild Fragmented Indexes',
    @subsystem = N'TSQL',
    @command = N'
        DECLARE @sql NVARCHAR(MAX) = N'''';
        SELECT @sql += ''ALTER INDEX '' + QUOTENAME(i.name)
            + '' ON '' + QUOTENAME(s.name) + ''.'' + QUOTENAME(t.name)
            + CASE
                WHEN ps.avg_fragmentation_in_percent > 30 THEN '' REBUILD WITH (ONLINE = ON);''
                WHEN ps.avg_fragmentation_in_percent > 10 THEN '' REORGANIZE;''
              END
        FROM sys.dm_db_index_physical_stats(DB_ID(), NULL, NULL, NULL, ''LIMITED'') ps
        JOIN sys.indexes i ON ps.object_id = i.object_id AND ps.index_id = i.index_id
        JOIN sys.tables t ON i.object_id = t.object_id
        JOIN sys.schemas s ON t.schema_id = s.schema_id
        WHERE ps.avg_fragmentation_in_percent > 10
          AND i.name IS NOT NULL;
        EXEC sp_executesql @sql;
    ',
    @database_name = N'YourDatabase';

EXEC sp_add_schedule
    @schedule_name = N'Nightly 2AM',
    @freq_type = 4,        -- Daily
    @freq_interval = 1,
    @active_start_time = 20000; -- 2:00 AM

EXEC sp_attach_schedule
    @job_name = N'Nightly Index Maintenance',
    @schedule_name = N'Nightly 2AM';

EXEC sp_add_jobserver
    @job_name = N'Nightly Index Maintenance',
    @server_name = @@SERVERNAME;

The SQL Server example above shows the idiomatic pattern: dynamic T-SQL builds the maintenance statement based on current fragmentation levels from the DMV, so the job adapts to the actual state of indexes each time it runs rather than blindly rebuilding everything. This kind of adaptive maintenance is significantly more efficient than rebuilding all indexes unconditionally.

For PostgreSQL environments using OS cron, the pattern is typically a shell script that calls psql. The script can include logic to decide what maintenance to run based on bloat queries, making it equally adaptive.

One subtlety to keep in mind: in PostgreSQL, REINDEX CONCURRENTLY cannot run inside a transaction block, which means you cannot wrap it in pg_cron's default single-transaction execution model. As of PostgreSQL 14, pg_cron supports running commands outside transaction blocks using the $$ quoting syntax, but verify the behavior in your specific version before relying on it in production.

Health Checks and Alerting

Scheduled maintenance keeps the database in good shape. Health checks catch problems before they become outages. Automated alerting then closes the loop by notifying the right people at the right time.

A health check script typically checks a set of known indicators: connection pool saturation, replication lag, long-running queries, lock waits, tablespace usage, and autovacuum activity. The goal is to detect drift from normal operating conditions early enough to intervene before users are affected.

PostgreSQL:

-- Check for sessions blocked longer than 5 minutes
SELECT
    pid,
    now() - pg_stat_activity.query_start AS duration,
    query,
    state,
    wait_event_type,
    wait_event
FROM pg_stat_activity
WHERE state != 'idle'
  AND (now() - pg_stat_activity.query_start) > INTERVAL '5 minutes'
ORDER BY duration DESC;

-- Check replication lag in bytes for each standby
SELECT
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    (sent_lsn - replay_lsn) AS replication_lag_bytes
FROM pg_stat_replication
ORDER BY replication_lag_bytes DESC;

SQL Server:

-- Check for blocked sessions longer than 5 minutes
SELECT
    r.session_id,
    r.blocking_session_id,
    r.wait_type,
    r.wait_time / 1000.0 AS wait_seconds,
    SUBSTRING(st.text, (r.statement_start_offset / 2) + 1,
        ((CASE r.statement_end_offset
            WHEN -1 THEN DATALENGTH(st.text)
            ELSE r.statement_end_offset
          END - r.statement_start_offset) / 2) + 1) AS current_statement,
    s.login_name,
    s.host_name,
    s.program_name
FROM sys.dm_exec_requests r
JOIN sys.dm_exec_sessions s ON r.session_id = s.session_id
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) st
WHERE r.blocking_session_id > 0
  AND r.wait_time > 300000  -- 5 minutes in milliseconds
ORDER BY r.wait_time DESC;

-- Check AG replica synchronization state
SELECT
    ag.name AS availability_group,
    ar.replica_server_name,
    ars.role_desc,
    ars.synchronization_health_desc,
    ars.connected_state_desc,
    drs.synchronization_state_desc,
    drs.log_send_queue_size,
    drs.redo_queue_size
FROM sys.availability_groups ag
JOIN sys.availability_replicas ar ON ag.group_id = ar.group_id
JOIN sys.dm_hadr_availability_replica_states ars
    ON ar.replica_id = ars.replica_id
JOIN sys.dm_hadr_database_replica_states drs
    ON ar.replica_id = drs.replica_id
ORDER BY ag.name, ar.replica_server_name;

These queries become useful as automation building blocks. A Python or Bash script can run them on a schedule, compare the results against thresholds, and send alerts when conditions are met. The alerting channel can be email, Slack, PagerDuty, or any webhook-capable system — the database query just produces the data, and the surrounding script handles the routing.

A practical pattern is to store health check results in a monitoring table rather than discarding them after each run. This creates a time series of operational state that can be queried later to understand trends — for example, whether replication lag has been gradually increasing over the past week, which might indicate a growing workload imbalance that requires attention before it becomes a failure.

For SQL Server environments, Database Mail provides a native alerting mechanism that can be triggered from T-SQL directly. Combined with SQL Server Agent alerts, you can configure alerts that fire when specific errors occur in the SQL Server error log or when performance counter thresholds are crossed, without any external scripting layer.

Automating Database Deployments and Schema Changes

Maintenance automation keeps the database healthy. Deployment automation keeps it current. Every schema change that goes to production — a new table, a modified column, a new index — should pass through a repeatable, auditable process rather than being applied by hand.

The gold standard here is migration-based deployments. Tools like Flyway, Liquibase, and Alembic manage a versioned history of schema changes. Each change is a numbered script. The tool tracks which scripts have been applied and applies any that have not. Running the tool twice is safe — it will not apply a script that has already been applied.

This approach works across both database platforms and solves several problems simultaneously. First, it makes deployments repeatable: the exact same scripts are applied to development, staging, and production environments in order. Second, it creates an audit trail: you can see exactly what changed, when, and in what order. Third, it enables rollback planning: each migration can optionally have a corresponding undo script.

For PostgreSQL, online schema changes require additional care. Adding a column with a default in PostgreSQL versions before 11 locked the table for the entire duration of the rewrite. Since PostgreSQL 11, adding a column with a non-volatile default is instant because the default is stored in metadata rather than written to every row. For index creation, CREATE INDEX CONCURRENTLY should be the default in production because it builds the index without holding a lock that blocks writes.

For SQL Server, online index operations are available in Enterprise Edition using the ONLINE = ON option. This allows ALTER INDEX REBUILD and CREATE INDEX to run without blocking reads and writes for the duration of the operation. In Standard Edition, you must plan these operations for low-traffic windows.

A well-designed deployment automation system also includes pre-deployment and post-deployment checks. Before applying a migration, the system verifies that the database is in the expected state — that the previous migration was applied successfully. After applying, it runs validation queries to confirm the new objects exist and are accessible. If anything fails, the deployment pipeline stops and raises an alert.

For environments with strict change management requirements, the pipeline can be configured to generate a SQL preview of what will be applied and require human approval before executing. This gives teams the benefit of automation — repeatability, auditability, dependency management — while preserving a checkpoint for sensitive environments.

Building Operational Runbooks as Code

A runbook is a documented procedure for handling a known situation: how to failover to a standby, how to recover from a corrupted index, how to add a new database user with the correct permissions. Traditionally, runbooks are Word documents or wiki pages. The problem with text-based runbooks is that they drift out of date, they depend on the reader correctly interpreting and executing each step, and they cannot be tested.

Runbooks as code means encoding those procedures as executable scripts. When a standby failover needs to happen, an engineer runs a script rather than reading a document and typing commands. The script encodes the current knowledge of the correct procedure, executes it consistently, logs what it did, and handles errors.

PostgreSQL:

-- Automated user provisioning with standard role assignment
-- This would live in a provisioning script called with parameters

DO $$
DECLARE
    v_username TEXT := 'app_service_acct';
    v_password TEXT := 'replace_with_vault_secret';
BEGIN
    -- Create the role if it does not exist
    IF NOT EXISTS (
        SELECT 1 FROM pg_catalog.pg_roles WHERE rolname = v_username
    ) THEN
        EXECUTE format(
            'CREATE ROLE %I WITH LOGIN PASSWORD %L NOSUPERUSER NOCREATEDB NOCREATEROLE',
            v_username, v_password
        );
    END IF;

    -- Grant standard application permissions
    EXECUTE format('GRANT CONNECT ON DATABASE appdb TO %I', v_username);
    EXECUTE format('GRANT USAGE ON SCHEMA public TO %I', v_username);
    EXECUTE format('GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO %I', v_username);
    EXECUTE format('GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO %I', v_username);

    -- Ensure future objects are also accessible
    EXECUTE format(
        'ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO %I',
        v_username
    );

    RAISE NOTICE 'User % provisioned successfully', v_username;
END;
$$;

SQL Server:

-- Automated login and user provisioning
DECLARE @login_name SYSNAME = N'app_service_acct';
DECLARE @db_name SYSNAME = N'AppDatabase';
DECLARE @sql NVARCHAR(MAX);

-- Create server login if it does not exist
IF NOT EXISTS (SELECT 1 FROM sys.server_principals WHERE name = @login_name)
BEGIN
    SET @sql = N'CREATE LOGIN ' + QUOTENAME(@login_name)
        + N' WITH PASSWORD = N''replace_with_vault_secret'','
        + N' DEFAULT_DATABASE = ' + QUOTENAME(@db_name) + N','
        + N' CHECK_EXPIRATION = OFF, CHECK_POLICY = ON;';
    EXEC sp_executesql @sql;
    PRINT 'Login created: ' + @login_name;
END
ELSE
    PRINT 'Login already exists: ' + @login_name;

-- Create database user and assign role
SET @sql = N'USE ' + QUOTENAME(@db_name) + N';
IF NOT EXISTS (SELECT 1 FROM sys.database_principals WHERE name = ' + QUOTENAME(@login_name, '''') + N')
BEGIN
    CREATE USER ' + QUOTENAME(@login_name) + N' FOR LOGIN ' + QUOTENAME(@login_name) + N';
    ALTER ROLE db_datareader ADD MEMBER ' + QUOTENAME(@login_name) + N';
    ALTER ROLE db_datawriter ADD MEMBER ' + QUOTENAME(@login_name) + N';
    PRINT ''User created and roles assigned: ' + @login_name + N''';
END
ELSE
    PRINT ''User already exists: ' + @login_name + N''';';

EXEC sp_executesql @sql;

The idempotency here is intentional. Both scripts check whether the object already exists before trying to create it. This means the script can be run multiple times safely — once during initial provisioning, again if someone is verifying the state, and again if something went wrong the first time. Idempotency is a design principle that makes automation reliable under real-world conditions where things do not always go as planned.

Beyond user provisioning, runbook scripts cover common scenarios like rotating backup retention policies, updating connection pool configurations across application servers, archiving old partitions, and refreshing statistics on tables that the autovacuum daemon has not analyzed recently. Each of these procedures, once encoded and tested, becomes a reliable tool rather than a source of anxiety.

Version controlling these scripts in Git provides the same benefits as version-controlling application code: history, diff review, rollback, and branching. When a procedure changes — because the environment has changed or a better approach has been found — the change goes through a pull request and code review before being deployed, just like any other code change.

Integrating Database Automation with Broader Infrastructure Pipelines

Modern infrastructure runs on CI/CD pipelines, infrastructure-as-code tools like Terraform and Ansible, container orchestration platforms, and monitoring systems that produce structured telemetry. Database automation does not live in isolation from these systems — it needs to integrate with them.

The most common integration point is the deployment pipeline. When an application deploys a new version, the pipeline should also apply any pending database migrations before the application containers start. This requires that migrations be idempotent and that the migration tool be available inside the pipeline environment. Flyway and Liquibase both publish Docker images for this purpose, making it straightforward to include a migration step in a Kubernetes-based deployment pipeline.

A second integration point is secrets management. Scripts that connect to databases need credentials, and those credentials must not be stored in plaintext in the script or in a version control system. Modern environments solve this with secrets managers — HashiCorp Vault, AWS Secrets Manager, Azure Key Vault. The automation script calls the secrets manager API at runtime to retrieve credentials and passes them to the database client. This eliminates hardcoded passwords and centralizes credential rotation: when a password rotates in the secrets manager, every script that fetches it at runtime automatically uses the new password.

A third integration point is observability. Automated jobs should emit structured logs that feed into the same logging and monitoring system used by the rest of the infrastructure. When a maintenance job runs, it should log its start time, end time, what it did, and whether it succeeded. When an alert fires, it should include enough context — which server, which database, what the metric value was, what the threshold is — for the on-call engineer to understand the situation without having to run additional queries. Storing job outcomes in a dedicated database table alongside timestamps enables trend analysis: you can tell whether nightly maintenance jobs are taking longer over time, which might indicate that table sizes are growing faster than expected.

For PostgreSQL environments running on Kubernetes, the pg_cron approach can be supplemented or replaced by Kubernetes CronJobs that run database maintenance scripts in containers. This gives you the full Kubernetes operational model — resource limits, restart policies, pod logs, events — applied to database maintenance. The trade-off is that the maintenance jobs are now external to the database process, which means they count toward the connection limit and must handle the case where the database is temporarily unavailable.

For SQL Server in Azure, Azure Automation Runbooks and Azure Logic Apps provide managed scheduling and workflow capabilities that can execute T-SQL against Azure SQL Database or Managed Instance. These services handle the infrastructure concerns of running scheduled jobs — compute, availability, retry logic — so the DBA can focus on the SQL logic rather than the operational concerns of keeping a scheduling service running.

The most sophisticated environments combine multiple layers: database-native scheduling for lightweight recurring tasks, pipeline-based automation for deployments and migrations, external orchestration tools for complex multi-step workflows, and centralized monitoring for visibility across all of it. Reaching that state is not a single project — it is the result of iterating on automation over time, replacing manual processes one at a time with scripted, tested, version-controlled equivalents.

---

Key Takeaways

  • → Automation makes DBA decisions persistent and reliable, replacing processes that depend on memory or availability with scripts that execute consistently whether or not anyone is watching.
  • → PostgreSQL automation relies on a combination of OS-level cron, the `pg_cron` extension for database-native scheduling, and external scripting in Bash or Python; SQL Server provides SQL Server Agent as a built-in, feature-complete job scheduling framework.
  • → Adaptive maintenance — where scripts query current state (fragmentation percentages, bloat ratios, vacuum statistics) and perform only the work that is actually needed — is significantly more efficient than fixed-scope maintenance that runs unconditionally.
  • → Runbooks encoded as idempotent scripts provide the repeatability and testability of automation while preserving the institutional knowledge that would otherwise live only in documentation that drifts out of date.
  • → Production-grade automation integrates with secrets management for secure credential handling, version control for change tracking and rollback, and observability pipelines so that job outcomes are visible in the same monitoring system used by the rest of the infrastructure.
