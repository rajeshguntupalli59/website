Chapter 13: Monitoring and Alerts

A database that runs unattended without proper monitoring is an accident waiting to happen. Monitoring is the practice of continuously observing the health, performance, and behavior of your database systems so that problems are caught before users notice them — or before they escalate into outages. Alerting is the mechanism that turns observations into action. Together, they form the operational backbone of any serious DBA practice. This chapter walks through what to monitor, how to monitor it effectively in both PostgreSQL and SQL Server, how to build alerts that are useful rather than noisy, and how to construct the kind of observability infrastructure that keeps production systems healthy and recoverable.

---

Why Monitoring Is a DBA's First Line of Defense

Most database problems do not appear out of nowhere. A table bloats slowly over weeks. A query plan changes and a query that used to run in 200 milliseconds starts taking 4 seconds. A disk fills to 95% capacity before anyone notices. Connection counts creep upward as an application leaks connections during a traffic spike. None of these are sudden events — they are trends, and trends are only visible if you are watching.

The distinction between *monitoring* and *alerting* is worth stating clearly. Monitoring is passive and continuous: it collects data about the state of your system over time. Alerting is reactive and conditional: it fires when monitored data crosses a threshold you define. You need both. Monitoring without alerting means someone has to manually look at dashboards to catch problems, which is neither scalable nor reliable. Alerting without monitoring means you have no historical context to understand why an alert fired or how a problem evolved.

A well-designed monitoring strategy covers four broad categories:

Availability — Is the database instance reachable? Are replicas healthy? Is the listener or service running?

Performance — How long are queries taking? Are waits accumulating? Is I/O throughput within expected bounds?

Capacity — How full are the disks? How large are tables and indexes? How much transaction log is being generated?

Correctness — Are backups succeeding? Are replication lag values acceptable? Are autovacuum and maintenance jobs running on schedule?

Each of these categories demands different tools and different alert thresholds. A DBA who only monitors query performance will be blindsided by a full disk. A DBA who only monitors availability will miss the slow degradation of query plans over time.

Core Metrics: What PostgreSQL and SQL Server Expose

Both database engines expose a rich set of internal metrics through system views and dynamic management views. Understanding what is available — and what each metric actually means — is the foundation of effective monitoring.

PostgreSQL System Catalog and Statistics Views

PostgreSQL exposes runtime statistics through the pg_stat_* family of views. These are populated by the statistics collector, a background process that tracks activity across the instance. The most important views for day-to-day monitoring include:

  • → `pg_stat_activity` — active connections, their state, what query they are running, and how long they have been waiting
  • → `pg_stat_bgwriter` — background writer and checkpoint activity
  • → `pg_stat_database` — per-database transaction counts, cache hit ratios, and deadlocks
  • → `pg_stat_user_tables` — table-level statistics including dead tuple counts, which are essential for tracking bloat
  • → `pg_stat_user_indexes` — index usage statistics
  • → `pg_locks` — current lock activity, essential for diagnosing blocking chains

    • SQL Server Dynamic Management Views

    SQL Server exposes runtime information through Dynamic Management Views (DMVs) in the sys schema. Key DMVs for monitoring include:

  • → `sys.dm_exec_requests` — currently executing requests, their wait types, CPU time, and elapsed time
  • → `sys.dm_exec_sessions` — all connected sessions with authentication and resource usage
  • → `sys.dm_os_wait_stats` — cumulative wait statistics since the last SQL Server restart
  • → `sys.dm_db_index_usage_stats` — index seeks, scans, and lookups since last restart
  • → `sys.dm_io_virtual_file_stats` — file-level I/O statistics
  • → `sys.dm_exec_query_stats` — aggregated performance statistics for cached query plans

    • Let's look at the two most foundational monitoring queries: checking active sessions and connection counts.

    PostgreSQL:

    -- Active connections grouped by state and database
SELECT
    datname,
    state,
    COUNT(*) AS connection_count,
    MAX(EXTRACT(EPOCH FROM (NOW() - state_change))) AS longest_in_state_seconds
FROM pg_stat_activity
WHERE pid <> pg_backend_pid()
GROUP BY datname, state
ORDER BY connection_count DESC;

    SQL Server:

    -- Active connections grouped by status and database
SELECT
    DB_NAME(s.database_id)      AS database_name,
    s.status,
    COUNT(*)                    AS connection_count,
    MAX(DATEDIFF(SECOND, s.last_request_start_time, GETDATE())) AS longest_in_state_seconds
FROM sys.dm_exec_sessions s
WHERE s.is_user_process = 1
GROUP BY s.database_id, s.status
ORDER BY connection_count DESC;

    Connection monitoring is often the first thing that reveals an application problem. A sudden spike in idle-in-transaction connections in PostgreSQL, or a surge of suspended sessions in SQL Server, almost always indicates a lock contention issue or a misbehaving application that is not committing or closing transactions properly.

    Cache hit ratio is another foundational metric. When the buffer cache hit ratio drops significantly, it usually means the working set no longer fits in memory, and you are experiencing increased physical I/O.

    PostgreSQL:

    -- Buffer cache hit ratio per database
SELECT
    datname,
    blks_hit,
    blks_read,
    ROUND(
        blks_hit::NUMERIC / NULLIF(blks_hit + blks_read, 0) * 100,
        2
    ) AS cache_hit_pct
FROM pg_stat_database
WHERE datname NOT IN ('template0', 'template1')
ORDER BY cache_hit_pct ASC;

    SQL Server:

    -- Buffer cache hit ratio from performance counters
SELECT
    object_name,
    counter_name,
    ROUND(cntr_value, 2) AS cache_hit_pct
FROM sys.dm_os_performance_counters
WHERE counter_name = 'Buffer cache hit ratio'
  AND object_name LIKE '%Buffer Manager%';

    A healthy PostgreSQL buffer cache hit ratio should be above 98–99% for an OLTP workload with a properly sized shared_buffers. In SQL Server, a Buffer Cache Hit Ratio below 95% on a production OLTP system is worth investigating.

    Identifying Blocking, Long-Running Queries, and Waits

    Blocking is one of the most common causes of user-visible slowness in relational databases. When one session holds a lock and another session is waiting for that same lock, the waiting session is blocked. If enough sessions accumulate in this state, the application becomes unresponsive — even though the database is technically "up."

    In PostgreSQL, blocking appears through pg_locks joined with pg_stat_activity. In SQL Server, blocking is exposed through sys.dm_exec_requests and the dedicated sys.dm_os_waiting_tasks DMV.

    PostgreSQL:

    -- Blocking chains: who is blocking whom
SELECT
    blocked.pid                    AS blocked_pid,
    blocked.query                  AS blocked_query,
    blocked.state                  AS blocked_state,
    EXTRACT(EPOCH FROM NOW() - blocked.query_start)::INT AS wait_seconds,
    blocking.pid                   AS blocking_pid,
    blocking.query                 AS blocking_query,
    blocking.state                 AS blocking_state
FROM pg_stat_activity AS blocked
JOIN pg_locks         AS bl  ON bl.pid = blocked.pid AND NOT bl.granted
JOIN pg_locks         AS kl  ON kl.transactionid = bl.transactionid AND kl.granted
JOIN pg_stat_activity AS blocking ON blocking.pid = kl.pid
WHERE blocked.pid <> blocking.pid
ORDER BY wait_seconds DESC;

    SQL Server:

    -- Blocking chains with head blocker and wait type
SELECT
    r.session_id                        AS blocked_session,
    r.blocking_session_id               AS blocking_session,
    r.wait_type,
    r.wait_time / 1000                  AS wait_seconds,
    SUBSTRING(st.text, (r.statement_start_offset / 2) + 1,
        ((CASE r.statement_end_offset
            WHEN -1 THEN DATALENGTH(st.text)
            ELSE r.statement_end_offset END
          - r.statement_start_offset) / 2) + 1) AS blocked_query,
    DB_NAME(r.database_id)              AS database_name
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) st
WHERE r.blocking_session_id > 0
ORDER BY r.wait_time DESC;

    Wait statistics provide a higher-level view of where time is being spent across the entire instance. Rather than looking at individual sessions, wait stats tell you what *type* of resource the database is waiting on the most — disk I/O, CPU, network, locks, or memory.

    In SQL Server, sys.dm_os_wait_stats is the primary source. The catch is that it accumulates since the last restart or last reset, so you need to either take delta snapshots or normalize by elapsed time to get meaningful trending data. Common wait types to watch include PAGEIOLATCH_SH (disk read bottleneck), LCK_M_* (lock waits), CXPACKET (parallelism), and ASYNC_NETWORK_IO (slow clients or network).

    In PostgreSQL, pg_stat_activity shows per-session wait events through the wait_event_type and wait_event columns. There is no cumulative wait stats view equivalent to SQL Server's dm_os_wait_stats in vanilla PostgreSQL, though extensions like pg_wait_sampling can be added to capture this data over time.

    Long-running queries are another category that demands attention. A query running for more than a few minutes in an OLTP system is almost always a problem — either a rogue report running against production, a bulk operation without batching, or a deadlock that has not yet been detected.

    Monitoring for long-running queries should be part of your baseline automation, run every few minutes, and alert when thresholds are exceeded.

    Table Bloat, Autovacuum, and Maintenance Monitoring in PostgreSQL

    PostgreSQL has a unique maintenance concern that SQL Server DBAs moving to Postgres often underestimate: table and index bloat caused by its MVCC implementation. When rows are updated or deleted in PostgreSQL, the old row versions are not immediately removed. They become "dead tuples" — invisible to new transactions but still occupying physical space on disk. The autovacuum daemon is responsible for cleaning these up, but it can fall behind on high-write tables, and when it does, the consequences include bloated tables, wasted disk space, degraded query performance, and eventually — on old enough transactions — transaction ID wraparound, which can take a database offline.

    Monitoring dead tuple accumulation and autovacuum activity is therefore not optional in PostgreSQL — it is a core DBA responsibility.

    PostgreSQL:

    -- Tables with high dead tuple ratios — bloat candidates
SELECT
    schemaname,
    relname                                              AS table_name,
    n_live_tup,
    n_dead_tup,
    ROUND(n_dead_tup::NUMERIC / NULLIF(n_live_tup + n_dead_tup, 0) * 100, 2)
                                                         AS dead_tuple_pct,
    last_autovacuum,
    last_autoanalyze,
    autovacuum_count
FROM pg_stat_user_tables
WHERE n_live_tup + n_dead_tup > 1000
ORDER BY dead_tuple_pct DESC
LIMIT 20;

    A dead tuple percentage above 10–20% on a heavily queried table is a signal that autovacuum is not keeping up. This could be because the table's autovacuum settings need tuning, because the autovacuum daemon is overloaded, or because there is a long-running transaction holding back the xmin horizon and preventing cleanup from proceeding.

    Transaction ID wraparound is the most serious autovacuum-related risk. PostgreSQL uses 32-bit transaction IDs, and when the age of a table's relfrozenxid approaches 2 billion, the database will begin refusing new writes to protect data integrity — a forced shutdown that requires manual intervention.

    -- PostgreSQL: Monitor transaction ID age — alert if approaching limits
SELECT
    datname,
    age(datfrozenxid)                        AS xid_age,
    2000000000 - age(datfrozenxid)           AS xids_remaining,
    ROUND(age(datfrozenxid)::NUMERIC / 2000000000 * 100, 2)
                                             AS pct_toward_wraparound
FROM pg_database
ORDER BY xid_age DESC;

    Alert when xid_age exceeds 1.5 billion. At that point you need to schedule a VACUUM FREEZE on the affected tables immediately. Most production PostgreSQL deployments set alert thresholds much earlier — around 500 million or 1 billion — to give enough lead time.

    SQL Server does not have an equivalent of autovacuum or MVCC dead tuples, but it does require monitoring for index fragmentation, which degrades over time as rows are inserted and updated:

    -- SQL Server: Index fragmentation for tables with >1000 pages
SELECT
    OBJECT_NAME(ips.object_id)              AS table_name,
    i.name                                  AS index_name,
    ips.index_type_desc,
    ROUND(ips.avg_fragmentation_in_percent, 2) AS fragmentation_pct,
    ips.page_count
FROM sys.dm_db_index_physical_stats(
         DB_ID(), NULL, NULL, NULL, 'LIMITED') ips
JOIN sys.indexes i
    ON i.object_id = ips.object_id
   AND i.index_id  = ips.index_id
WHERE ips.page_count > 1000
  AND ips.avg_fragmentation_in_percent > 10
ORDER BY fragmentation_pct DESC;

    General convention: rebuild indexes above 30% fragmentation, reorganize indexes between 10% and 30%, and leave indexes below 10% alone. This query forms the basis of an automated index maintenance job.

    Building Alerts That Are Useful, Not Noisy

    The goal of an alert is to notify the right person about the right problem with enough context to act. Poorly designed alerts destroy their own value. If an alert fires constantly for minor fluctuations, the team starts ignoring it — a dangerous situation called "alert fatigue." If thresholds are set too conservatively, real problems go unreported. Getting this right requires thought about what each metric actually means in context, what normal looks like, and what constitutes a real emergency versus a transient blip.

    Alert Design Principles

    *Threshold awareness* — Not every metric has a universal threshold. A cache hit ratio of 96% might be completely acceptable in a read-intensive analytics workload that regularly scans large portions of disk, while the same ratio would be alarming for a highly optimized OLTP system. Set thresholds based on your specific system's baseline, not on generic internet benchmarks.

    *Duration-based triggering* — Alert on sustained conditions, not instantaneous ones. If CPU spikes to 90% for 10 seconds during an autovacuum run, that is not worth waking someone up. If CPU is at 90% for 5 consecutive minutes, that is worth knowing about. Build alerts to require a condition to persist over multiple collection intervals before firing.

    *Severity tiering* — Not every problem is a P1 incident. Define tiers: informational (log it, review weekly), warning (Slack notification, review within a few hours), critical (PagerDuty, wake someone up). A disk at 80% full might be a warning; a disk at 95% full is critical.

    *Alert on trends, not just thresholds* — A disk that goes from 60% to 85% full over 24 hours is more urgent than one that has been sitting at 82% for six months. Rate-of-change alerting catches problems before they breach absolute thresholds.

    Common Alert Scenarios and Their Recommended Thresholds

    | Metric | Warning | Critical |

    |---|---|---|

    | Disk usage | 80% | 90% |

    | Connection usage (vs. max) | 75% | 90% |

    | Cache hit ratio (OLTP) | < 98% | < 95% |

    | Long-running query | > 5 min | > 15 min |

    | Blocked sessions | Any block > 1 min | Block chain > 5 sessions |

    | Replication lag | > 30 seconds | > 5 minutes |

    | XID age (PostgreSQL) | > 500M | > 1.5B |

    | Autovacuum failures | Any | Any |

    | Backup age | > 24 hours | > 48 hours |

    The backup age alert deserves special mention. A backup that has not run in 48 hours is a serious operational risk, but many teams do not monitor this at all until they need to restore something. Build a simple check: query your backup history and alert if the most recent successful backup is older than your defined RPO.

    PostgreSQL:

    -- Check age of last successful pg_basebackup or pgBackRest backup
-- In environments using a backup log table:
SELECT
    backup_type,
    MAX(completed_at)                        AS last_success,
    EXTRACT(EPOCH FROM NOW() - MAX(completed_at)) / 3600
                                             AS hours_since_last_backup
FROM dba_backup_log
WHERE status = 'SUCCESS'
GROUP BY backup_type
ORDER BY hours_since_last_backup DESC;

    SQL Server:

    -- Age of most recent successful backup per database and type
SELECT
    bs.database_name,
    bs.type                                  AS backup_type,
    MAX(bs.backup_finish_date)               AS last_success,
    DATEDIFF(HOUR, MAX(bs.backup_finish_date), GETDATE())
                                             AS hours_since_last_backup
FROM msdb.dbo.backupset bs
GROUP BY bs.database_name, bs.type
HAVING DATEDIFF(HOUR, MAX(bs.backup_finish_date), GETDATE()) > 24
ORDER BY hours_since_last_backup DESC;

    Building a Monitoring Infrastructure: Tools, Automation, and Dashboards

    Individual queries are useful for ad hoc investigation, but production monitoring requires infrastructure that collects data continuously, stores it historically, visualizes trends, and delivers alerts automatically. No DBA can — or should — manually run monitoring queries every few minutes around the clock.

    Open Source and Community Tools

    For PostgreSQL, pgBadger parses log files and produces detailed HTML reports on slow queries, lock waits, and connection patterns. It is excellent for after-the-fact analysis but does not provide real-time monitoring. pg_activity is a top-like tool for watching live PostgreSQL activity from the command line. For more comprehensive monitoring, Prometheus with postgres_exporter has become the dominant open-source stack: postgres_exporter scrapes pg_stat_* views on a configurable interval, exposes metrics in Prometheus format, and feeds into Grafana dashboards for visualization and alerting. Community-built Grafana dashboards for PostgreSQL are available on grafana.com and provide an excellent starting point.

    For SQL Server, SQL Server Management Studio (SSMS) includes Activity Monitor for real-time session viewing. sp_WhoIsActive, written by Adam Machanic and freely available, is the community-standard stored procedure for real-time session analysis — far more useful than the built-in sp_who2. For longer-term collection, SQL Server's built-in Performance Data Collector or third-party tools like SolarWinds DPA, SentryOne/SQL Sentry, and Redgate SQL Monitor provide enterprise-grade historical monitoring. For teams running SQL Server in cloud environments, Azure Monitor and the built-in Query Performance Insight in Azure SQL Database provide native monitoring without additional tooling.

    Query Store in SQL Server

    Query Store, available since SQL Server 2016, is one of the most valuable built-in monitoring features in the platform. It automatically captures query plans and runtime statistics over time, allowing you to compare plan performance across time periods, identify plan regressions, and force specific execution plans when a regression occurs. Enable Query Store on every production database:

    -- SQL Server: Enable Query Store with appropriate retention settings
ALTER DATABASE YourDatabase
SET QUERY_STORE = ON (
    OPERATION_MODE        = READ_WRITE,
    CLEANUP_POLICY        = (STALE_QUERY_THRESHOLD_DAYS = 30),
    DATA_FLUSH_INTERVAL_SECONDS = 900,
    INTERVAL_LENGTH_MINUTES     = 60,
    MAX_STORAGE_SIZE_MB         = 1024,
    QUERY_CAPTURE_MODE          = AUTO
);

    Once enabled, Query Store surfaces plan regressions through its built-in reports in SSMS, or you can query it directly through sys.query_store_* views.

    pg_stat_statements in PostgreSQL

    The equivalent in PostgreSQL is the pg_stat_statements extension, which must be explicitly enabled. Once loaded, it aggregates performance statistics for every unique query shape that passes through the instance — total calls, mean execution time, total I/O, and variance in execution time.

    -- PostgreSQL: Top 10 queries by total execution time
SELECT
    LEFT(query, 100)                          AS query_snippet,
    calls,
    ROUND(total_exec_time::NUMERIC, 2)        AS total_ms,
    ROUND(mean_exec_time::NUMERIC, 2)         AS avg_ms,
    ROUND(stddev_exec_time::NUMERIC, 2)       AS stddev_ms,
    rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;

    Queries with a high stddev_ms relative to mean_exec_time are candidates for investigation — the high variance indicates that the query performs very differently across executions, which often points to plan instability, parameter sniffing effects, or lock waits intermittently inflating runtime.

    Automated Alerting with Scripts and Jobs

    Not every team has budget for enterprise monitoring tools. A pragmatic alternative is to build lightweight alert scripts that run on a schedule. These scripts query the monitoring views, evaluate thresholds, and send notifications via email, Slack webhooks, or PagerDuty APIs when conditions are met.

    A typical structure for a PostgreSQL monitoring script might be a Python or bash script that connects via psql, runs a set of diagnostic queries, compares results to hardcoded or config-file-defined thresholds, and posts to a Slack webhook if any threshold is breached. The script runs every 5 minutes via cron. This approach is low-cost, transparent, and entirely under your control.

    For SQL Server, SQL Agent jobs can run T-SQL alert procedures on a schedule. These procedures query DMVs, check thresholds, and use msdb.dbo.sp_send_dbmail to fire email alerts. SQL Agent provides built-in alerting for SQL Server Agent jobs, error log events, and performance condition alerts — a feature set that is often underused.

    Regardless of which tools you use, the monitoring data is only valuable if it is *retained*. Time-series data that is deleted after 24 hours cannot help you debug why last Tuesday's batch job took three times longer than usual. Plan your retention: high-resolution metrics (every 30 seconds) for 7 days, rolled up to 1-minute averages for 30 days, and hourly aggregates for 1 year. This gives you both recent detail and long-term trend visibility without consuming excessive storage.

    Key Takeaways

  • → Monitoring covers four pillars — availability, performance, capacity, and correctness — and gaps in any one of them create blind spots that lead to preventable incidents.
  • → Both PostgreSQL and SQL Server expose rich internal metrics through system views and DMVs; the `pg_stat_*` family in PostgreSQL and the `sys.dm_*` DMV family in SQL Server are your primary data sources for everything from connection counts to query performance.
  • → Blocking and long-running queries are the most common causes of user-visible database slowness; monitoring for them in real time, with clear escalation
