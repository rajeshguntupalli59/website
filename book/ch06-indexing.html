Chapter 6: Indexing and Retrieval

Few decisions affect database performance more than how you index your tables. Get indexing right and queries that once took minutes run in milliseconds. Get it wrong and you'll spend your career chasing slow queries, watching disk I/O spike, and apologizing to application teams. This chapter walks through how indexes actually work under the hood, how query planners decide whether to use them, how to build and maintain a sensible indexing strategy, and how to push that strategy further when standard B-tree indexes aren't enough. Both PostgreSQL and SQL Server are covered throughout, because while the concepts overlap significantly, the implementation details diverge in ways that matter in production.

---

What an Index Is and How It Actually Works

An index is a separate data structure maintained alongside a table, designed to let the database engine locate rows without scanning every page in the table. The analogy to a book's index is apt but incomplete — a database index also maintains physical or logical pointers back to the actual data, and the database engine must decide at query time whether using that index is faster than ignoring it entirely.

The default index type in both PostgreSQL and SQL Server is the B-tree (balanced tree). A B-tree organizes values in sorted order across a hierarchy of nodes. At the top sits the root node, branching down through intermediate nodes to leaf nodes at the bottom. Each leaf node contains index entries and a pointer to the corresponding heap row (PostgreSQL) or row locator (SQL Server). When a query filters on an indexed column, the engine traverses the tree from root to leaf — typically in O(log n) time — rather than reading every data page in the table linearly.

The key word is "sorted." A B-tree index on a last_name column stores last names alphabetically across its leaf pages. This means the index supports not just equality lookups (WHERE last_name = 'Smith') but also range predicates (WHERE last_name BETWEEN 'R' AND 'T'), prefix matches (WHERE last_name LIKE 'Sm%'), and ORDER BY operations without a separate sort step. The sort order embedded in the index is one of the most underappreciated features of B-tree indexes — when a query's ORDER BY matches the index order, the planner can often skip sorting entirely, which is expensive for large result sets.

Under the hood, both databases store the index in pages (8KB by default in PostgreSQL, also 8KB by default in SQL Server). As rows are inserted, updated, or deleted, index pages fragment over time. Insertions near a full page cause a page split, dividing the page and creating slack space that gradually accumulates. This fragmentation is why index maintenance — rebuilding or reorganizing indexes — is a routine DBA task, not an occasional emergency.

In SQL Server, the leaf level of a clustered index is the table itself. The data pages are physically ordered by the clustered index key, which is why a table can have only one clustered index. Every other index is a nonclustered index, whose leaf pages store the clustered index key (or the Row ID if there's no clustered index) as a row locator. In PostgreSQL, there is no native clustered index in the SQL Server sense. The CLUSTER command physically reorders a table's heap pages to match an index, but this ordering is not maintained on subsequent writes — it's a one-time operation, not an ongoing constraint.

This distinction has real consequences. In SQL Server, a range scan on the clustered index reads contiguous data pages, which is extremely efficient. In PostgreSQL, a range scan using any index must follow heap pointers from index leaf entries back to the heap, which may involve random I/O across many heap pages — a pattern PostgreSQL calls a heap fetch, and the cost of which depends heavily on how physically ordered the heap already is (tracked as the correlation statistic in pg_stats).

How the Query Planner Decides Whether to Use an Index

Building an index doesn't guarantee the planner will use it. Both PostgreSQL and SQL Server use cost-based query optimizers that estimate the total cost of different execution plans and choose the cheapest one. Understanding this decision-making process is what separates a DBA who creates indexes from one who creates *effective* indexes.

The optimizer estimates two things: how selective the index is (how many rows it will eliminate), and how expensive the I/O path through that index will be compared to alternative paths. For a highly selective predicate — say, looking up a single customer by their email address — the index path is almost always cheaper. For a low-selectivity predicate — say, filtering a status column that contains only three distinct values across 50 million rows — a full table scan may actually be cheaper, because the index lookup would scatter I/O across too many heap pages.

The optimizer's cost estimates depend heavily on statistics: the distribution of values in each column, the number of distinct values, histograms of value frequency, and the correlation between column order and physical storage order. In PostgreSQL, the ANALYZE command gathers these statistics and stores them in pg_statistic, exposed through the view pg_stats. In SQL Server, statistics are gathered automatically by default and stored in objects accessible via sys.stats and viewable with DBCC SHOW_STATISTICS.

PostgreSQL:

-- View statistics for a specific table and column
SELECT
    attname,
    n_distinct,
    correlation,
    most_common_vals,
    most_common_freqs,
    histogram_bounds
FROM pg_stats
WHERE tablename = 'orders'
  AND attname = 'status';

SQL Server:

-- View statistics for a specific table and column
DBCC SHOW_STATISTICS ('dbo.orders', 'IX_orders_status');

-- Or query the stats metadata
SELECT
    s.name         AS stats_name,
    s.auto_created,
    s.user_created,
    sp.last_updated,
    sp.rows,
    sp.rows_sampled,
    sp.unfiltered_rows
FROM sys.stats s
CROSS APPLY sys.dm_db_stats_properties(s.object_id, s.stats_id) sp
WHERE OBJECT_NAME(s.object_id) = 'orders';

The EXPLAIN (PostgreSQL) and execution plan (SQL Server) outputs reveal what the planner chose and why. Reading these plans is one of the most important skills a DBA develops. The plan shows whether an index scan, index-only scan, bitmap index scan, or sequential scan was used, what the estimated vs. actual row counts were, and where in the plan execution time is concentrated.

PostgreSQL:

-- Analyze the execution plan with actual runtime statistics
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT customer_id, order_date, total_amount
FROM orders
WHERE customer_id = 10042
  AND order_date >= '2024-01-01';

SQL Server:

-- View estimated plan in SSMS or capture actual plan at runtime
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

SELECT customer_id, order_date, total_amount
FROM dbo.orders
WHERE customer_id = 10042
  AND order_date >= '2024-01-01';

-- Or use the DMV to capture recent query plans
SELECT TOP 10
    qs.total_elapsed_time / qs.execution_count AS avg_elapsed_us,
    qs.execution_count,
    SUBSTRING(st.text, (qs.statement_start_offset/2)+1,
        ((CASE qs.statement_end_offset WHEN -1 THEN DATALENGTH(st.text)
          ELSE qs.statement_end_offset END - qs.statement_start_offset)/2)+1) AS query_text,
    qp.query_plan
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
ORDER BY avg_elapsed_us DESC;

When you see the planner ignoring an index you believe should help, the common culprits are stale statistics, parameter sniffing (SQL Server), or plan caching effects. Running ANALYZE (PostgreSQL) or UPDATE STATISTICS (SQL Server) after significant data changes is often the first thing to try. If statistics are fresh but the index is still ignored, check the actual selectivity — the planner may simply be right that the index isn't worth using for that predicate.

One scenario worth understanding explicitly: a query that returns a large fraction of the table may trigger a sequential scan even when an index exists. PostgreSQL's enable_indexscan and enable_seqscan configuration parameters let you force the planner one way or another for testing purposes. In SQL Server, you can use query hints (WITH (INDEX(...))) to test the same. Never leave these forcing mechanisms in production code — use them only for investigation, then fix the underlying cause (usually statistics, index design, or the query itself).

Building and Maintaining Indexes in Production

Creating an index in production without disrupting users is something every DBA eventually needs to do. Both databases support online index creation, but the mechanics differ.

In PostgreSQL, the CREATE INDEX CONCURRENTLY option builds the index without taking a full table lock. The build happens in multiple passes: first, it scans the table without blocking writes; then, it processes changes that occurred during the first scan; finally, it validates the index. The downside is that the build takes longer, and if the process is interrupted, it leaves behind an invalid index that must be manually dropped. Always check pg_indexes or pg_stat_user_indexes after a concurrent build to confirm the index is valid.

In SQL Server, CREATE INDEX ... WITH (ONLINE = ON) performs a similar function for Enterprise Edition. The index is built alongside ongoing DML, with a brief schema modification lock taken only at the start and end of the operation. For Standard Edition, online index builds are not available, which is a real operational constraint in busy systems.

PostgreSQL:

-- Create an index without locking the table (safe for production)
CREATE INDEX CONCURRENTLY idx_orders_customer_date
ON orders (customer_id, order_date DESC);

-- Verify the index is valid after creation
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'orders'
  AND indexname = 'idx_orders_customer_date';

-- Check for invalid indexes (left behind by failed concurrent builds)
SELECT indexrelid::regclass AS index_name,
       indisvalid
FROM pg_index
WHERE indisvalid = false;

SQL Server:

-- Create an index online (Enterprise Edition)
CREATE NONCLUSTERED INDEX IX_orders_customer_date
ON dbo.orders (customer_id, order_date DESC)
WITH (ONLINE = ON, SORT_IN_TEMPDB = ON);

-- Check index fragmentation before deciding rebuild vs. reorganize
SELECT
    i.name                          AS index_name,
    ips.avg_fragmentation_in_percent,
    ips.page_count,
    ips.avg_page_space_used_in_percent
FROM sys.dm_db_index_physical_stats(
         DB_ID(), OBJECT_ID('dbo.orders'), NULL, NULL, 'LIMITED') ips
JOIN sys.indexes i
    ON ips.object_id = i.object_id
   AND ips.index_id  = i.index_id
WHERE ips.page_count > 100
ORDER BY ips.avg_fragmentation_in_percent DESC;

Index maintenance strategy should be data-driven. In SQL Server, the standard guidance is to reorganize indexes with 10–30% fragmentation (a lightweight, online operation that defragments leaf pages in place) and rebuild indexes above 30% (which rewrites the entire index, reclaims space, and updates statistics). In PostgreSQL, index bloat is addressed with REINDEX or REINDEX CONCURRENTLY (PostgreSQL 12+). The autovacuum process helps prevent table bloat but doesn't rebuild indexes — periodic REINDEX CONCURRENTLY is still necessary for heavily updated tables.

Fill factor deserves mention here. When an index page is built, fill factor determines how full each leaf page is left. A fill factor of 80 (SQL Server) or fillfactor = 80 (PostgreSQL) leaves 20% of each page empty, providing room for future insertions without immediate page splits. The trade-off is larger index size and slightly more I/O for reads. For tables that are mostly read, use a high fill factor (90–100). For tables with heavy insert or update patterns on sequential key columns, a lower fill factor reduces split activity. For tables with random inserts (e.g., UUID primary keys), low fill factors help considerably.

Index maintenance should also account for unused indexes. Every index on a table adds overhead to every INSERT, UPDATE, and DELETE operation on that table. A table with 12 indexes has 12 structures that must be kept current on every write — the read benefits of those indexes must outweigh the write cost. This is the balance that makes indexing a judgment call rather than a mechanical exercise.

Index Types and When Each One Fits

B-tree indexes handle the majority of workloads, but both PostgreSQL and SQL Server offer specialized index types that dramatically outperform B-trees for specific access patterns.

Hash Indexes

In PostgreSQL, hash indexes store a hash of the indexed value and support only equality lookups — no ranges, no ordering, no LIKE. Before PostgreSQL 10, hash indexes were not WAL-logged and were unsafe for crash recovery. Since PostgreSQL 10, they are fully crash-safe and can be a good choice for equality-only lookups on wide columns (like long strings), since the hash reduces the index entry size. SQL Server also supports hash indexes, but only for memory-optimized (In-Memory OLTP) tables, where they are the dominant index type for equality access on in-memory data.

GIN — Generalized Inverted Index (PostgreSQL)

GIN indexes are built for cases where a single column value contains multiple items that you want to search individually. The canonical use cases are full-text search (tsvector columns), JSONB columns, and array columns. A GIN index on a JSONB column allows queries like WHERE doc @> '{"status": "active"}' to run efficiently by indexing every key-value pair within the document. GIN indexes are larger and slower to build than B-trees, but they make queries against complex data types practical that would otherwise require full table scans.

GiST — Generalized Search Tree (PostgreSQL)

GiST is a framework that supports a family of index types: geometric data (PostGIS), range types, full-text search (as an alternative to GIN), and more. A GiST index on a geometry column supports spatial queries like "find all locations within 5km of this point." GiST trades some lookup efficiency for flexibility — it allows lossy indexing, meaning the index may return false positives that the engine then filters out in a recheck step.

BRIN — Block Range Index (PostgreSQL)

BRIN indexes are extremely compact. Rather than indexing every individual row, a BRIN index records the minimum and maximum value of an indexed column within each block range (a contiguous group of heap pages). For queries that filter on a column that is naturally ordered in the table — such as an order_date on an append-only orders table, or a sensor reading timestamp — BRIN can be 100x smaller than a B-tree with nearly comparable performance. A BRIN index on a 1TB time-series table might occupy less than 100MB. The limitation is that BRIN only helps when there is strong correlation between column values and physical storage order.

Filtered/Partial Indexes

Both PostgreSQL and SQL Server support partial indexes (called filtered indexes in SQL Server). A partial index is built only over rows that satisfy a WHERE condition. This is one of the most powerful indexing techniques available, and it's underused in most production systems.

Consider a jobs table with 100 million rows, where only 50,000 rows have status = 'pending'. A standard index on status would index all 100 million rows. A partial index on status = 'pending' indexes only 50,000 rows — it's smaller, faster to scan, and faster to update since completed jobs never touch it.

PostgreSQL:

-- Partial index covering only active/pending work
CREATE INDEX idx_jobs_pending
ON jobs (created_at, priority)
WHERE status = 'pending';

-- This query will use the partial index
SELECT job_id, created_at, priority
FROM jobs
WHERE status = 'pending'
  AND created_at > NOW() - INTERVAL '24 hours'
ORDER BY priority DESC;

SQL Server:

-- Filtered index covering only pending jobs
CREATE NONCLUSTERED INDEX IX_jobs_pending
ON dbo.jobs (created_at, priority)
WHERE status = 'pending';

-- Queries must include the filter predicate to use the filtered index
SELECT job_id, created_at, priority
FROM dbo.jobs
WHERE status = 'pending'
  AND created_at > DATEADD(HOUR, -24, GETDATE())
ORDER BY priority DESC;

Covering Indexes and INCLUDE Columns

A covering index satisfies a query entirely from the index without touching the base table. When the index contains all the columns needed by a query — both for filtering and for projection — the engine never has to follow a pointer back to the heap (PostgreSQL) or clustered index (SQL Server). This eliminates what PostgreSQL calls a heap fetch and SQL Server calls a key lookup, both of which can be expensive under high concurrency.

Both databases allow extra columns to be stored in the leaf level of the index without participating in the sort order, using the INCLUDE clause. This lets you add projection columns to an index without widening the sort key (which would make the index larger and slower for range scans).

PostgreSQL:

-- Covering index: customer_id is the search key, remaining columns are INCLUDEd
CREATE INDEX idx_orders_covering
ON orders (customer_id)
INCLUDE (order_date, total_amount, status);

SQL Server:

-- Covering index with included columns
CREATE NONCLUSTERED INDEX IX_orders_covering
ON dbo.orders (customer_id)
INCLUDE (order_date, total_amount, status);

In PostgreSQL, the Index-Only Scan is the plan node you want to see when you've built a covering index. The plan will confirm whether the index-only scan is being used, and the BUFFERS output will show whether heap pages are still being fetched (which happens when the visibility map indicates that not all heap pages are fully visible — a consequence of autovacuum lag). Running VACUUM on the table after bulk operations brings the visibility map current and allows index-only scans to skip heap fetches entirely.

Finding What's Missing, What's Redundant, and What's Hurting You

A production database that's been running for two or three years accumulates indexing debt: missing indexes on columns that are now heavily queried, duplicate indexes created by different developers who didn't realize an equivalent index already existed, and indexes that made sense when a feature launched but are now completely unused. Auditing the index landscape is a regular DBA responsibility, not a one-off exercise.

Identifying missing indexes

Both databases offer tooling to identify frequently accessed queries that lack supporting indexes.

In SQL Server, sys.dm_db_missing_index_details and sys.dm_db_missing_index_group_stats capture information about index opportunities the optimizer identified during query compilation. These DMVs are reset when SQL Server restarts, so they reflect only the period since the last restart (or the last DMV clear). The output is useful but should be treated as suggestions — the optimizer sometimes recommends indexes that would help individual queries but would harm overall workload throughput.

In PostgreSQL, there's no built-in equivalent, but pg_stat_statements combined with EXPLAIN automation gives a similar picture. Many organizations run automated pipelines that collect slow query logs, analyze query plans, and flag sequential scans on large tables where a viable index predicate exists.

PostgreSQL:

-- Find tables with high sequential scan counts — potential missing index candidates
SELECT
    relname                        AS table_name,
    seq_scan,
    seq_tup_read,
    idx_scan,
    seq_tup_read / NULLIF(seq_scan, 0) AS avg_rows_per_seq_scan,
    n_live_tup                     AS estimated_row_count
FROM pg_stat_user_tables
WHERE seq_scan > 0
  AND n_live_tup > 10000
ORDER BY seq_tup_read DESC
LIMIT 20;

SQL Server:

-- Missing index recommendations from the optimizer
SELECT
    mid.statement                              AS table_name,
    migs.avg_total_user_cost * migs.avg_user_impact
        * (migs.user_seeks + migs.user_scans)  AS index_benefit_score,
    migs.user_seeks,
    migs.user_scans,
    mid.equality_columns,
    mid.inequality_columns,
    mid.included_columns
FROM sys.dm_db_missing_index_groups mig
JOIN sys.dm_db_missing_index_group_stats migs
    ON mig.index_group_handle = migs.group_handle
JOIN sys.dm_db_missing_index_details mid
    ON mig.index_handle = mid.index_handle
WHERE migs.avg_total_user_cost * migs.avg_user_impact
        * (migs.user_seeks + migs.user_scans) > 1000
ORDER BY index_benefit_score DESC;

Identifying unused indexes

Unused indexes are pure write overhead. Every time a row is modified, the database must update all indexes on that table, whether or not those indexes are ever read. Removing unused indexes reduces DML overhead and can meaningfully improve write throughput on busy OLTP tables.

In PostgreSQL, pg_stat_user_indexes tracks index scans since the last statistics reset. An index with zero or near-zero idx_scan after weeks of normal workload is a strong candidate for removal — but verify against your pg_stat_statements data and consider whether the index might be heavily used during period batch jobs that haven't run during your observation window.

In SQL Server, sys.dm_db_index_usage_stats provides read and write counts per index since the last restart. An index with high user_updates (write overhead) and zero or near-zero user_seeks + user_scans + user_lookups is a candidate for removal.

Before dropping any index in production, always script it first. A single CREATE INDEX statement stored somewhere accessible means you can recreate the index within minutes if something that depended on it surfaces after the drop.

Identifying duplicate and redundant indexes

Duplicate indexes are exact duplicates — same columns, same order. Redundant indexes are indexes where one index's leading columns are a prefix of another's. For example, if you have an index on (customer_id) and another on (customer_id, order_date), the second index can serve all the same equality lookups as the first, making the first index redundant for most purposes (unless you need to support an index-only scan that doesn't need order_date).

Detecting these requires querying index metadata and comparing column lists. There is no single built-in tool for this in either database, but the query is straightforward to write against pg_index/pg_attribute in PostgreSQL or sys.index_columns/sys.columns in SQL Server.

Advanced Indexing for High-Throughput and Complex Workloads

At sufficient scale, standard indexing practice reaches its limits, and the strategies that work for a 10GB database need rethinking for a 10TB one. The following scenarios represent the kind of indexing problems that separate routine database administration from performance engineering.

UUID Primary Keys and Random Index Fragmentation

One of the most common performance problems in PostgreSQL and SQL Server systems that use UUID (universally unique identifier) primary keys is extreme index fragmentation. Because UUIDs are randomly distributed, new index entries land randomly across the B-tree leaf level rather than at the end. Every page fills up and splits constantly, leading to high fragmentation, low page density, and excessive I/O.

In PostgreSQL, using gen_random_uuid() (which produces UUID v4, fully random) as a primary key creates this problem. The practical solutions are: use a sequential surrogate key (BIGSERIAL) with the UUID as an alternate key, use UUID v7 (time-ordered UUIDs, available in PostgreSQL 17 via uuidv7() or installable via extensions), or use pg_idkit for ordered ID generation. In SQL Server, NEWSEQUENTIALID() generates UUIDs in sequential order suitable for clustered index keys, while NEWID() produces random UUIDs that cause the same fragmentation problem.
