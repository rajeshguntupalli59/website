Chapter 26: Event-Driven Design

Modern applications rarely operate in isolation. A user places an order, and a warehouse system needs to know. A payment clears, and a notification service fires. A record changes in one table, and three downstream consumers need to react. This chapter is about how databases fit into that world — how PostgreSQL and SQL Server can both participate in event-driven architectures, either as producers that emit events when data changes or as consumers that process event streams. We'll cover the mechanisms each platform provides natively, how to build reliable event pipelines on top of them, and where the edge cases and failure modes live in production systems that depend on this pattern.

---

What Event-Driven Architecture Means for a Database

Event-driven design is a pattern where systems communicate through discrete, timestamped facts about things that happened. Rather than one service querying another for current state ("what is the balance?"), services react to events ("the balance changed"). The database is almost always at the center of this, because the database is where state lives and where changes are authoritative.

For a DBA, event-driven architecture introduces a category of questions that don't arise in request/response systems. How do you guarantee an event is emitted if and only if the underlying data change commits? How do you ensure events are delivered in order? What happens when a consumer is offline for two hours? How do you avoid the dual-write problem, where you write to a database and then try to write to a message queue, and one of those writes fails?

The canonical answer to the dual-write problem is the Transactional Outbox Pattern: instead of writing directly to an external message broker, you write an event record into a table within the same transaction as your business data change. A separate process then reads from that outbox table and forwards events onward. If the process crashes, it can replay from where it left off. The event is guaranteed to exist if and only if the business transaction committed — atomicity is preserved because both writes happen in the same transaction.

This concept is straightforward, but the implementation details determine whether it works in production. Let's walk through how it works structurally, and then we'll look at the native mechanisms both PostgreSQL and SQL Server provide that build on or replace it.

---

The Transactional Outbox Pattern in Practice

The outbox table is simple by design. It holds event payloads alongside enough metadata for a processor to understand ordering, type, and delivery status.

PostgreSQL:

-- PostgreSQL outbox table
CREATE TABLE event_outbox (
    id            BIGSERIAL PRIMARY KEY,
    aggregate_id  TEXT        NOT NULL,
    aggregate_type TEXT       NOT NULL,
    event_type    TEXT        NOT NULL,
    payload       JSONB       NOT NULL,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    processed_at  TIMESTAMPTZ,
    retry_count   INT         NOT NULL DEFAULT 0
);

CREATE INDEX idx_outbox_unprocessed
    ON event_outbox (created_at)
    WHERE processed_at IS NULL;

SQL Server:

-- SQL Server outbox table
CREATE TABLE event_outbox (
    id             BIGINT IDENTITY(1,1) PRIMARY KEY,
    aggregate_id   NVARCHAR(255)  NOT NULL,
    aggregate_type NVARCHAR(255)  NOT NULL,
    event_type     NVARCHAR(255)  NOT NULL,
    payload        NVARCHAR(MAX)  NOT NULL,  -- store JSON as NVARCHAR(MAX)
    created_at     DATETIMEOFFSET NOT NULL DEFAULT SYSDATETIMEOFFSET(),
    processed_at   DATETIMEOFFSET NULL,
    retry_count    INT            NOT NULL DEFAULT 0
);

CREATE INDEX idx_outbox_unprocessed
    ON event_outbox (created_at)
    WHERE processed_at IS NULL;

Within application code, the business insert and the outbox insert share a transaction:

PostgreSQL:

-- PostgreSQL: atomic business write + event emission
BEGIN;

INSERT INTO orders (customer_id, total_amount, status)
VALUES ($1, $2, 'pending')
RETURNING id INTO TEMP TABLE new_order;

INSERT INTO event_outbox (aggregate_id, aggregate_type, event_type, payload)
SELECT
    id::TEXT,
    'Order',
    'OrderPlaced',
    jsonb_build_object(
        'order_id',     id,
        'customer_id',  $1,
        'total_amount', $2,
        'occurred_at',  NOW()
    )
FROM new_order;

COMMIT;

SQL Server:

-- SQL Server: atomic business write + event emission
BEGIN TRANSACTION;

DECLARE @new_order_id BIGINT;

INSERT INTO orders (customer_id, total_amount, status)
VALUES (@customer_id, @total_amount, 'pending');

SET @new_order_id = SCOPE_IDENTITY();

INSERT INTO event_outbox (aggregate_id, aggregate_type, event_type, payload)
VALUES (
    CAST(@new_order_id AS NVARCHAR(255)),
    'Order',
    'OrderPlaced',
    (SELECT
        @new_order_id  AS order_id,
        @customer_id   AS customer_id,
        @total_amount  AS total_amount,
        SYSDATETIMEOFFSET() AS occurred_at
     FOR JSON PATH, WITHOUT_ARRAY_WRAPPER)
);

COMMIT TRANSACTION;

The outbox relay process — whether it's a scheduled job, a dedicated microservice, or a database agent job — polls for unprocessed rows, forwards them to a broker like Kafka or RabbitMQ, and marks them processed. The relay must be idempotent: if it crashes between forwarding and marking, it will re-read the same row on restart. Downstream consumers therefore need to handle duplicate events, which is a design constraint worth communicating explicitly to application teams.

The partial index on processed_at IS NULL is not cosmetic. In a high-volume system, the outbox table can accumulate millions of rows. A full table scan on every relay cycle would quickly become a bottleneck. The partial index keeps the relay query fast because it only ever needs to see unprocessed rows — the rest are invisible to it.

One production concern that catches teams off guard: old processed rows accumulate indefinitely. You need a background archival or deletion job, or the table grows without bound and the partial index stops fitting in memory. The relay job should also enforce a maximum batch size so it doesn't lock large sets of rows during marking.

Change Data Capture: Letting the Database Emit Events Natively

Writing events from application code is reliable when applications are disciplined. It breaks down when data is modified through migration scripts, bulk imports, administrative tools, or legacy code paths that bypass the event-writing logic. The more durable solution is to have the database itself emit events for every data change, regardless of source. This is Change Data Capture — CDC.

Both PostgreSQL and SQL Server implement CDC by reading the transaction log rather than polling tables. The log is the most faithful record of what changed and when, and it imposes almost no overhead on the primary write path.

PostgreSQL Logical Replication and Logical Decoding

PostgreSQL exposes its write-ahead log (WAL) for consumption through logical decoding. The pg_logical replication slot mechanism lets an external process subscribe to a decoded stream of row-level changes. The popular open-source tool Debezium uses this mechanism to forward changes into Kafka, but you can also consume the stream directly using PostgreSQL's built-in replication protocol.

To use logical decoding, the server must have wal_level = logical set in postgresql.conf. This is now the default in PostgreSQL 15 and above.

PostgreSQL:

-- PostgreSQL: create a logical replication slot using the pgoutput plugin
SELECT pg_create_logical_replication_slot(
    'orders_cdc_slot',
    'pgoutput'
);

-- Peek at changes without advancing the slot (useful for testing)
SELECT *
FROM pg_logical_slot_peek_changes(
    'orders_cdc_slot',
    NULL,
    NULL,
    'proto_version', '1',
    'publication_names', 'orders_publication'
);

-- Create a publication for specific tables
CREATE PUBLICATION orders_publication
FOR TABLE orders, order_items;

-- Check slot lag — critical for monitoring
SELECT
    slot_name,
    database,
    active,
    pg_wal_lsn_diff(pg_current_wal_lsn(), confirmed_flush_lsn) AS lag_bytes
FROM pg_replication_slots;

Slot lag is the single most important metric in any logical replication setup. When a consumer falls behind, the WAL files it hasn't yet processed cannot be deleted. Unchecked slot lag can fill your disk and crash the primary. Every production deployment using logical replication slots should alert when lag exceeds a threshold — typically a few hundred megabytes in a busy system — and should drop the slot if the consumer has been absent for more than a few hours.

SQL Server Change Data Capture

SQL Server's CDC feature captures changes to specified tables by reading the transaction log and writing them into a set of system-maintained change tables. It's enabled at the database level and then per table.

SQL Server:

-- SQL Server: enable CDC on the database
EXEC sys.sp_cdc_enable_db;

-- Enable CDC on a specific table
EXEC sys.sp_cdc_enable_table
    @source_schema      = N'dbo',
    @source_name        = N'orders',
    @role_name          = N'cdc_reader',   -- security role for access
    @supports_net_changes = 1;             -- 1 = expose net changes per key

-- Query captured changes between two LSN values
DECLARE @from_lsn BINARY(10), @to_lsn BINARY(10);

SET @from_lsn = sys.fn_cdc_get_min_lsn('dbo_orders');
SET @to_lsn   = sys.fn_cdc_get_max_lsn();

SELECT
    __$operation,          -- 1=delete, 2=insert, 3=before-update, 4=after-update
    __$start_lsn,
    __$seqval,
    id,
    customer_id,
    total_amount,
    status
FROM cdc.fn_cdc_get_all_changes_dbo_orders(
    @from_lsn, @to_lsn, 'all'
);

The __$operation column is the key to understanding what type of change occurred. Updates appear as two rows: the before-image (__$operation = 3) and the after-image (__$operation = 4). If your consumer only needs the current state, filter for inserts (2) and after-update images (4), and treat deletes (1) as tombstone events.

SQL Server CDC depends on SQL Server Agent to run the capture and cleanup jobs. If Agent is stopped, change data accumulates in the transaction log and the change tables stop being populated. This is a common operational oversight — teams enable CDC and forget to monitor Agent health.

What CDC Does Not Give You

CDC provides the what and when of data changes, but it doesn't provide business context. A CDC event says "a row in the orders table changed" — it doesn't say "an order was placed" versus "an order was cancelled" versus "a shipping address was corrected." For event-driven systems where consumers care about semantic meaning, CDC events often need to be enriched or translated into domain events before being published. The outbox pattern and CDC are complementary: some teams use CDC to capture outbox table changes, avoiding the need for a separate relay process entirely.

Using LISTEN/NOTIFY and Service Broker for Lightweight Messaging

Not every event-driven need requires a full CDC pipeline or an external message broker. Both databases have built-in lightweight notification mechanisms that work well for low-to-moderate volume use cases: real-time dashboard updates, cache invalidation, workflow triggers, and inter-process coordination within a single database.

PostgreSQL LISTEN/NOTIFY

PostgreSQL's LISTEN/NOTIFY mechanism lets database sessions send and receive messages over named channels. A notifying session sends a string payload over a channel; all listening sessions on that channel receive it asynchronously. Notifications survive within a session but are not persisted — if no one is listening when a notification fires, it is silently dropped.

This makes LISTEN/NOTIFY unsuitable as a guaranteed delivery mechanism, but excellent as a wake-up signal for a relay process. The canonical pattern is: write to the outbox table within your transaction, then issue a NOTIFY at the end of the transaction. The relay process, which is sleeping in a LISTEN loop, wakes up immediately and processes the new rows. Without this, the relay must poll on a timer, introducing unnecessary latency.

A trigger on the outbox table can fire the notification automatically:

PostgreSQL:

-- PostgreSQL: trigger-based NOTIFY on outbox insert
CREATE OR REPLACE FUNCTION notify_outbox_insert()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
    PERFORM pg_notify(
        'outbox_event',
        json_build_object(
            'id',          NEW.id,
            'event_type',  NEW.event_type,
            'aggregate_id', NEW.aggregate_id
        )::TEXT
    );
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_notify_outbox
AFTER INSERT ON event_outbox
FOR EACH ROW EXECUTE FUNCTION notify_outbox_insert();

The relay subscribes with LISTEN outbox_event and blocks until a notification arrives, then immediately polls the outbox for unprocessed rows. This hybrid approach — outbox for durability, NOTIFY for low latency — is robust and elegant.

One constraint: the payload in a NOTIFY is limited to 8000 bytes. This is plenty for identifiers and metadata but not for large JSON documents. Keep notification payloads small; include identifiers and let the receiver query for full data if needed.

SQL Server Service Broker

SQL Server Service Broker is a more heavyweight native messaging system built directly into the SQL Server engine. It provides persistent queues, guaranteed ordered delivery, and transactional message enqueue/dequeue — all within the database. Unlike LISTEN/NOTIFY, messages survive server restarts and are not dropped if there's no active receiver.

Service Broker models communication as conversations between services, each backed by queues. The setup is verbose by modern standards, but the guarantees are strong.

SQL Server:

-- SQL Server: minimal Service Broker setup for event notifications
-- Enable Service Broker on the database
ALTER DATABASE OrdersDB SET ENABLE_BROKER WITH ROLLBACK IMMEDIATE;

-- Create message type, contract, queues, and services
CREATE MESSAGE TYPE [OrderEventMessage]
    VALIDATION = WELL_FORMED_XML;

CREATE CONTRACT [OrderEventContract]
    ([OrderEventMessage] SENT BY INITIATOR);

CREATE QUEUE OrderEventQueue;
CREATE SERVICE [OrderEventService]
    ON QUEUE OrderEventQueue ([OrderEventContract]);

-- Send a message within a transaction
DECLARE @conversation UNIQUEIDENTIFIER;
BEGIN DIALOG CONVERSATION @conversation
    FROM SERVICE [OrderEventService]
    TO SERVICE 'OrderEventService'
    ON CONTRACT [OrderEventContract]
    WITH ENCRYPTION = OFF;

SEND ON CONVERSATION @conversation
    MESSAGE TYPE [OrderEventMessage]
    (N'<OrderPlaced><OrderId>1001</OrderId><CustomerId>42</CustomerId></OrderPlaced>');

-- Receive a message
DECLARE @msg NVARCHAR(MAX);
RECEIVE TOP(1)
    @msg = CAST(message_body AS NVARCHAR(MAX))
FROM OrderEventQueue;

Service Broker is transactional — sending and receiving participate in BEGIN TRANSACTION/COMMIT semantics. A message that is received inside a transaction that rolls back returns to the queue. This makes it genuinely at-least-once with automatic retry semantics.

In practice, Service Broker fell out of fashion as external message brokers matured, partly because its XML-based message model is awkward in modern JSON-centric architectures, and partly because managing Broker infrastructure inside a database creates operational coupling that teams prefer to avoid. It remains a solid choice for internal-to-SQL-Server workflows, particularly for implementing background processing, saga choreography, or workflow retry patterns without introducing external dependencies.

Ordering, Idempotency, and Consumer Guarantees

The mechanics of emitting events are only half the challenge. The harder problems emerge on the consumer side: what order do events arrive in, what happens when the same event arrives twice, and what happens when events arrive out of order?

Ordering

A common misconception is that events produced from a single database table are naturally ordered by transaction commit time. They are not — not in a distributed sense, and not even locally in the presence of concurrent transactions. Consider two transactions, T1 and T2, that start in that order. T2 might commit first. A consumer reading the CDC stream or the outbox by created_at will see T2's events before T1's, even though T1 started first.

For most event-driven systems, per-entity ordering is what matters. Events about order #1001 should be processed in the order they were committed for that order. This is why CDC tools like Debezium partition Kafka topics by primary key — all changes to a given row go to the same partition, preserving order for that entity.

In an outbox setup, you can enforce per-entity ordering at the relay level by querying events grouped by aggregate_id and processing each group sequentially, or by using a sequence number per aggregate. Global ordering across all entities is usually unnecessary and expensive to guarantee.

Idempotency

At-least-once delivery is the realistic default in any reliable messaging system. Networks fail; processes crash; relay jobs restart. Consumers will receive duplicate events. The correct response is to design consumers to be idempotent — processing the same event twice should produce the same result as processing it once.

Idempotency strategies include: tracking processed event IDs in a table and skipping duplicates; using upsert semantics (INSERT ... ON CONFLICT DO UPDATE in PostgreSQL, MERGE in SQL Server) so re-processing a create event doesn't fail; or making business logic naturally idempotent (setting a status to "shipped" twice is harmless; incrementing a counter twice is not).

PostgreSQL:

-- PostgreSQL: idempotent event processing using upsert
INSERT INTO order_projections (order_id, status, last_event_id, updated_at)
VALUES ($1, $2, $3, NOW())
ON CONFLICT (order_id) DO UPDATE
    SET status        = EXCLUDED.status,
        last_event_id = EXCLUDED.last_event_id,
        updated_at    = EXCLUDED.updated_at
WHERE order_projections.last_event_id < EXCLUDED.last_event_id;
-- The WHERE clause prevents older events from overwriting newer state

SQL Server:

-- SQL Server: idempotent event processing using MERGE
MERGE INTO order_projections AS target
USING (VALUES (@order_id, @status, @last_event_id, SYSDATETIMEOFFSET()))
    AS source (order_id, status, last_event_id, updated_at)
ON target.order_id = source.order_id
WHEN MATCHED AND target.last_event_id < source.last_event_id THEN
    UPDATE SET
        status        = source.status,
        last_event_id = source.last_event_id,
        updated_at    = source.updated_at
WHEN NOT MATCHED THEN
    INSERT (order_id, status, last_event_id, updated_at)
    VALUES (source.order_id, source.status,
            source.last_event_id, source.updated_at);

The conditional on last_event_id in both examples is an application of optimistic ordering: only accept an event if it represents newer state than what's already stored. This pattern, sometimes called a "fenced" upsert, protects against out-of-order delivery corrupting your read models.

Dead Letter Queues

No event-driven system is complete without handling the case where a consumer repeatedly fails to process an event. An event that causes a null pointer exception in consumer code, or that references data that doesn't yet exist due to an ordering problem, will fail every time it's retried. Without a dead letter queue (DLQ), the consumer either blocks indefinitely or silently skips the event — both bad outcomes.

In an outbox-based system, the retry_count column does this work. After a configurable number of failures, the relay marks the event as permanently failed and routes it to a dead letter table for human inspection. SQL Server Service Broker has built-in poison message handling: a message that is rolled back five times is automatically moved to a dead letter queue. Designing this path explicitly, before you need it in production, prevents a single bad event from halting all processing.

Performance, Retention, and Production Operations

Event-driven systems introduce write amplification. Every business write now produces at least one additional row in an outbox or change table. At low volume this is invisible; at high volume it becomes a significant operational concern.

Write Amplification and Batching

A system inserting 10,000 orders per minute also inserts 10,000 outbox rows per minute. Depending on table layout and index cardinality, that can double the I/O budget. The mitigation is batching at the relay level — reading and forwarding events in batches rather than one at a time. This trades latency (events take slightly longer to be forwarded) for throughput.

The outbox relay should also use SELECT ... FOR UPDATE SKIP LOCKED in PostgreSQL or WITH (UPDLOCK, READPAST) in SQL Server to allow multiple relay instances to run concurrently without contending for the same rows. This is the correct pattern for competing consumers on a database-backed queue.

PostgreSQL:

-- PostgreSQL: batch relay with SKIP LOCKED for concurrent relay workers
WITH batch AS (
    SELECT id, aggregate_id, event_type, payload
    FROM event_outbox
    WHERE processed_at IS NULL
    ORDER BY created_at
    LIMIT 500
    FOR UPDATE SKIP LOCKED
)
UPDATE event_outbox
SET processed_at = NOW()
FROM batch
WHERE event_outbox.id = batch.id
RETURNING batch.*;

SQL Server:

-- SQL Server: batch relay with READPAST for concurrent relay workers
WITH batch AS (
    SELECT TOP 500 id, aggregate_id, event_type, payload
    FROM event_outbox WITH (UPDLOCK, READPAST)
    WHERE processed_at IS NULL
    ORDER BY created_at
)
UPDATE event_outbox
SET processed_at = SYSDATETIMEOFFSET()
OUTPUT
    DELETED.id,
    DELETED.aggregate_id,
    DELETED.event_type,
    DELETED.payload
FROM event_outbox
INNER JOIN batch ON event_outbox.id = batch.id;

SKIP LOCKED and READPAST are semantically equivalent in this context: they skip rows that are already locked by another session, so concurrent workers automatically partition the work without explicit coordination.

Retention and Archival

Processed outbox rows and CDC change tables both accumulate over time. A system producing 10,000 events per minute will accumulate 14 million rows per day in the outbox table. Without a retention policy, query performance degrades and storage costs rise.

The simplest approach is periodic deletion of processed rows older than a configured window — 24 to 72 hours is common, giving a replay window for debugging. Deletion should be done in small batches to avoid long-running locks:

PostgreSQL:

-- PostgreSQL: batched deletion of old processed outbox rows
DELETE FROM event_outbox
WHERE id IN (
    SELECT id FROM event_outbox
    WHERE processed_at < NOW() - INTERVAL '48 hours'
    LIMIT 5000
);

SQL Server:

-- SQL Server: batched deletion of old processed outbox rows
DELETE TOP (5000) FROM event_outbox
WHERE processed_at < DATEADD(HOUR, -48, SYSDATETIMEOFFSET());

For SQL Server CDC, the cleanup job retention is configured when enabling CDC:

SQL Server:

-- SQL Server: configure CDC cleanup retention (in minutes; 4320 = 72 hours)
EXEC sys.sp_cdc_change_job
    @job_type       = N'cleanup',
    @retention      = 4320,
    @threshold      = 5000;  -- max rows deleted per cleanup cycle

Monitoring

Production event-driven systems need observability across three dimensions: lag (how far behind are consumers?), error rate (how many events are landing in the dead letter queue?), and throughput (events per second, which reveals upstream problems when it drops unexpectedly).
