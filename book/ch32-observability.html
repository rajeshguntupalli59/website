Chapter 32: Observability Enhancements

Knowing that a database is slow is easy. Knowing *why* it is slow — and catching it before users notice — is the real work of a DBA. Observability is the discipline of making your database systems legible: turning raw metrics, logs, wait events, and execution data into actionable understanding. This chapter walks through the tools and techniques that modern PostgreSQL and SQL Server offer for deep observability, from the foundational statistics views that ship with each engine to the extended monitoring pipelines that production teams build on top of them. By the end, you will have a clear picture of how to instrument your databases so that nothing important hides in the dark.

---

What Observability Actually Means for a Database

The word *observability* comes from control theory, but in the database world it has a practical meaning: can you look at what the system is doing right now, reconstruct what it did in the past, and understand why it behaved the way it did? Monitoring asks "is the system up?" Observability asks "what is the system doing, and why?"

A fully observable database exposes three pillars:

Metrics are numeric measurements sampled over time — queries per second, buffer hit ratio, lock wait count, replication lag. They answer "how much?" and "how often?"

Logs capture discrete events — slow queries, errors, autovacuum runs, checkpoint completions. They answer "what happened?"

Traces (or query-level telemetry) capture the execution of individual requests — how long each step took, how many rows were processed, which waits occurred. They answer "why did this specific thing take so long?"

Most DBA teams start with metrics, add logs, and only instrument traces when they run into problems they cannot explain with the first two. Building all three in from the beginning is far cheaper than retrofitting them later.

Both PostgreSQL and SQL Server have matured significantly in this space. PostgreSQL ships with the pg_stat_* family of catalog views, the pg_stat_statements extension, and the auto_explain module. SQL Server provides Dynamic Management Views (DMVs), Extended Events, Query Store, and Wait Statistics. The challenge is not finding data — it is knowing which data to look at, at what cadence, and how to correlate it when something goes wrong.

The Statistics Infrastructure: pg_stat_* and SQL Server DMVs

Every query your database executes leaves a fingerprint somewhere in the system's internal counters. Learning to read those counters is the first real skill in database observability.

**PostgreSQL's pg_stat_* Views**

PostgreSQL maintains a family of cumulative statistics views that the statistics collector process updates asynchronously. The most useful ones for day-to-day work are:

  • → `pg_stat_activity` — currently running sessions and their states
  • → `pg_stat_statements` — aggregated execution stats per normalized query
  • → `pg_stat_bgwriter` — checkpoint and background writer activity
  • → `pg_stat_user_tables` — heap fetches, index scans, live/dead tuples per table
  • → `pg_stat_user_indexes` — scans and tuples read per index
  • → `pg_statio_user_tables` — buffer hits vs disk reads at the table level

    • The pg_stat_activity view is where you start whenever something is running slow right now:

    PostgreSQL:

    -- Find long-running queries, their state, and any blocking PIDs
SELECT
    pid,
    usename,
    application_name,
    state,
    wait_event_type,
    wait_event,
    now() - query_start AS duration,
    left(query, 120) AS query_snippet
FROM pg_stat_activity
WHERE state <> 'idle'
  AND query_start < now() - INTERVAL '30 seconds'
ORDER BY duration DESC;

    SQL Server:

    -- Find long-running requests with wait info and blocking
SELECT
    r.session_id,
    s.login_name,
    s.program_name,
    r.status,
    r.wait_type,
    r.wait_time / 1000.0 AS wait_seconds,
    r.blocking_session_id,
    DATEDIFF(SECOND, r.start_time, GETDATE()) AS duration_seconds,
    LEFT(t.text, 120) AS query_snippet
FROM sys.dm_exec_requests r
JOIN sys.dm_exec_sessions s ON r.session_id = s.session_id
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE r.session_id <> @@SPID
  AND DATEDIFF(SECOND, r.start_time, GETDATE()) > 30
ORDER BY duration_seconds DESC;

    SQL Server's DMV Ecosystem

    SQL Server's Dynamic Management Views and Functions (DMVs/DMFs) are the rough equivalent of PostgreSQL's pg_stat_* views, but they are much more granular and cover areas PostgreSQL does not expose directly at the view level — things like index usage statistics, missing index hints, and memory grant information.

    Key DMVs every SQL Server DBA should have committed to memory:

  • → `sys.dm_exec_requests` — currently executing requests
  • → `sys.dm_exec_query_stats` — aggregated plan-level performance over time
  • → `sys.dm_os_wait_stats` — cumulative wait statistics since last restart
  • → `sys.dm_db_index_usage_stats` — seeks, scans, lookups per index
  • → `sys.dm_db_missing_index_details` — indexes the optimizer wished existed
  • → `sys.dm_os_ring_buffers` — memory pressure and scheduler events

    • One important behavioral difference: SQL Server's DMVs reset on service restart, and sys.dm_os_wait_stats resets when you run DBCC SQLPERF('sys.dm_os_wait_stats', CLEAR). PostgreSQL's statistics reset with pg_stat_reset() or when the cluster restarts. If you are trending these metrics over time, you need to snapshot them externally — your monitoring agent should be pulling these values and writing them to a time-series store, not relying on the database's own reset cadence.

    Query-Level Observability: pg_stat_statements and Query Store

    Knowing that a query is slow in the moment is one thing. Knowing that a query has been getting progressively slower over the past two weeks, and that it spiked at 3:00 AM on Tuesdays, is something else entirely. That kind of longitudinal query intelligence requires a persistent query performance store.

    pg_stat_statements in PostgreSQL

    pg_stat_statements is a contrib extension that normalizes every query (replacing literal values with placeholders) and accumulates execution statistics per unique query shape. It must be loaded via shared_preload_libraries and created with CREATE EXTENSION pg_stat_statements.

    The view exposes mean execution time, total time, rows returned, buffer hits, shared block reads, and — since PostgreSQL 14 — planning time separate from execution time. This separation matters because a query with a stable plan will have near-zero planning overhead, but a query whose statistics are stale will spend significant time replanning.

    PostgreSQL:

    -- Top 10 queries by total execution time with buffer and planning detail
SELECT
    queryid,
    calls,
    round(total_exec_time::numeric, 2)   AS total_exec_ms,
    round(mean_exec_time::numeric, 2)    AS mean_exec_ms,
    round(total_plan_time::numeric, 2)   AS total_plan_ms,
    round(stddev_exec_time::numeric, 2)  AS stddev_exec_ms,
    shared_blks_hit,
    shared_blks_read,
    rows,
    left(query, 100) AS query_snippet
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;

    The stddev_exec_time column is often overlooked but tells you something important: a query with a high standard deviation is not just slow, it is *inconsistently* slow. That pattern usually points to parameter sniffing issues, plan instability, or lock contention that only hits some executions.

    SQL Server's Query Store

    Query Store (introduced in SQL Server 2016) is the most important observability improvement Microsoft has ever made to the engine. It persists query plans and runtime statistics across restarts, which sys.dm_exec_query_stats cannot do. You enable it per database:

    ALTER DATABASE YourDatabase
SET QUERY_STORE = ON (
    OPERATION_MODE = READ_WRITE,
    CLEANUP_POLICY = (STALE_QUERY_THRESHOLD_DAYS = 30),
    DATA_FLUSH_INTERVAL_SECONDS = 900,
    INTERVAL_LENGTH_MINUTES = 60,
    MAX_STORAGE_SIZE_MB = 1000,
    QUERY_CAPTURE_MODE = AUTO
);

    QUERY_CAPTURE_MODE = AUTO is the production-safe setting — it avoids capturing trivial one-off queries that would bloat the store. ALL is useful for debugging environments where you want complete coverage.

    SQL Server:

    -- Top 10 queries by average CPU time from Query Store
SELECT TOP 10
    qt.query_sql_text,
    q.query_id,
    rs.count_executions,
    ROUND(rs.avg_cpu_time / 1000.0, 2)      AS avg_cpu_ms,
    ROUND(rs.avg_duration / 1000.0, 2)      AS avg_duration_ms,
    ROUND(rs.avg_logical_io_reads, 0)       AS avg_logical_reads,
    rs.avg_rowcount,
    p.plan_id
FROM sys.query_store_query_text qt
JOIN sys.query_store_query q       ON qt.query_text_id = q.query_text_id
JOIN sys.query_store_plan p        ON q.query_id = p.query_id
JOIN sys.query_store_runtime_stats rs ON p.plan_id = rs.plan_id
ORDER BY rs.avg_cpu_time DESC;

    One of Query Store's most powerful features is forced plan management — if a plan regression is identified, you can force the optimizer to use a known-good plan with sp_query_store_force_plan, which persists across restarts. This is a production lifesaver when an optimizer update or statistics refresh causes a sudden plan change at 2:00 AM.

    Wait Event Analysis: Finding the Real Bottleneck

    Every database request spends its time either working (on CPU) or waiting (for I/O, locks, memory, network, or latency). Wait event analysis is the technique of profiling *where* time goes, rather than just *how much* time is taken. It shifts the diagnostic question from "the query is slow" to "the query is spending 78% of its time waiting on a lock held by this other session."

    PostgreSQL Wait Events

    PostgreSQL exposes waits through pg_stat_activity.wait_event and wait_event_type. The categories are: Lock, LWLock, IO, BufferPin, Activity, Client, Extension, IPC, and Timeout. The most operationally significant ones are:

  • → **Lock**: Row-level or table-level lock contention — investigate `pg_locks` immediately
  • → **LWLock:BufferMapping**: Heavy concurrent access to shared buffers; usually a sign of insufficient `shared_buffers` or extremely hot pages
  • → **IO:DataFileRead**: Buffer cache miss, reading from disk
  • → **Client:ClientRead**: Waiting for the application to send the next query — often long idle transactions

    • Because pg_stat_activity is a snapshot, you cannot rely on a single query to characterize wait patterns. You need to sample it repeatedly and aggregate. A simple approach is to run a sampling script every second for 60 seconds and group the results:

    PostgreSQL:

    -- Create a snapshot table, then aggregate waits (run in a loop or via cron)
-- Single snapshot for illustration:
SELECT
    wait_event_type,
    wait_event,
    state,
    COUNT(*) AS session_count
FROM pg_stat_activity
WHERE pid <> pg_backend_pid()
  AND state <> 'idle'
GROUP BY wait_event_type, wait_event, state
ORDER BY session_count DESC;

    Tools like pgBadger, pganalyze, and Prometheus + postgres_exporter do this sampling for you continuously, building the kind of wait profile that is impossible to construct with a single query.

    SQL Server Wait Statistics

    SQL Server's sys.dm_os_wait_stats accumulates total wait time per wait type since the last reset. The classic analysis approach is to take a delta snapshot over a meaningful interval (say, 60 seconds during peak load) and calculate the percentage of total wait time each type represents.

    SQL Server:

    -- Snapshot waits, wait 60 seconds, take second snapshot, calculate delta
-- Simplified single-point view: top waits by total wait time
SELECT TOP 20
    wait_type,
    waiting_tasks_count,
    wait_time_ms,
    max_wait_time_ms,
    signal_wait_time_ms,
    wait_time_ms - signal_wait_time_ms AS resource_wait_time_ms,
    ROUND(100.0 * wait_time_ms 
          / NULLIF(SUM(wait_time_ms) OVER (), 0), 2) AS pct_of_total
FROM sys.dm_os_wait_stats
WHERE wait_type NOT IN (
    -- Benign background waits to exclude
    'SLEEP_TASK', 'BROKER_TO_FLUSH', 'BROKER_TASK_STOP',
    'CLR_AUTO_EVENT', 'DISPATCHER_QUEUE_SEMAPHORE',
    'FT_IFTS_SCHEDULER_IDLE_WAIT', 'HADR_WORK_QUEUE',
    'LAZYWRITER_SLEEP', 'LOGMGR_QUEUE', 'ONDEMAND_TASK_QUEUE',
    'REQUEST_FOR_DEADLOCK_SEARCH', 'RESOURCE_QUEUE',
    'SERVER_IDLE_CHECK', 'SLEEP_DBSTARTUP', 'SLEEP_DBRECOVER',
    'SLEEP_MASTERDBREADY', 'SLEEP_MASTERMDREADY',
    'SLEEP_MASTERUPGRADED', 'SLEEP_MSDBSTARTUP',
    'SLEEP_SYSTEMTASK', 'SLEEP_TEMPDBSTARTUP',
    'SNI_HTTP_ACCEPT', 'SP_SERVER_DIAGNOSTICS_SLEEP',
    'SQLTRACE_BUFFER_FLUSH', 'WAITFOR', 'XE_DISPATCHER_WAIT',
    'XE_TIMER_EVENT', 'REDO_THREAD_PENDING_WORK',
    'HADR_FILESTREAM_IOMGR_IOCOMPLETION'
)
ORDER BY wait_time_ms DESC;

    The benign wait exclusion list is important — without it, background SQL Server housekeeping waits dominate the output and obscure real contention. The standard reference list maintained by Paul Randal at SQLskills is a good starting point and should be part of every SQL Server DBA's toolkit.

    The signal_wait_time_ms column tells you how long waits were in the runnable queue before actually getting CPU time. If signal waits are a significant fraction of total wait time (more than 15–20%), you have CPU pressure — add cores or reduce CPU-intensive workloads before tuning anything else.

    Extended Events and Auto-Explain: Deep Diagnostic Tracing

    Statistics views tell you the aggregate picture. When you need to understand what happened in a specific transaction, or capture a class of events that occurs rarely but causes severe impact, you need event tracing infrastructure.

    PostgreSQL auto_explain

    auto_explain is a contrib module that logs the execution plan of any query exceeding a configurable duration threshold. It must be loaded via shared_preload_libraries. The key parameters:

    auto_explain.log_min_duration = '1s'
auto_explain.log_analyze = on
auto_explain.log_buffers = on
auto_explain.log_timing = on
auto_explain.log_format = 'json'
auto_explain.log_nested_statements = on

    log_analyze = on runs EXPLAIN ANALYZE on every captured query, which adds execution overhead. In high-traffic systems, set log_min_duration high enough (5–10 seconds) to avoid capturing too many plans, or use auto_explain.sample_rate (available since PostgreSQL 12) to capture a fraction of all queries regardless of duration — useful for identifying plans that are individually fast but have a bad shape at scale.

    The JSON format output can be parsed by external tools like explain.depesz.com or pev2 for visual analysis. The key things to look for in a captured plan: sequential scans on large tables, hash joins with high memory usage that spill to disk, and nested loop joins with high row estimates that turn out to be wrong.

    SQL Server Extended Events

    Extended Events (XEvents) replaced SQL Trace and Profiler as SQL Server's primary event capture system starting with SQL Server 2008, and became significantly more capable in 2012. It is a lightweight, asynchronous event infrastructure that lets you capture almost any engine event — query compilations, lock acquisitions, deadlocks, waits, I/O stalls, memory grants — with minimal overhead when configured correctly.

    A production-safe session for capturing slow queries looks like this:

    CREATE EVENT SESSION [SlowQueryCapture] ON SERVER
ADD EVENT sqlserver.sql_statement_completed (
    ACTION (
        sqlserver.sql_text,
        sqlserver.plan_handle,
        sqlserver.database_name,
        sqlserver.username,
        sqlserver.client_app_name
    )
    WHERE (
        [duration] > 5000000  -- microseconds: 5 seconds
        AND [sqlserver].[database_name] = N'YourDatabase'
    )
),
ADD EVENT sqlserver.rpc_completed (
    ACTION (
        sqlserver.sql_text,
        sqlserver.plan_handle,
        sqlserver.database_name
    )
    WHERE ([duration] > 5000000)
)
ADD TARGET package0.ring_buffer (SET MAX_MEMORY = 51200)
WITH (
    MAX_DISPATCH_LATENCY = 30 SECONDS,
    TRACK_CAUSALITY = OFF,
    STARTUP_STATE = ON
);

ALTER EVENT SESSION [SlowQueryCapture] ON SERVER STATE = START;

    The ring_buffer target stores events in memory — cheap and fast, but events are lost on restart. For persistent capture, use package0.asynchronous_file_target pointing to a dedicated drive. Never write XEvents files to the same volume as your data or log files.

    Deadlock Detection

    Both databases generate deadlock information through their event systems. In PostgreSQL, deadlocks are logged to postgresql.log with log_lock_waits = on and deadlock_timeout = 1s. The log entry includes the PIDs, the queries involved, and the lock types. For more structured capture, pg_stat_activity joined with pg_locks gives you a live lock graph.

    In SQL Server, the system_health XEvents session (which runs by default since SQL Server 2008 R2) captures deadlock graphs as XML. You can query them directly:

    SELECT
    xdr.value('@timestamp', 'datetime2') AS deadlock_time,
    xdr.query('.') AS deadlock_graph_xml
FROM (
    SELECT CAST(target_data AS XML) AS target_data
    FROM sys.dm_xe_session_targets t
    JOIN sys.dm_xe_sessions s ON t.event_session_address = s.address
    WHERE s.name = 'system_health'
      AND t.target_name = 'ring_buffer'
) AS data
CROSS APPLY target_data.nodes('//RingBufferTarget/event[@name="xml_deadlock_report"]') AS XEventData(xdr)
ORDER BY deadlock_time DESC;

    This is often the first place to go when users report intermittent errors about transactions being chosen as deadlock victims, because the XML graph shows you exactly which resources were contested and in what order the locks were acquired.

    Building a Production Observability Pipeline

    Having the right queries and tools is necessary but not sufficient. In a production environment, you need continuous collection, retention, and alerting — not manual queries run reactively after something breaks.

    What to Collect and How Often

    The collection cadence depends on the volatility of the metric. Lock waits and active session state should be sampled every 10–30 seconds; table and index statistics every few minutes; checkpoint and bgwriter counters every minute. Anything more frequent than 10 seconds for most metrics will add non-trivial overhead to PostgreSQL's statistics collector and SQL Server's DMV queries.

    For PostgreSQL, postgres_exporter is the standard Prometheus integration point. It exposes pg_stat_statements, pg_stat_activity, pg_stat_bgwriter, pg_stat_replication, and many other views as Prometheus metrics, which you then scrape into Grafana for dashboards. The pg_activity and pgBadger tools complement this for real-time session monitoring and log analysis respectively.

    For SQL Server, sql_exporter (or the Telegraf SQL Server input plugin) scrapes DMVs on a configurable interval and feeds into Prometheus or InfluxDB. SQL Server also integrates with Azure Monitor, SQL Server Management Studio's Activity Monitor, and third-party tools like SolarWinds DPA, SentryOne (now SolarWinds), and Redgate SQL Monitor — all of which are built on the same DMV infrastructure you query manually.

    Alerting Thresholds That Matter

    The difference between a good monitoring setup and a great one is the quality of the alerting thresholds. Alerting on CPU > 80% will wake you up constantly for no reason on a busy OLTP system. Better alerts are:

  • → Buffer hit ratio drops below 95% (PostgreSQL: `pg_statio_user_tables`) — sustained drops indicate working set exceeds shared buffers
  • → Replication lag exceeds 30 seconds on a standby (PostgreSQL: `pg_stat_replication.replay_lag`; SQL Server: `sys.dm_hadr_database_replica_states`)
  • → Long-running transactions exceeding a threshold (older than 5 minutes in OLTP, older than 30 minutes in batch systems)
  • → Deadlock rate exceeding baseline by more than 2x
  • → Checkpoint completion ratio falling below 0.9 in PostgreSQL (indicating `checkpoint_completion_target` tuning may be needed or I/O is saturated)
  • → Query Store plan count growing rapidly in SQL Server (often indicates parameter sniffing causing plan proliferation)

    • Correlating Across Pillars

    The real payoff of a mature observability pipeline comes when you can correlate metrics, logs, and traces together. An alert fires at 14:22: average query duration spiked from 80ms to 4 seconds. You go to your Grafana dashboard and see that buffer reads spiked at the same moment. You check pg_stat_bgwriter and see that checkpoints started occurring more frequently at 14:20. You check the logs and find that a large bulk insert job started at 14:19, generating enough dirty pages to trigger more aggressive checkpointing, which created I/O pressure that slowed everyone else down.

    None of those individual data points would have told you the story alone. Together, they give you a root cause in under five minutes instead of an hour of guesswork.

    A final note on instrumentation philosophy: instrument first, analyze later. The cost of not having observability data when an incident occurs is almost always higher than the marginal cost of collecting it. Storage is cheap. Engineer time spent reconstructing what happened without data is expensive. Build the pipeline when the system is healthy, so it is already working when things go wrong.

    Key Takeaways

  • → **Observability has three pillars** — metrics, logs, and traces — and all three are needed to fully understand database behavior. Most problems are only fully explainable when data from all three pillars is correlated together.
  • → **PostgreSQL's `pg_stat_statements` and SQL Server's Query Store** are the most important query-level observability tools in each engine, providing normalized, aggregated, and persistent execution statistics that point directly to the queries consuming the most resources.
  • → **Wait event analysis** is the most reliable way to identify the *type* of bottleneck a database is experiencing. CPU pressure, lock contention, I/O saturation, and memory pressure each produce a distinctive wait signature, and no other technique pinpoints them as directly.
