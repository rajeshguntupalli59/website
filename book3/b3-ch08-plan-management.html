Chapter 8: Autonomous Execution Plan Management

Query execution plans are the contracts between your application's intent and the database engine's physical execution strategy. When those contracts degrade — through statistics drift, parameter sensitivity, schema evolution, or optimizer regressions introduced by version upgrades — performance collapses in ways that are rarely caught by traditional monitoring until users are already impacted. An autonomous execution plan management system detects plan instability before it manifests as user-visible latency, evaluates candidate plans against production workloads, promotes or pins the best-performing plan, and rolls back regressions — all without a DBA touching a keyboard. This chapter constructs that system in full, covering the detection infrastructure, the decision engine, the execution and feedback loop, and the production patterns that make it safe to run unsupervised in regulated, high-traffic environments.

---

8.1 Anatomy of Plan Instability: What the Autonomous System Must Observe

Before automation can act, it must observe the right signals. Plan instability does not always announce itself as a slow query. More often it presents as variance — the same query running in 8ms one minute and 800ms the next. The autonomous system must distinguish between three categories of instability that require different responses.

Regression is the simplest case: a plan that was stable is replaced by a worse plan. This happens when statistics are stale, when a new index is created and the optimizer picks it incorrectly, or when a minor schema change shifts cardinality estimates past a threshold that flips a join strategy from nested loop to hash join with the wrong side as the build input.

Parameter sensitivity (also called parameter sniffing in SQL Server, or the analogous plan cache pollution in PostgreSQL's prepared statement protocol) describes a plan compiled for one parameter value that performs poorly for a different distribution. A stored procedure compiled on a date range of one day serves a query asking for three months of data with the same nested loop plan — and collapses.

Optimizer regression occurs during version upgrades. PostgreSQL 15 to 16 may change the cost model for parallel hash join. SQL Server CU updates regularly adjust cardinality estimator behavior. A plan that was correct under the prior optimizer model becomes wrong under the new one without any data or schema change.

The observability infrastructure must capture plan identifiers, execution statistics, and plan text at a granularity that makes these three categories distinguishable.

PostgreSQL:

-- Capture plan hash, mean execution time, and stddev for variance detection
-- Requires pg_stat_statements with track_planning = on
SELECT
    queryid,
    query,
    calls,
    mean_exec_time,
    stddev_exec_time,
    stddev_exec_time / NULLIF(mean_exec_time, 0) AS coefficient_of_variation,
    min_exec_time,
    max_exec_time,
    total_exec_time,
    rows / NULLIF(calls, 0) AS avg_rows
FROM pg_stat_statements
WHERE calls > 50
  AND mean_exec_time > 10  -- ms, filter out trivial queries
ORDER BY coefficient_of_variation DESC NULLS LAST
LIMIT 50;

SQL Server:

-- Identify plan instability via execution count variance and plan count per query hash
SELECT
    qs.query_hash,
    COUNT(DISTINCT qs.plan_handle) AS plan_variants,
    SUM(qs.execution_count)        AS total_executions,
    AVG(qs.total_elapsed_time / NULLIF(qs.execution_count, 0)) AS avg_elapsed_us,
    MAX(qs.total_elapsed_time / NULLIF(qs.execution_count, 0)) AS max_elapsed_us,
    MIN(qs.total_elapsed_time / NULLIF(qs.execution_count, 0)) AS min_elapsed_us,
    -- Coefficient of variation proxy using max/avg ratio
    MAX(qs.total_elapsed_time / NULLIF(qs.execution_count, 0)) /
        NULLIF(AVG(qs.total_elapsed_time / NULLIF(qs.execution_count, 0)), 0) AS volatility_ratio
FROM sys.dm_exec_query_stats qs
GROUP BY qs.query_hash
HAVING COUNT(DISTINCT qs.plan_handle) > 1
    OR MAX(qs.total_elapsed_time / NULLIF(qs.execution_count, 0)) /
       NULLIF(AVG(qs.total_elapsed_time / NULLIF(qs.execution_count, 0)), 0) > 3.0
ORDER BY avg_elapsed_us DESC;

The coefficient of variation (stddev/mean) and the volatility ratio (max/avg elapsed) are your primary instability signals. A CV above 1.5 or a volatility ratio above 3.0 flags a query as a candidate for automated plan investigation. These thresholds are calibrated starting points — your production system will tune them over time based on false-positive rates in your workload.

For PostgreSQL, pg_stat_statements gives you aggregate statistics but not plan-level differentiation. To get plan-level tracking, you need auto_explain for sampling and a custom collection table, or the commercial pg_store_plans extension. For SQL Server, the Query Store provides the full plan-per-query-hash history that makes plan-level analysis native.

The autonomous collector runs this detection query on a configurable interval — typically every five minutes for OLTP systems — and writes results to a plan instability log table. This table is the input queue for the decision engine described in section 8.3.

8.2 Plan Baseline Architecture: Building the Ground Truth Store

An autonomous plan management system needs a reference point — a "known good" state to measure against and to roll back to. This is the plan baseline. The architecture of your baseline store determines what the automation can do: a shallow baseline supports only regression detection, while a deep baseline with execution context supports full plan pinning, forced plan rotation, and A/B evaluation.

PostgreSQL baseline store using pg_hint_plan and a custom schema:

The PostgreSQL optimizer does not have a native plan forcing mechanism equivalent to SQL Server's Query Store forced plans. The production pattern uses pg_hint_plan, which intercepts queries and injects optimizer hints before planning. The baseline store must therefore persist both the plan description and the hint string required to reproduce it.

-- Schema for the PostgreSQL autonomous plan baseline
CREATE SCHEMA IF NOT EXISTS ade_plan_mgmt;

CREATE TABLE ade_plan_mgmt.plan_baselines (
    baseline_id         BIGSERIAL PRIMARY KEY,
    queryid             BIGINT NOT NULL,           -- from pg_stat_statements
    query_fingerprint   TEXT NOT NULL,             -- normalized query text
    hint_string         TEXT NOT NULL,             -- pg_hint_plan compatible hint
    plan_json           JSONB NOT NULL,            -- EXPLAIN (FORMAT JSON) output
    captured_at         TIMESTAMPTZ DEFAULT now(),
    last_validated_at   TIMESTAMPTZ,
    mean_exec_time_ms   NUMERIC(12,3),
    validation_count    INTEGER DEFAULT 0,
    status              TEXT DEFAULT 'candidate'   -- candidate | accepted | rejected | pinned
        CHECK (status IN ('candidate','accepted','rejected','pinned')),
    promoted_by         TEXT,                      -- 'autonomous' | 'manual'
    notes               TEXT
);

CREATE INDEX ON ade_plan_mgmt.plan_baselines (queryid, status);
CREATE INDEX ON ade_plan_mgmt.plan_baselines (captured_at);

-- Execution history for trend analysis
CREATE TABLE ade_plan_mgmt.plan_execution_log (
    log_id          BIGSERIAL PRIMARY KEY,
    baseline_id     BIGINT REFERENCES ade_plan_mgmt.plan_baselines(baseline_id),
    observed_at     TIMESTAMPTZ DEFAULT now(),
    exec_time_ms    NUMERIC(12,3),
    rows_returned   BIGINT,
    buffers_hit     BIGINT,
    buffers_read    BIGINT
);

SQL Server baseline using Query Store forced plans:

SQL Server's Query Store provides a native plan forcing mechanism. The autonomous system's baseline store is a thin management layer on top of Query Store — it tracks which plans were forced, when, why, and what the performance outcome was.

-- Autonomous plan management overlay for Query Store
CREATE TABLE dbo.ade_plan_baselines (
    baseline_id         INT IDENTITY(1,1) PRIMARY KEY,
    query_id            BIGINT NOT NULL,    -- sys.query_store_query.query_id
    plan_id             BIGINT NOT NULL,    -- sys.query_store_plan.plan_id
    query_hash          BINARY(8),
    query_sql_text      NVARCHAR(MAX),
    avg_duration_us     BIGINT,             -- when baseline was captured
    captured_at         DATETIME2 DEFAULT SYSUTCDATETIME(),
    last_validated_at   DATETIME2,
    force_status        NVARCHAR(20) DEFAULT 'candidate'
        CHECK (force_status IN ('candidate','forced','rejected','retired')),
    forced_by           NVARCHAR(50),       -- 'autonomous' | 'manual'
    regression_threshold_pct INT DEFAULT 20,
    notes               NVARCHAR(1000)
);

CREATE INDEX IX_ade_plan_baselines_queryid
    ON dbo.ade_plan_baselines (query_id, force_status);

The key architectural decision is the promotion pipeline: candidate → accepted → pinned. A plan enters as a candidate when the autonomous system captures it during a clean performance window. It advances to accepted after surviving validation runs (section 8.3). It becomes pinned when the system actively forces it in response to a detected regression. This state machine prevents the automation from forcing plans that were only observed once under favorable conditions.

For SQL Server, the native Query Store forced plan mechanism is invoked with:

EXEC sys.sp_query_store_force_plan @query_id = ?, @plan_id = ?;

For PostgreSQL, the automation writes the hint string into the pg_hint_plan.hint_plan table at plan promotion time:

INSERT INTO hint_plan.hints (norm_query_string, application_name, hints)
VALUES (?, '', ?)
ON CONFLICT (norm_query_string, application_name)
DO UPDATE SET hints = EXCLUDED.hints;

The baseline store also integrates with your version control system. Each promotion event generates a migration record — the hint string or forced plan identifier — that can be tracked in Git alongside schema changes. This audit trail is non-negotiable in regulated environments.

8.3 The Autonomous Decision Engine: From Signal to Action

The decision engine is the intelligence layer. It consumes instability signals from section 8.1, evaluates candidate plans from the baseline store, and makes promotion or rollback decisions according to a policy model. The policy model must be explicit and auditable — "when X, do Y" — because autonomous systems that act on implicit logic cannot be reviewed, tested, or trusted in production.

The decision policy for autonomous plan management has four rules, evaluated in priority order:

1. Regression rule: If mean execution time for a query has increased by more than regression_threshold_pct (default 20%) compared to the accepted baseline, and more than min_regression_samples (default 100 executions) have been observed in the degraded state, evaluate whether a pinned plan exists. If yes, force the pinned plan immediately. If no, escalate to human review.

2. Instability rule: If the coefficient of variation exceeds the configured threshold for a query with no pinned plan, capture the current plan as a candidate baseline during the next clean performance window (CV < 0.3 and mean time within 10% of 30-day low).

3. Validation rule: Before promoting a candidate to accepted, the engine generates a EXPLAIN ANALYZE (PostgreSQL) or SET STATISTICS TIME ON (SQL Server) against a read replica with production-representative data and records the measured execution time.

4. Escalation rule: Any action that would affect a query running more than 500 times per second, or any query touching more than 10 tables, requires human confirmation before proceeding. The system opens a Slack/PagerDuty alert and waits up to 4 hours before timing out and escalating to critical.

# Autonomous plan decision engine — core evaluation loop
# Runs as a scheduled task every 5 minutes (Kubernetes CronJob or systemd timer)

import psycopg2
import pyodbc
import json
import logging
from datetime import datetime, timedelta
from dataclasses import dataclass
from enum import Enum
from typing import Optional
import requests  # for Slack alerting

logging.basicConfig(level=logging.INFO,
                    format='%(asctime)s %(levelname)s %(message)s')
logger = logging.getLogger('ade.plan_mgmt')


class PlanAction(Enum):
    FORCE_EXISTING_BASELINE = "force_existing_baseline"
    CAPTURE_NEW_CANDIDATE   = "capture_new_candidate"
    ESCALATE_HUMAN          = "escalate_human"
    MONITOR_ONLY            = "monitor_only"


@dataclass
class QuerySignal:
    queryid: int
    query_hash: str
    mean_exec_time_ms: float
    coefficient_of_variation: float
    calls_per_minute: float
    table_count: int
    baseline_mean_ms: Optional[float]
    pinned_plan_id: Optional[int]
    regression_threshold_pct: int = 20


def evaluate_plan_action(signal: QuerySignal) -> tuple[PlanAction, str]:
    """
    Applies the four-rule policy model in priority order.
    Returns (action, rationale) for audit logging.
    """
    # Safety gate: high-frequency or complex queries require human confirmation
    if signal.calls_per_minute > 500 or signal.table_count > 10:
        return (
            PlanAction.ESCALATE_HUMAN,
            f"Safety gate: {signal.calls_per_minute:.0f} calls/min or "
            f"{signal.table_count} tables exceeds autonomous threshold"
        )

    # Rule 1: Regression with existing baseline
    if signal.baseline_mean_ms is not None:
        regression_pct = (
            (signal.mean_exec_time_ms - signal.baseline_mean_ms)
            / signal.baseline_mean_ms * 100
        )
        if regression_pct > signal.regression_threshold_pct:
            if signal.pinned_plan_id is not None:
                return (
                    PlanAction.FORCE_EXISTING_BASELINE,
                    f"Regression detected: {regression_pct:.1f}% above baseline "
                    f"({signal.mean_exec_time_ms:.1f}ms vs "
                    f"{signal.baseline_mean_ms:.1f}ms baseline). "
                    f"Forcing plan_id={signal.pinned_plan_id}"
                )
            else:
                return (
                    PlanAction.ESCALATE_HUMAN,
                    f"Regression of {regression_pct:.1f}% detected but no pinned "
                    f"baseline available. Human review required."
                )

    # Rule 2: Instability — capture candidate if in clean window
    if signal.coefficient_of_variation > 1.5:
        if signal.coefficient_of_variation < 0.3:  # currently stable — clean window
            return (
                PlanAction.CAPTURE_NEW_CANDIDATE,
                f"Instability pattern detected (CV={signal.coefficient_of_variation:.2f}). "
                f"Current window is stable — capturing candidate baseline."
            )
        return (
            PlanAction.MONITOR_ONLY,
            f"Instability detected (CV={signal.coefficient_of_variation:.2f}) "
            f"but no clean window available. Continuing observation."
        )

    return (PlanAction.MONITOR_ONLY, "No action criteria met.")


def dispatch_action(action: PlanAction, signal: QuerySignal,
                    rationale: str, db_conn, slack_webhook: str):
    """Execute the decided action and log the outcome."""
    logger.info(f"QueryID {signal.queryid}: {action.value} — {rationale}")

    if action == PlanAction.FORCE_EXISTING_BASELINE:
        # For SQL Server: sp_query_store_force_plan
        # For PostgreSQL: update hint_plan.hints
        force_plan(signal.queryid, signal.pinned_plan_id, db_conn)
        log_action_audit(signal.queryid, action, rationale, db_conn)

    elif action == PlanAction.CAPTURE_NEW_CANDIDATE:
        capture_candidate_plan(signal.queryid, db_conn)
        log_action_audit(signal.queryid, action, rationale, db_conn)

    elif action == PlanAction.ESCALATE_HUMAN:
        send_escalation_alert(signal, rationale, slack_webhook)
        log_action_audit(signal.queryid, action, rationale, db_conn)

    # MONITOR_ONLY: log only, no structural change
    else:
        log_action_audit(signal.queryid, action, rationale, db_conn)

The decision engine is intentionally stateless between runs — all context comes from the database tables. This means it can be restarted, replaced, or scaled horizontally without losing decision continuity. The audit log table records every evaluation, the signal values at decision time, the action taken, and the outcome measured on the next evaluation cycle. This retrospective data is what you use to tune thresholds.

8.4 Plan Validation Before Promotion: The Shadow Execution Framework

Autonomous systems that force execution plans without validating them first will cause outages. The validation framework is the safety net between the decision engine's recommendation and the actual plan forcing action. It runs candidate plans against production-representative data in a controlled environment before they advance to accepted status.

The shadow execution framework has three components: a read replica or isolated test database with current statistics, a workload replay mechanism, and a comparison evaluator.

Setting up the validation environment:

For PostgreSQL, the validation target is a streaming replica with hot_standby = on. The automation connects to the replica, runs EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) with the candidate hint applied, and records the actual execution metrics. This gives you real statistics and real data distributions without risk to production.

PostgreSQL:

-- Validation query: run on replica with pg_hint_plan hint injected
-- The automation sets pg_hint_plan.enable_hint = on and injects the hint
-- via the hint comment syntax before executing

SET pg_hint_plan.enable_hint = on;
SET pg_hint_plan.debug_print = on;

-- Example: force index scan with SeqScan disabled hint
/*+ SeqScan(orders) IndexScan(orders idx_orders_customer_date) */
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON, TIMING ON)
SELECT o.order_id, o.total_amount, c.email
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
WHERE o.created_at >= now() - INTERVAL '30 days'
  AND o.status = 'pending';

SQL Server:

-- Validation via USE PLAN hint with Query Store plan XML
-- Forces a specific plan from Query Store during testing
DECLARE @plan_xml NVARCHAR(MAX);

SELECT @plan_xml = CAST(qsp.query_plan AS NVARCHAR(MAX))
FROM sys.query_store_plan qsp
WHERE qsp.plan_id = ?;  -- candidate plan_id from baseline store

-- Execute with plan forcing on the validation workload
-- (run against a secondary read-only AG replica)
EXEC sp_executesql
    N'SELECT o.OrderID, o.TotalAmount, c.Email
      FROM dbo.Orders o
      JOIN dbo.Customers c ON o.CustomerID = c.CustomerID
      WHERE o.CreatedDate >= DATEADD(DAY, -30, GETUTCDATE())
        AND o.Status = ''pending''
      OPTION (USE PLAN @hint_plan)',
    N'@hint_plan NVARCHAR(MAX)',
    @hint_plan = @plan_xml;

The comparison evaluator applies statistical significance testing to the validation results. A candidate plan must demonstrate at minimum a 10% improvement over the current plan's mean execution time, with the improvement being statistically significant at p < 0.05 across at least 30 validation executions. A plan that is merely equivalent to the current plan is not promoted — it offers no benefit and introducing any plan change carries operational risk.

# Plan validation comparison with statistical significance testing
import numpy as np
from scipy import stats


def evaluate_candidate_plan(
    baseline_times_ms: list[float],
    candidate_times_ms: list[float],
    improvement_threshold_pct: float = 10.0,
    alpha: float = 0.05
) -> dict:
    """
    Compare baseline vs. candidate plan execution times.
    Returns promotion decision with full statistical context.
    """
    baseline = np.array(baseline_times_ms)
    candidate = np.array(candidate_times_ms)

    baseline_mean = np.mean(baseline)
    candidate_mean = np.mean(candidate)
    improvement_pct = (baseline_mean - candidate_mean) / baseline_mean * 100

    # Welch's t-test — does not assume equal variance
    t_stat, p_value = stats.ttest_ind(baseline, candidate, equal_var=False)

    # Mann-Whitney U test for non-normal distributions (common in DB latency)
    u_stat, u_p_value = stats.mannwhitneyu(
        baseline, candidate, alternative='greater'
    )

    meets_improvement_threshold = improvement_pct >= improvement_threshold_pct
    statistically_significant = p_value < alpha and u_p_value < alpha
    should_promote = meets_improvement_threshold and statistically_significant

    return {
        "decision":             "PROMOTE" if should_promote else "REJECT",
        "baseline_mean_ms":     round(baseline_mean, 3),
        "candidate_mean_ms":    round(candidate_mean, 3),
        "improvement_pct":      round(improvement_pct, 2),
        "t_test_p_value":       round(p_value, 6),
        "mannwhitney_p_value":  round(u_p_value, 6),
        "statistically_significant": statistically_significant,
        "meets_improvement_threshold": meets_improvement_threshold,
        "sample_size":          len(candidate_times_ms),
        "evaluated_at":         datetime.utcnow().isoformat()
    }

A plan that passes validation gets its status updated to accepted in the baseline store and its measured performance characteristics recorded. If a regression is subsequently detected, the system now has a validated plan it can force with confidence.

The shadow execution framework should also run continuously against accepted baselines — not just candidates — on a weekly cycle. Statistics drift can make yesterday's validated baseline wrong for today's data distribution. Baselines that fail re-validation are demoted to candidate status automatically, and a new capture cycle begins.
