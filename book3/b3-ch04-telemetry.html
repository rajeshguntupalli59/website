Chapter 4: Telemetry, Metrics, and Data Required for Autonomy

An autonomous database system is only as intelligent as the data it can see about itself. Before any self-healing loop executes, before any workload gets rerouted, before any index gets dropped or any connection pool gets resized — the system must first observe its own state with enough fidelity to make a sound decision. This chapter is about building that observational foundation: the telemetry pipelines, metric taxonomies, and contextual data structures that transform a passively monitored database into a fully aware autonomous engine. Getting this layer right is the prerequisite for everything else in this book. Getting it wrong means your autonomous actions will be based on incomplete or misleading signals — and autonomous actions taken on bad data are worse than no action at all.

---

4.1 The Telemetry Taxonomy: What Autonomous Systems Actually Need to See

Most monitoring stacks are built for humans — dashboards, alerts, on-call pages. Autonomous systems need something different: structured, timestamped, low-latency signal feeds that can be ingested by decision engines without human interpretation. Before touching a single query or Prometheus scrape config, the autonomous database architect needs a taxonomy of what categories of data actually drive autonomous decisions.

The four layers of database telemetry are:

1. Resource telemetry — CPU, memory, disk I/O, network throughput, and storage capacity at both the OS and database engine level. These are necessary but not sufficient. An autonomous system that reacts only to CPU spikes is a basic autoscaler, not an autonomous database.

2. Engine-internal metrics — buffer cache hit ratios, lock wait chains, transaction throughput, vacuum progress (PostgreSQL), checkpoint frequency, query plan cache hit rates (SQL Server), and replication lag. These metrics describe what the database engine itself is experiencing, independent of what the hardware is doing.

3. Workload telemetry — query-level execution data: individual query durations, I/O reads per query, wait events, plan changes over time, and parameter distributions. This is the layer most monitoring systems underinvest in, and it's where the highest-value autonomous decisions live.

4. State and topology telemetry — cluster membership, replica health, failover readiness, connection pool saturation, and schema version state. Autonomous systems cannot safely act without understanding the current topology — promoting a standby without knowing whether it's actually current is a path to split-brain.

Each layer feeds different decision classes. Resource telemetry drives scaling and capacity decisions. Engine-internal metrics drive tuning and maintenance scheduling. Workload telemetry drives query optimization and plan pinning. State telemetry drives failover and topology management. A production autonomous system maintains all four layers simultaneously, with different collection frequencies appropriate to each.

For PostgreSQL, the engine-internal layer lives primarily in pg_stat_* views, pg_catalog, and the WAL subsystem. For SQL Server, the equivalent surfaces are Dynamic Management Views (DMVs), the Query Store, and the Extended Events system. These are your primary observational surfaces — everything else is derived from them.

4.2 Instrumenting PostgreSQL and SQL Server for Machine-Readable Metric Collection

The distinction between a monitoring-friendly database and an autonomy-ready database comes down to whether your metric collection is designed for human dashboards or for programmatic consumption. Autonomous systems need metrics that are timestamped precisely, labeled with enough dimensional context to drive decisions, and collected at frequencies that match the dynamics of the phenomena they describe.

#### PostgreSQL: Core Metric Surfaces

The pg_stat_activity view is the primary source for real-time workload visibility. For autonomous systems, the critical fields are wait_event_type, wait_event, state, query_start, and query_id (PostgreSQL 14+). Querying this view every 5–10 seconds and persisting the results gives your decision engine a timeline of what the database was doing at any moment.

PostgreSQL:

-- Workload snapshot for autonomous ingestion
-- Collect every 5-10 seconds and store in a time-series table or emit to Prometheus
SELECT
    now()                           AS collected_at,
    pid,
    usename,
    application_name,
    client_addr,
    state,
    wait_event_type,
    wait_event,
    EXTRACT(EPOCH FROM (now() - query_start))::NUMERIC(10,3) AS query_age_seconds,
    query_id,
    LEFT(query, 200)                AS query_snippet,
    backend_type
FROM pg_stat_activity
WHERE backend_type = 'client backend'
  AND state != 'idle'
ORDER BY query_age_seconds DESC NULLS LAST;

SQL Server:

-- Equivalent workload snapshot using DMVs
-- Designed for programmatic collection; include in a scheduled job or monitoring agent
SELECT
    GETUTCDATE()                                              AS collected_at,
    r.session_id,
    s.login_name,
    s.program_name,
    s.host_name,
    r.status,
    r.wait_type,
    r.wait_time / 1000.0                                     AS wait_time_seconds,
    r.total_elapsed_time / 1000.0                            AS elapsed_seconds,
    r.logical_reads,
    r.writes,
    r.cpu_time / 1000.0                                      AS cpu_seconds,
    qt.text                                                  AS query_text,
    qp.query_plan
FROM sys.dm_exec_requests r
JOIN sys.dm_exec_sessions s ON r.session_id = s.session_id
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle)  qt
CROSS APPLY sys.dm_exec_query_plan(r.plan_handle) qp
WHERE r.session_id != @@SPID
  AND s.is_user_process = 1
ORDER BY r.total_elapsed_time DESC;

These snapshots are the raw feed. They become useful to an autonomous system only when they're persisted with timestamps and queryable across time. A TimescaleDB hypertable on PostgreSQL, or a columnar table in SQL Server, is appropriate for storing these at scale. The autonomous decision engine then queries the history, not just the present-moment snapshot.

#### Structured Metric Emission with pg_stat_statements and Query Store

For workload telemetry at the query level, pg_stat_statements (PostgreSQL) and the Query Store (SQL Server) are the authoritative sources. These surfaces aggregate execution statistics per normalized query, giving you mean execution time, standard deviation, total I/O, and call count — exactly the inputs needed to identify regression candidates for autonomous tuning.

PostgreSQL:

-- Query-level performance telemetry for regression detection
-- Run this on a schedule (every 1-5 minutes) and delta against prior snapshot
SELECT
    queryid,
    userid::regrole::TEXT                       AS username,
    dbid::regdatabase::TEXT                     AS database_name,
    calls,
    total_exec_time                             AS total_ms,
    ROUND((total_exec_time / NULLIF(calls, 0))::NUMERIC, 3)  AS avg_ms,
    ROUND(stddev_exec_time::NUMERIC, 3)         AS stddev_ms,
    rows,
    shared_blks_hit,
    shared_blks_read,
    shared_blks_dirtied,
    blk_read_time,
    blk_write_time,
    LEFT(query, 300)                            AS query_text
FROM pg_stat_statements
WHERE calls > 10
ORDER BY total_exec_time DESC
LIMIT 50;

SQL Server:

-- Query Store equivalent for regression detection
-- Surfaces queries with high average duration and significant variability
SELECT TOP 50
    GETUTCDATE()                                              AS collected_at,
    qs.query_id,
    qt.query_sql_text,
    qp.plan_id,
    rs.count_executions,
    ROUND(rs.avg_duration / 1000.0, 3)                       AS avg_duration_ms,
    ROUND(rs.stdev_duration / 1000.0, 3)                     AS stdev_duration_ms,
    ROUND(rs.avg_logical_io_reads, 0)                        AS avg_logical_reads,
    ROUND(rs.avg_cpu_time / 1000.0, 3)                       AS avg_cpu_ms,
    rs.last_execution_time
FROM sys.query_store_query qs
JOIN sys.query_store_query_text qt ON qs.query_text_id = qt.query_text_id
JOIN sys.query_store_plan qp       ON qs.query_id = qp.query_id
JOIN sys.query_store_runtime_stats rs ON qp.plan_id = rs.plan_id
WHERE rs.last_execution_time > DATEADD(MINUTE, -10, GETUTCDATE())
  AND rs.count_executions > 5
ORDER BY rs.avg_duration DESC;

The delta pattern is important here: you capture a snapshot, store it, capture the next snapshot 60 seconds later, and compute the difference in calls, total_exec_time, and I/O. This gives you per-interval throughput rates rather than cumulative totals, which is what decision logic actually needs. A query whose avg_ms jumps from 12ms to 340ms between two consecutive snapshots is a regression event — and that event should trigger autonomous investigation without waiting for a human to notice a dashboard spike.

4.3 Building a Real-Time Telemetry Pipeline with Prometheus, OpenTelemetry, and TimescaleDB

Raw metric collection from database views is only half the architecture. Those metrics need to flow into a storage and query layer that the autonomous decision engine can interrogate in real time. The canonical stack for PostgreSQL environments in 2024 is Prometheus + postgres_exporter for scraping, combined with TimescaleDB for long-retention analytical queries and Alertmanager as a trigger surface for autonomous actions. For SQL Server, the equivalent uses sql_exporter or a custom telemetry agent, with the same downstream stack.

#### Prometheus Exporter Configuration

The postgres_exporter from Prometheus Community exposes the majority of pg_stat_* views as Prometheus metrics. Its custom query capability is what makes it useful for autonomous systems — you can expose any metric you need, not just the defaults.

# custom_queries.yaml — extend postgres_exporter for autonomy-relevant metrics
pg_replication_lag:
  query: |
    SELECT
      application_name,
      EXTRACT(EPOCH FROM (now() - pg_last_xact_replay_timestamp())) AS lag_seconds
    FROM pg_stat_replication
    UNION ALL
    SELECT
      'self' AS application_name,
      0      AS lag_seconds
    WHERE pg_is_in_recovery() = false
  metrics:
    - application_name:
        usage: "LABEL"
        description: "Replication target name"
    - lag_seconds:
        usage: "GAUGE"
        description: "Replication lag in seconds"

pg_long_running_queries:
  query: |
    SELECT COUNT(*) AS count
    FROM pg_stat_activity
    WHERE state = 'active'
      AND query_start < now() - interval '30 seconds'
      AND backend_type = 'client backend'
  metrics:
    - count:
        usage: "GAUGE"
        description: "Number of queries running longer than 30 seconds"

pg_bloat_ratio:
  query: |
    SELECT
      schemaname,
      tablename,
      ROUND(100 * (n_dead_tup::FLOAT / NULLIF(n_live_tup + n_dead_tup, 0))::NUMERIC, 2)
        AS dead_tuple_ratio
    FROM pg_stat_user_tables
    WHERE n_live_tup + n_dead_tup > 10000
  metrics:
    - schemaname:
        usage: "LABEL"
    - tablename:
        usage: "LABEL"
    - dead_tuple_ratio:
        usage: "GAUGE"
        description: "Percentage of dead tuples; trigger autonomous VACUUM at threshold"

This YAML feeds directly into the postgres_exporter --extend.query-path argument. The dead tuple ratio metric is particularly important: when dead_tuple_ratio exceeds 20% on a table with more than 100,000 rows, the autonomous maintenance engine should schedule a targeted VACUUM ANALYZE for that specific table — not a global vacuum, and not an alert to a human. This is what machine-readable telemetry enables: surgical, targeted autonomous action.

#### Alertmanager as an Autonomous Trigger Surface

Most teams use Alertmanager to page engineers. In an autonomous architecture, Alertmanager fires webhooks that trigger autonomous action scripts, not pagers. The alert becomes an event in the autonomous decision pipeline, not a notification.

# alertmanager.yml — route database alerts to autonomous action endpoint
route:
  group_by: ['alertname', 'database', 'instance']
  group_wait: 10s
  group_interval: 30s
  repeat_interval: 5m
  receiver: 'autonomous-action-handler'
  routes:
    - match:
        alertname: PostgresReplicationLagHigh
      receiver: 'autonomous-failover-handler'
      continue: false
    - match:
        alertname: PostgresLongRunningQueryDetected
      receiver: 'autonomous-query-killer'
      continue: false
    - match:
        alertname: PostgresTableBloatHigh
      receiver: 'autonomous-vacuum-scheduler'
      continue: false

receivers:
  - name: 'autonomous-action-handler'
    webhook_configs:
      - url: 'http://ade-controller:8080/api/v1/action'
        send_resolved: true
        http_config:
          bearer_token: "${ADE_WEBHOOK_TOKEN}"

  - name: 'autonomous-failover-handler'
    webhook_configs:
      - url: 'http://ade-controller:8080/api/v1/failover'
        send_resolved: false

  - name: 'autonomous-query-killer'
    webhook_configs:
      - url: 'http://ade-controller:8080/api/v1/terminate-query'
        send_resolved: false

  - name: 'autonomous-vacuum-scheduler'
    webhook_configs:
      - url: 'http://ade-controller:8080/api/v1/schedule-vacuum'
        send_resolved: true

The ADE controller at http://ade-controller:8080 is the autonomous decision engine — a service (covered in detail in Chapter 7) that receives webhook payloads from Alertmanager, validates the trigger conditions against a second source of truth, and executes the appropriate action. The double-check against a second source is non-negotiable: acting on a single metric signal without validation is how autonomous systems cause outages.

4.4 Wait Event Analysis: The Highest-Signal Metric Category for Autonomous Diagnosis

Wait events are the most information-dense signal category in any relational database engine. A CPU spike tells you that something is consuming CPU. A wait event tells you *why* the database is blocked — whether it's waiting for a lock held by another transaction, waiting for a buffer to be read from disk, waiting for network I/O to complete, or waiting for a WAL write to fsync. This distinction is what separates a monitoring system from an autonomous diagnostic system.

For PostgreSQL, wait events are surfaced in pg_stat_activity.wait_event_type and wait_event. The full taxonomy includes Lock, LWLock, IO, IPC, Timeout, Activity, Client, Extension, and BufferPin categories. Each category implies a different autonomous response.

The autonomous decision tree for PostgreSQL wait events looks like this:

  • → **`Lock` / `relation`**: A table-level lock is blocking sessions. Autonomous action: identify the lock holder via `pg_blocking_pids()`, evaluate its age and importance, and if it exceeds the threshold (e.g., >120 seconds with >5 blocked waiters), terminate the blocker.
  • → **`IO` / `DataFileRead`**: Sessions are waiting for data to be read from disk. Autonomous action: trigger `pg_prewarm` for the hot table, evaluate whether `shared_buffers` or `effective_cache_size` needs adjustment, and flag for capacity review if sustained.
  • → **`LWLock` / `WALWrite`**: High WAL write contention. Autonomous action: evaluate whether `wal_writer_delay`, `commit_delay`, or `synchronous_commit` settings can be safely tuned without durability risk given current replication topology.
  • → **`IPC` / `BgWorkerStartup`**: Background workers are taking too long to start. Autonomous action: check `max_worker_processes` against current utilization; scale if headroom is insufficient.

    • PostgreSQL:

    -- Wait event aggregation for autonomous pattern matching
-- This query produces a decision-table row: event category, severity, recommended action class
WITH wait_snapshot AS (
    SELECT
        wait_event_type,
        wait_event,
        COUNT(*)                                            AS waiter_count,
        MAX(EXTRACT(EPOCH FROM (now() - query_start)))      AS max_wait_seconds,
        AVG(EXTRACT(EPOCH FROM (now() - query_start)))      AS avg_wait_seconds
    FROM pg_stat_activity
    WHERE state = 'active'
      AND wait_event IS NOT NULL
      AND backend_type = 'client backend'
    GROUP BY wait_event_type, wait_event
),
lock_chains AS (
    SELECT
        blocked.pid                                         AS blocked_pid,
        blocking.pid                                        AS blocking_pid,
        blocking.query_start                                AS blocker_start,
        EXTRACT(EPOCH FROM (now() - blocking.query_start))  AS blocker_age_seconds
    FROM pg_stat_activity blocked
    JOIN pg_stat_activity blocking
        ON blocking.pid = ANY(pg_blocking_pids(blocked.pid))
    WHERE blocked.cardinality(pg_blocking_pids(blocked.pid)) > 0
)
SELECT
    ws.wait_event_type,
    ws.wait_event,
    ws.waiter_count,
    ROUND(ws.max_wait_seconds::NUMERIC, 2)                  AS max_wait_sec,
    ROUND(ws.avg_wait_seconds::NUMERIC, 2)                  AS avg_wait_sec,
    COUNT(lc.blocked_pid)                                   AS blocked_by_lock_chain,
    MAX(lc.blocker_age_seconds)                             AS oldest_blocker_age_sec,
    CASE
        WHEN ws.wait_event_type = 'Lock' AND ws.waiter_count >= 5
             AND MAX(lc.blocker_age_seconds) > 120          THEN 'TERMINATE_BLOCKER'
        WHEN ws.wait_event_type = 'IO' AND ws.waiter_count >= 10
             AND ws.avg_wait_seconds > 5                    THEN 'PREWARM_AND_REVIEW_CACHE'
        WHEN ws.wait_event_type = 'LWLock'
             AND ws.wait_event = 'WALWrite'
             AND ws.waiter_count >= 8                       THEN 'TUNE_WAL_SETTINGS'
        ELSE 'MONITOR'
    END                                                     AS recommended_action
FROM wait_snapshot ws
LEFT JOIN lock_chains lc ON ws.wait_event_type = 'Lock'
GROUP BY ws.wait_event_type, ws.wait_event, ws.waiter_count,
         ws.max_wait_seconds, ws.avg_wait_seconds
ORDER BY ws.waiter_count DESC;

    SQL Server:

    -- Wait type aggregation with autonomous action classification
-- Run on a schedule; compare delta between snapshots for rate-of-change analysis
WITH wait_stats AS (
    SELECT
        wait_type,
        waiting_tasks_count,
        wait_time_ms,
        max_wait_time_ms,
        signal_wait_time_ms,
        wait_time_ms - signal_wait_time_ms                  AS resource_wait_time_ms
    FROM sys.dm_os_wait_stats
    WHERE wait_type NOT IN (
        -- Exclude benign background waits
        'SLEEP_TASK', 'BROKER_TO_FLUSH', 'BROKER_TASK_STOP',
        'CLR_AUTO_EVENT', 'DISPATCHER_QUEUE_SEMAPHORE',
        'FT_IFTS_SCHEDULER_IDLE_WAIT', 'HADR_WORK_QUEUE',
        'HADR_FILESTREAM_IOMGR_IOCOMPLETION', 'ONDEMAND_TASK_QUEUE',
        'REQUEST_FOR_DEADLOCK_SEARCH', 'RESOURCE_QUEUE',
        'SERVER_IDLE_CHECK', 'SLEEP_DBSTARTUP', 'SLEEP_DBRECOVER',
        'SLEEP_MASTERDBREADY', 'SLEEP_MASTERMDREADY',
        'SLEEP_MASTERUPGRADED', 'SLEEP_MSDBSTARTUP',
        'SLEEP_SYSTEMTASK', 'SLEEP_TEMPDBSTARTUP',
        'SNI_HTTP_ACCEPT', 'SP_SERVER_DIAGNOSTICS_SLEEP',
        'SQLTRACE_BUFFER_FLUSH', 'WAITFOR', 'XE_DISPATCHER_WAIT',
        'XE_TIMER_EVENT', 'BROKER_EVENTHANDLER', 'CHECKPOINT_QUEUE',
        'DBMIRROR_EVENTS_QUEUE', 'SQLTRACE_INCREMENTAL_FLUSH_SLEEP',
        'WAIT_XTP_OFFLINE_CKPT_NEW_LOG'
    )
)
SELECT TOP 20
    wait_type,
    waiting_tasks_count,
    ROUND(wait_time_ms / 1000.0, 2)                         AS wait_time_seconds,
    ROUND(resource_wait_time_ms / 1000.0, 2)                AS resource_wait_seconds,
    ROUND(
        CASE WHEN waiting_tasks_count > 0
             THEN wait_time_ms * 1.0 / waiting_tasks_count
             ELSE 0 END, 2)                                  AS avg_wait_ms_per_task,
    CASE
        WHEN wait_type LIKE 'LCK_%'
             AND waiting_tasks_count > 20                   THEN 'INVESTIGATE_LOCK_CHAIN'
        WHEN wait_type = 'PAGEIOLATCH_SH'
             AND wait_time_ms > 30000                       THEN 'REVIEW_BUFFER_POOL_AND_IO'
        WHEN wait_type = 'WRITELOG'
             AND wait_time_ms > 10000                       THEN 'REVIEW_LOG_IO_SUBSYSTEM'
        WHEN wait_type = 'RESOURCE_SEMAPHORE'               THEN 'EVALUATE_MEMORY_GRANT'
        WHEN wait_type = 'THREADPOOL'                       THEN 'EVALUATE_MAX_WORKER_THREADS'
        WHEN wait_type LIKE 'HADR_%'
             AND wait_time_ms > 5000                        THEN 'CHECK_AG_REPLICA_HEALTH'
        ELSE 'MONITOR'
    END                                                     AS recommended_action
FROM wait_stats
ORDER BY wait_time_ms DESC;

    The recommended_action column in these queries is the bridge between metric and decision. An autonomous system ingests this output, reads the action class, and dispatches to the appropriate handler. This pattern — embedding decision logic directly in metric queries — is a valid approach for low-complexity decision rules, and it keeps the logic close to the data where it belongs.
